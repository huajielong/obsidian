# 预训练数据工程四方向对比分析

> 基于预训练数据工程师 JD 的四个招聘方向，从技术栈、工程定位、能力要求、职业光谱等多维度进行横向对比。

---

## 一、全景总览

```
数据采集 Pipeline       语言数据处理           多模态数据            数据基建
（离数据源最近）         （离模型最近）          （离多样性最近）        （离基础设施最近）

  数据源                   清洗质量              异构结构              底层性能
  ┌──────┐               ┌──────┐              ┌──────┐             ┌──────┐
  │ 采集  │─────→         │ 清洗  │─────→        │ 多模态 │─────→      │ 基建  │
  │策略+  │               │ 去重  │              │ 存储+  │            │ KV+   │
  │环路   │               │ 流水线 │             │ 适配  │             │ 数据湖│
  └──────┘               └──────┘              └──────┘             └──────┘
     ↓                       ↓                     ↓                   ↓
  数据从哪里来          数据怎么变干净        不同形态的数据           底层怎么跑
                         怎么组织             怎么统一管理
```

## 二、核心维度对比矩阵

| 维度 | 数据采集 Pipeline | 语言数据处理 | 多模态数据 | 数据基建 |
|------|-----------------|------------|-----------|---------|
| **Pipeline 位置** | 上游（数据入口） | 中游（核心加工） | 中上游（异构数据） | 下游/全链路（支撑） |
| **数据规模** | 百亿级 URL/链接 | TB 级 tokens | PB 级原始存储 | PB 级 + 元数据 |
| **核心算法** | 调度策略、质量预估 | MinHash、Embedding | CV 模型理解 | 一致性协议、数据分布 |
| **实时性要求** | 中（流式 + 批量） | 低（离线为主） | 低（离线为主） | 高（在线 KV/消息队列） |
| **瓶颈类型** | I/O + 网络带宽 | CPU + 内存 | CPU + 存储 | CPU + 内存 + 网络 |
| **典型工具链** | Scrapy/自定义、Kafka | Spark、DuckDB、Pandas | PyTorch、Megatron、vLLM | RocksDB、Kafka、Iceberg |
| **编程语言重心** | Python/C++ | Python (+ SQL) | Python | Rust/C++/Python |
| **调试难度** | 中等（分布式系统） | 低（数据可重现） | 高（异构数据） | 高（并发/分布式） |
| **跨团队协作频率** | 低-中 | 高（算法/训练/搜索） | 高（算法/框架） | 中（全链路） |

## 三、技术栈深度对比

### 3.1 数据采集 Pipeline 方向

**核心技术栈：**
```
分布式系统    → 任务调度、流式处理、DNS/连通性
数据策略      → 链接质量预估模型、属性聚合算法
网络工程      → 高并发下载、带宽管理、代理/IP 策略
链接处理      → URL 归一化、镜像检测、站群识别
```

**技术特征：**
- 离数据源最近，考验做"全网数据地图"的策略能力
- 需要区分"优质数据源"和"低质站群"——本质是一个大规模分类/排序问题
- 核心挑战在于规模化（百亿级 URL）和多样性（不同类型数据源的差异化策略）

**与 [[Harness_Engineering]] 的关联：**
数据采集环节是 Harness Engineering 中"Tool Registry"和"Retry/Recovery"理念在大规模分布式系统中的应用——把 Agent 中的单次 Tool Call 放大到全网规模的调度与容错。

### 3.2 语言数据处理方向

**核心技术栈：**
```
数据处理框架  → Spark、DuckDB、Pandas
去重算法      → MinHash（Jaccard 近似）、Embedding（语义）
清洗规则      → 规则引擎、质量评分模型
生产运维      → 任务调度、增量处理、质量回归、监控
```

**技术特征：**
- 离模型最近，清洗质量直接影响模型行为
- 去重是最核心的技术环节：MinHash 处理大规模近似去重，向量去重补充语义级去重
- 需要"对模型行为有自己独特的理解"——提示这是一个**数据品味**驱动的方向
- TB 级 tokens 处理能力是基本门槛

**与 [[Context_Engineering]] 的关联：**
语料清洗与去重是 Context Engineering 中"Select"子问题的前置步骤——决定哪些信息进入训练上下文窗口。

### 3.3 多模态数据方向

**核心技术栈：**
```
CV/多模态模型 → 内容理解（去重/质量筛选/打标）
数据结构      → 图像/视频/图文的结构化存储设计
存储架构      → 全生命周期管理、安全管控、自动化校验
训练框架      → PyTorch、Megatron、vLLM（了解即可）
可观测性      → 数据分析与可视化体系
```

**技术特征：**
- 离数据多样性最近，考验处理异构数据的能力
- 需要同时理解"数据工程"和"多模态模型"两个领域——最交叉的方向
- 针对 Pretrain/SFT/RL 三个阶段设计不同数据方案——高度依赖对训练流程的理解
- 加分项明确提到 NOI/IOI/ICPC/CCPC 竞赛——算法能力要求最高

**关键挑战：** 异构数据的结构化存储与统一管理。文本是序列，图像是矩阵，视频是时序+空间——如何用统一的数据结构描述它们？

### 3.4 数据基建方向

**核心技术栈：**
```
数据库/存储   → KV Store（RocksDB/LevelDB）、对象存储
消息队列      → Kafka、Pulsar
数据湖        → Iceberg/Hudi/LakeFS
分布式框架    → 自定义分布式处理引擎、任务调度
系统调优      → CPU/内存/网络 profiling、零拷贝、异步 I/O
```

**技术特征：**
- 离基础设施最近，考验系统底层功底
- 需要深入理解计算机组成、操作系统、计算机网络
- 核心工作是"让前面的三个方向跑得更快更稳定"
- Rust/C++ 要求最高（其他方向以 Python 为主）
- 加分项指向系统方向的学术研究（高水平系统会议论文）

**与 [[Harness_Engineering]] 的关联：**
数据基建是 Harness Engineering 的物理基础设施层——类似于 Agent Runtime 中"Context Manager"和"Safety Layer"的底层支撑。

## 四、编程语言需求梯度

```
方向            Python    C++     Rust     SQL     其他
──────────────  ───────  ──────  ──────  ───────  ──────────
数据采集 Pipeline   ★★★★    ★★★     ★★      ★       Shell/网络
语言数据处理        ★★★★★   ★       ★      ★★★★    Spark SQL
多模态数据         ★★★★★   ★★      ★       ★★      CUDA（加分）
数据基建           ★★★    ★★★★   ★★★★★    ★★      assembly（微调）
```

> Python 是所有方向的通用语言，但在不同方向的地位不同：
> - 采集方向：Python 粘合 C++ 性能模块
> - 语言数据方向：Python 是主力，配合 Spark SQL/DuckDB
> - 多模态方向：Python + PyTorch 是核心
> - 数据基建方向：Python 仅用于工具和胶水代码，核心是 Rust/C++

## 五、职业发展光谱

### 从"数据上游"到"底层基建"的技能光谱

```
                        通用能力
                    ┌────────────┐
                    │ Python     │
                    │ 分布式系统  │
                    │ 数据处理   │
                    └────────────┘

数据采集                    语言处理                   多模态                    数据基建
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│ 网络爬虫技术      │ │ 数据清洗方法论    │ │ CV/多模态模型    │ │ 系统编程（Rust）  │
│ 调度策略设计      │ │ 去重算法（MinHash)│ │ 异构存储架构     │ │ KV 数据库        │
│ 链接质量评估      │ │ 质量评分模型      │ │ 全生命周期管理   │ │ 消息队列         │
│ URL 归一化        │ │ 生产流水线运维    │ │ 训练阶段适配     │ │ 数据湖           │
│ DNS/连通性        │ │ 跨团队协作        │ │ 算法竞赛能力     │ │ 性能调优         │
└─────────────────┘ └─────────────────┘ └─────────────────┘ └─────────────────┘
    ↓                    ↓                    ↓                    ↓
数据源策略专家         数据质量专家         多模态架构师        基础设施工程师
```

### 横向迁移路径

```
数据采集 Pipeline ←──→ 语言数据处理
        │                      │
        │      数据基建        │
        │    （共同的底层）      │
        │                      │
        └────→ 多模态数据 ←────┘
```

- **采集 → 语言数据**：采集的数据进入清洗管道，容易理解上下游关系
- **语言数据 → 多模态**：文本清洗经验可迁移到多模态，需补充 CV 知识
- **数据基建 → 其他三个方向**：基建是其他方向的底层支撑，基建经验让其他方向"知其所以然"
- **多模态 → 其他方向**：多模态最交叉，向任一方向迁移都有基础

## 六、与 DeepSeek Harness 团队 JD 的交叉对比

| 维度 | 预训练数据工程师 | DeepSeek Harness |
|------|---------------|-----------------|
| **核心对象** | 预训练数据（语料/图像/视频） | Agent Runtime & Harness |
| **团队定位** | 数据燃料生产方 | Agent 能力交付方 |
| **用户** | 模型训练团队（内部） | 开发者/用户（外部） |
| **工程模型定位** | Context Engineering 为主体 | Harness Engineering 为主体 |
| **重叠领域** | 分布式系统、性能调优、Pipeline 工程 | 分布式系统、性能调优、Pipeline 工程 |
| **编程语言** | Python/C++/Rust（数据基建 Rust 最重） | Rust 为主（Harness 本身） |
| **核心差异** | 数据是从互联网"开采"的 | 数据是由用户/Agent 产生的 |
| **SOTA 追求** | 数据清洗效果的 SOTA | Agent 能力的 SOTA |

两个团队虽方向不同，但在**分布式系统、大规模 Pipeline 工程、性能调优**上有显著的重叠技能需求——这反映了 LLM 时代对基础设施工程能力的基础要求。

## 七、反向洞察：这份 JD 告诉我们什么

### 1. 数据工程是"硬功夫"
JD 中反复强调"硬功夫"——数据采集、清洗、基建都需要深厚的系统功底。这与"AI 时代只需要调 API"的流行叙事形成鲜明对比。

### 2. 数据品味成为核心竞争力
语言数据处理方向的加分项提到"对模型行为有自己独特的理解"——这说明在 AI 领域，**数据的价值已从"量"转向"质"**，而判断数据质量需要一种经验驱动的直觉（即"数据品味"）。

### 3. Pre-training → Post-training 全覆盖
JD 覆盖了 Pretrain / SFT / RL 三个训练阶段的数据需求，说明数据团队已经从"只管预训练数据"扩展到全训练周期的数据供给。

### 4. Rust 成为"基础设施语言"
数据基建方向明确以 Rust 为主，C++ 为辅——这反映了 Rust 在 LLM 基础设施领域（存储、消息队列、分布式框架）的崛起趋势。

### 5. 没有"AI 研究员"类岗位
四个方向全部是工程类岗位，没有一个方向叫"数据研究员"或"数据科学家"——说明预训练数据工程的核心需求是**工程能力**，研究能力是加分项而非必备项。
