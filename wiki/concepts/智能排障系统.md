## 概述

智能排障系统是指利用 LLM Agent 对系统异常（日志错误、服务降级、性能劣化等）进行**自动感知 → 归因分析 → 修复执行**的闭环系统。这是 [[Harness_Engineering]] 中 Agent Loop 的典型应用场景，也是 [[摘要-devops-ai-architect-xiamen]] 中"通用排障"职责的技术框架。

## 排障闭环

```
异常触发 → 感知检测 → 上下文聚合 → 归因分析 → 方案生成 → 执行修复 → 验证确认 → 知识沉淀
     ↑                                                                             │
     └───────────────────────── 闭环反馈（持续学习） ──────────────────────────────┘
```

## 核心模块

### 1. 异常感知 (Detection)
| 方式 | 说明 | 技术手段 |
|------|------|---------|
| **被动感知** | 告警接入（Prometheus Alert / PagerDuty / 自定义 Webhook） | AlertManager、Webhook Gateway |
| **主动巡检** | 定时/触发式健康检查（日志关键词/接口可用性/性能基线） | 巡检 Agent、Synthetic Monitoring |
| **预测感知** | 基于时序指标的异常预测（提前发现潜在故障） | 时序预测模型、Anomaly Detection ML |

### 2. 上下文聚合 (Context Aggregation)
将分散在多系统的信息聚合为 LLM 可理解的排障上下文：
- 错误日志（Last N 条，ELK/Loki 查询）
- 服务拓扑（依赖关系、上下游状态）
- 变更记录（最近部署/Git 提交/配置变更）
- 指标数据（CPU/内存/延迟/错误率趋势）
- 历史排障记录（相似问题的历史处理方案）

### 3. 归因分析 (Root Cause Analysis)
Agent 基于聚合的上下文进行推理，定位根因：
- **日志分析**：提取异常堆栈 → 识别错误模式 → 关联已知问题库
- **多信号交叉验证**：日志 + 指标 + 事件 → 排除干扰 → 定位根因
- **时间线重建**：按时间线排列事件 → 识别因果链 → 定位触发点
- 输出格式：根因描述 + 置信度分数 + 证据链

### 4. 自动修复 (Auto-Remediation)
| 修复类型 | 示例 | 安全等级 |
|---------|------|---------|
| **零风险** | 重启服务、清除缓存、重试失败任务 | 自动执行 |
| **低风险** | 回滚上次部署、调整配置参数 | 自动 + 通知 |
| **中风险** | 扩缩容、切换流量、数据库只读切换 | 人工确认后执行 |
| **高风险** | 数据恢复、权限变更、跨集群操作 | 人工执行 + 审批 |

### 5. 知识沉淀 (Knowledge Base)
- 每次排障记录结构化存入知识库（RAG 索引）
- 相似问题自动匹配历史方案
- 排障频率统计 → 推动根本性修复（避免重复排障）

## 技术架构

```
               ┌─────────────────────────────┐
               │    智能排障 Orchestrator      │ (Agent Loop 编排)
               │  (感知 → 归因 → 修复 → 验证)  │
               └──────────┬──────────────────┘
                          │
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼
  ┌──────────┐    ┌──────────────┐   ┌──────────┐
  │ 工具链    │    │ LLM Gateway  │   │ 知识库   │
  │ (K8s API/ │ ←→ │ (模型路由)    │ ←→ │ (RAG)    │
  │  Prom/ELK)│    └──────────────┘   └──────────┘
  └──────────┘
```

## 与三层工程模型映射

```
Prompt Engineering    ← 排障分析 Prompt、修复方案生成指令
Context Engineering  ← 日志/指标/事件上下文聚合、历史知识检索
Harness Engineering ← 工具编排（K8s/Prom/ELK）、修复执行沙箱、审批流程
```

## 设计原则

- **安全优先**：默认只读（分析），写操作（修复）分层级授权
- **渐进自治**：先做归因分析（人执行）→ 低风险自动修复 → 全闭环自治
- **可干预**：Agent 决策过程全透明，人可随时接管
- **不重复犯错**：每次排障结果沉淀到知识库，避免同类问题重复修复
- **熔断机制**：修复后指标恶化自动回滚/停止
