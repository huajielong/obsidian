# AI 集群可靠性工程（AI Cluster Reliability Engineering）

## 定义

AI 集群可靠性工程是**以大规模 AI 算力集群（万卡→数十万卡）的持续稳定运行为目标，通过故障管理、可观测性体系、自动化运维和性能验证，让大模型训练与推理始终建立在可靠算力底座之上的系统工程实践**。

其核心使命是：

> **让巨大而复杂的 AI 超算集群成为稳定可靠的生产力。** 在硬件故障常态化的十万卡规模下，通过系统化的容量管理、故障响应、自动化和可观测性，保障每一次训练和推理的可靠性。

AI 集群可靠性工程与 [[超算集群工程]] 构成 **"Build vs Run"** 的互补关系——超算集群工程回答"集群怎么建成"，可靠性工程回答"集群怎么持续稳定运行"。

```
Prompt Engineering                    ← 怎么问
    ↓
Context Engineering                   ← 给什么信息
    ↓
Harness Engineering                   ← 怎么跑（Agent 控制逻辑）
    ↓
═══════════════════════════════════════  ← 应用层 / 基础设施层 分界线
    ↓
AI训练推理系统工程                    ← 怎么训练/推理
    ↓
AI计算引擎工程                         ← 怎么算（算子/通信/编译器）
    ↓                              ↘
超算集群工程                           ← 怎么建成（架构/调度/网络/散热）
AI集群可靠性工程                       ← 怎么持续运行（监控/故障/自愈）★
    ↓

    ↓

```

> **Build vs Run 的分工**：超算集群工程设计并建设集群；AI集群可靠性工程确保集群持续输出生产力。两者需要紧密协作，但思维模式和成功指标不同。

### 与超算集群工程的对比

| 维度 | 超算集群工程 | AI集群可靠性工程 |
|------|------------|----------------|
| **核心问题** | "集群怎么设计、建成、上线？" | "集群怎么持续稳定运行？" |
| **时间焦点** | 建设期、架构设计、硬件选型 | 运营期、持续保障、故障响应 |
| **核心动作** | 设计、开发、选型、部署 | 监控、排查、修复、优化、预防 |
| **成功指标** | 系统是否能工作？吞吐是否达标？ | MTTR、可用性、告警准确性、自愈覆盖率 |
| **面对的问题** | "如何构建最好的系统？" | "坏了怎么办？怎么不再坏？" |
| **思维模式** | 建设者（Builder） | 守卫者（Guardian） |
| **技术栈侧重** | 调度系统、网络拓扑、编译优化 | 监控告警、自动化运维、故障注入、混沌工程 |

### 与 SRE（Site Reliability Engineering）的关系

| 维度 | 传统 SRE | AI 集群可靠性工程 |
|------|---------|-----------------|
| **服务对象** | 面向用户的 Web 服务 | 面向训练作业/模型的 **算力服务** |
| **规模单位** | 服务器/容器数量 | **GPU 卡数、节点数** |
| **核心资源** | CPU/内存/网络 | **GPU/NPU/DDR/HBM/NVLink** |
| **故障模式** | 进程崩溃、延迟飙升 | 卡掉线、NCCL 超时、NVLink 降速、温度过高 |
| **恢复手段** | 重启容器、切换流量 | 硬件置换、作业迁移、Checkpoint 回滚 |
| **可观测性焦点** | 请求延迟、错误率、饱和度 | **GPU 利用率、通信效率、Effective TFLOPs** |

## 四大核心能力

### 1. 故障管理（守护"吞金兽"）

**核心原则**：在十万卡规模下，故障不是异常，而是常态。

#### 关键指标

| 指标 | 定义 | 目标 |
|------|------|------|
| **MTTR（Mean Time To Recover）** | 故障发生到恢复的平均时间 | 持续缩短 |
| **MTBF（Mean Time Between Failures）** | 两次故障之间的平均时间 | 十万卡规模下此指标无意义 |
| **RCA Rate（Root Cause Analysis Rate）** | 故障后提交根因分析的比例 | 100%（拒绝"重启就好了"）|
| **自愈率** | 自动化处理的故障占比 | 持续提升 |

#### 故障分层处理模型

```
故障发生
    ↓
L1：自动检测（监控/告警触发）
    ↓
L2：自动诊断（规则引擎/知识库匹配）
    ↓
    ├── 已知模式 → L3：自动修复（重启/切换/降级）→ 验证 → 闭环
    └── 未知模式 → L4：人工介入（值班工程师）
                        ↓
                   根因分析（RCA）
                        ↓
                   知识沉淀 → 自动化覆盖扩展
```

#### RCA 文化

JD 中明确要求"每次故障都交出根因分析，而不是一句'重启就好了'"。这对应 **Blameless Postmortem（无指责事后复盘）** 文化：

```
好的 RCA：                  不好的 RCA：
"NVLink 错误计数超阈值 →     "GPU 卡掉了，重启后好了"
由于散热异常导致降速 →        
已修复散热策略"              
```

### 2. 新硬件验收（开箱即巅峰）

每一批新到货的硬件都需要系统化的验收流程，确保**每一台都以最佳状态投入生产**。

#### 验收流水线

| 阶段 | 检查内容 | 通过标准 |
|------|---------|---------|
| **物理检查** | 外观、固件版本、序列号登记 | 无物理损伤，固件版本一致 |
| **节点级基准** | GPU 性能（带宽/算力/Tensor Core）、内存带宽、CPU 性能 | 达到硬件规格书标称值 ± 容差 |
| **网络级基准** | RDMA 延迟/带宽、NCCL All-Reduce 性能、RoCE PFC 检测 | 无丢包，带宽 ≥ 理论值 95% |
| **存储级基准** | 并行文件系统读写吞吐、元数据性能 | 达到集群基准线 |
| **稳定性测试** | 72h 压力跑测（混合负载） | 0 故障，性能无退化 |
| **生产模拟** | 运行典型训练任务，与已有节点对比 | 训练吞吐 ≥ 已有节点平均值 |
| **交付** | 标记为就绪，加入生产集群 | 全部检查项通过 |

#### 性能差距分析

验收的核心问题：

> 理论峰值是多少？实际跑到了多少？差距来自硬件、驱动、网络还是配置？

```
理论峰值（Roofline Model）   ← 硬件规格书
    ↓ 差距分析
实际可达峰值                 ← 实验室最佳配置
    ↓ 差距分析
生产环境实际性能              ← 本节点的基准测试结果
```

每层差距的归因：

| 差距层次 | 典型原因 | 修复手段 |
|---------|---------|---------|
| 理论→实际可达 | 散热限制、功耗封顶、硅片质量差异 | 优化散热、调整功耗策略 |
| 实际可达→生产 | 驱动版本、内核参数、网络配置、邻居干扰 | 配置基线化、环境一致性检查 |

### 3. 可观测性体系（打开"黑盒"）

十万卡集群的可观测性不同于传统 IT 监控——信号量级和信息维度都远超传统系统。

#### 三层观测架构

```
┌─────────────────────────────────────────────────┐
│  业务层观测                                       │
│  有效 TFLOPs / 训练吞吐 / 作业完成率 / Checkpoint 耗时  │
├─────────────────────────────────────────────────┤
│  系统层观测                                       │
│  GPU 利用率/温度/功耗 / NCCL 延迟 / 网络吞吐 / I/O 带宽  │
├─────────────────────────────────────────────────┤
│  硬件层观测                                       │
│  NVLink 错误计数 / PCIe 重试 / 内存 ECC / 链路丢包   │
└─────────────────────────────────────────────────┘
```

#### 信号降噪

十万卡集群产生的告警量级远超人工处理能力。降噪策略：

| 策略 | 做法 | 效果 |
|------|------|------|
| **聚合** | 相似告警合并为事件 | 300条/小时 → 3个事件 |
| **关联** | 因果链分析，只报根因 | 网络丢包 → 不再报导致的所有连接超时 |
| **抑制** | 已知故障模式静默 | 已知维修窗口内的节点离线不报警 |
| **动态阈值** | 基于历史数据自动调整告警阈值 | 温度在夏季自动放宽阈值 |

#### 三个核心指标

JD 中的经典问题——如果只给 3 个指标判断集群健康度：

| 核心指标 | 揭示什么 | 如何度量 |
|---------|---------|---------|
| **① Effective TFLOPs Utilization** | 训练产出效率，反映计算/通信/框架综合表现 | 实际 TFLOPs ÷ 理论峰值 TFLOPs |
| **② 节点健康评分** | 硬件/OS 层面的整体健康度 | 聚合 GPU 错误、NVLink 降速、温度、磁盘健康等信号 |
| **③ 作业异常率** | 训练作业的非正常终止比例 | 失败作业 ÷ 总运行作业 |

### 4. 自动化与自愈（让集群自己救自己）

在每天近 300 次故障的规模下，自动化不是加分项，是生死线。

#### 自动化成熟度模型

```
L0：纯人工          ← 一切靠人
L1：告警通知        ← 故障发生→通知人→人处理
L2：自动化诊断      ← 故障发生→系统定位原因→人处理  ★ JD 当前阶段目标
L3：自动化修复      ← 故障发生→系统定位并修复→人确认
L4：自动预防        ← 系统在故障前自动调整避免发生
```

#### 故障场景覆盖矩阵

设计自动化方案时需识别每个故障场景：

| 故障类型 | 发生率 | 能否自愈 | 自愈方案 | 边界条件 | 误判代价 |
|---------|--------|---------|---------|---------|---------|
| GPU 掉卡（瞬时） | 高 | ✅ | 重置 GPU → 恢复训练 | 连续掉卡 > 3 次 → 换卡 | 一次无效重置（1min 训练损失）|
| NCCL 超时 | 高 | ✅ | 重置通信链路 → 重新建立 | 同一链路连续超时 > 5 次 → 报修 | 通信中断（恢复时间 < 手动）|
| 节点宕机 | 中 | ✅ | 自动 Checkpoint → 转移任务 | 节点上运行关键作业 → 人工确认 | 计算资源浪费（< 5min）|
| 存储节点故障 | 低 | ❌ | 人工介入 | 涉及数据一致性 | 数据损坏风险 |
| 网络分区 | 低 | ❌ | 人工介入 | 影响面大，需要决策 | 训练回退 |

#### 自动化边界意识

JD 中明确要求思考自动化的边界：

> 当自动化判断错误时，代价是什么？

这是可靠性工程中最核心的权衡——**自动化的召回率 vs 精确率**：

```
高召回率（少漏报）→ 更多误伤（False Positive）
     ↕ 权衡
高精确率（少误伤）→ 更多漏报（False Negative）
```

**安全法则**：宁可漏报 100 次，不可误伤 1 次训练作业。因为一次误判的自动重启可能导致数小时训练进度归零。

## 关键思维框架

### 故障定位方法论

面对模糊的"训练变慢了"，结构化的排查路径：

```
训练变慢
    ↓
Step 1：范围确定
   ├── 单个节点的问题？→ 定位到卡/网络/存储
   ├── 全集群的问题？→ 检查调度/网络/共享存储
   └── 特定任务的问题？→ 检查代码/数据

Step 2：维度排除（逐层排查）
   ├── GPU 利用率是否正常？→ DCGM + nvidia-smi
   ├── NCCL 通信是否正常？→ NCCL 测试 + 网络遥测
   ├── 存储 I/O 是否正常？→ iostat + 文件系统指标
   ├── CPU 是否有瓶颈？→ 内核调度/中断
   └── 散热是否降频？→ 温度历史曲线

Step 3：根因锁定
   └── 确认 → 修复 → RCA
```

### Change Management（变更管理）

> 每次变更前本能地想好"怎么回滚"。

| 变更类型 | 风险等级 | 回滚方案 |
|---------|---------|---------|
| 驱动升级 | 高 | 保留旧驱动，双分区启动 |
| 内核参数调整 | 中 | ansible 批量回滚配置 |
| 监控规则变更 | 低 | 旧规则 Git 版本管理 |
| 网络配置变更 | 极高 | 全量备份 + 分批灰度 |

## 与 Harness Engineering 的关系

### Harness 8 核心元件的可靠性映射

| Harness 元件 | 集群可靠性的贡献 | 直接影响 |
|-------------|----------------|---------|
| **① Agent Loop** | 训练作业的调度/执行生命周期管理 | Agent Loop 的稳定性依赖集群可靠性 |
| **② Tool Registry** | 资源配额与访问管控的可靠性 | 工具调用的权限控制 |
| **③ Context Manager** | — | 间接（不直接依赖）|
| **④ Safety Layer** | 作业隔离、故障隔离 | 集群级容灾保障作业不互相干扰 |
| **⑤ Retry / Recovery** | **最核心关联** | 集群故障时 Checkpoint 恢复是最后的防线 |
| **⑥ Telemetry / Observability** | **第二核心关联** | 集群可观测性 = Agent 可观测性的物理层 |
| **⑦ Eval Harness** | 评测作业的稳定性保障 | Eval 不被集群故障中断 |
| **⑧ Cost / Latency** | 可靠性 = 成本优化 | 故障减少 → 训练资源不浪费 → 省钱 |

> **核心公式**：集群可靠性 × 算力规模 = 有效训练产出。在十万卡规模下，可靠性降低 1% = 每天 1000 卡·天的算力蒸发。

## 关键挑战

### 1. 告警疲劳（Alert Fatigue）

十万卡集群的告警量级足以让任何值班团队失效。核心问题不是"装什么监控"，而是**如何让告警变得值得关注**。

### 2. 故障复现难度

AI 集群故障往往是 Heisenbug——排查时消失、运行时复现。这要求：
- 保留故障现场快照（GPU 状态、网络计数器、日志）
- 系统化的复现测试（故障注入/混沌工程）
- 基于统计的故障模式识别而非单次复现

### 3. 自动化与人工的边界

自动化越多越好吗？不——需要考虑：
- **误判成本**：某次自动重启可能打断已运行 3 周的训练
- **信任曲线**：自动化频繁误报 → 团队关掉自动化 → 回到纯人工
- **回退机制**：自动化的失败模式是什么？怎么兜底？

### 4. 新硬件的"磨合期"

每一代新 GPU/交换机/NIC 都有独特的"性格"——初期故障率曲线（Infant Mortality）可能很高：

```
故障率
  ↑
  │    ★ 早期故障    稳定期    ★ 老化期
  │   /  \                    /
  │  /    \                  /
  │ /      \________________/
  └────────────────────────────→ 时间
       新硬件交付               数月后稳定
```

这要求在交付初期增加监控密度、降低自动化的风险容忍度。
