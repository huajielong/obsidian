# 从无代码到Agent范式的转移

## 概述

传统无代码平台（如 Zion、Retool、Appsmith）通过**可视化拖拽**让非技术用户构建应用。而 **Skill+CLI+Workflow** 范式则通过**自然语言描述 + AI 执行编排**实现同样的目标——但路径完全不同。这不是简单的"过时"，而是一次**能力升维**。

## 对比：传统无代码 vs Agent 范式

| 维度 | 传统无代码（Zion 类） | Agent 范式（Skill+CLI+Workflow） |
|------|---------------------|--------------------------------|
| **交互方式** | 拖拽图形界面 | 自然语言对话 |
| **表达能力** | 受限于预置组件库 | 无上限，AI 可动态生成任意逻辑 |
| **扩展机制** | 插件/代码组件 | Skill（封装能力）+ CLI（执行接口） |
| **编排层** | 可视化流程设计器 | Workflow 脚本（确定性编排） |
| **维护成本** | 平台升级、组件兼容 | AI 自动适配 |
| **学习曲线** | 需学习平台的"方言" | 零学习，说人话即可 |
| **锁定风险** | Vendor lock-in | 开源工具链，数据自主 |
| **协作方式** | 共享可视化设计稿 | 共享 Prompt/Skill/Workflow 配置 |

## 范式转移的底层逻辑

### 三层工程模型的映射

传统无代码平台的各层在 AI 时代有了对应的替代形态：

```
传统无代码平台          AI Agent 范式
─────────────────      ─────────────────
可视化搭建界面    →     Prompt / Skill（描述意图）
组件库 / 插件      →     CLI Tool（原子操作）
流程编排器         →     Workflow（组合编排）
部署 / 运维面板    →     Harness（观测/重试/沙箱）
```

### 关键转折点

以前需要可视化是因为"人看不懂代码"。现在 AI 能理解自然语言，**可视化层不再是必需品**。[[Harness_Engineering]] 所定义的"为 AI 设计约束机制与反馈回路"取代了"为人类设计图形界面"。

### 更深层的变革：从"翻译"到"执行"

- **过去**：把代码翻译成图形，让人能操作
- **现在**：把意图翻译成执行，让 AI 去完成

这本质上是 [[Harness_Engineering]] 的崛起——**未来的"无代码"不是拖拽，而是编排；未来的"开发"不是写代码，而是设计 Harness**。

## 传统无代码的剩余价值

以下场景中传统无代码**暂时**仍有优势：

| 场景 | 原因 |
|------|------|
| **企业治理** | RBAC/审批流/审计日志 —— AI Agent 的治理成熟度还不够 |
| **高度协作的团队环境** | 非技术成员之间共享视觉设计稿比共享 Prompt 更直观 |
| **确定性流程** | 表单/审批/BPM 等标准化流程，拖拽比让 AI 每次重新理解更可靠 |
| **低延迟/离线场景** | 无需每次调用 LLM，本地直执行 |
| **合规要求严格的行业** | 金融/医疗等领域对"黑箱"AI 的可解释性存疑 |

这些场景中的**治理/权限/合规**部分，会以 **AI Agent 的外围护栏**形态继续存在，而非被彻底淘汰。

## 趋势时间线

```
                    传统无代码
                       │
  2022-2024            ▼          （Zion、Retool、Appsmith 高峰期）
                "可视化就是未来"
                       │
  2024-2025            ▼          （Cursor、Claude Code 崛起）
                "自然语言就是新UI"
                       │
  2025-2026            ▼          （Skill+CLI+Workflow 融合）
                "编排层替代搭建层"
                       │
  2026+               ▼
            传统无代码 → 退化为"企业治理层"
            AI Agent  → 成为真正的"开发范式"
```

## 知识冲突

- 有一种观点认为传统无代码平台并未被取代，而是与 AI Agent 共存——前者负责标准化流程的前端，后者负责灵活推理的后端。本页面持"范式转移"立场，认为**搭建层正在被侵蚀，治理层以新形态存活**。两种观点并非互斥，而是观察角度不同（前端体验 vs 后端架构）。
