# AI 基础设施硬件工程（AI Infrastructure Hardware Engineering）

## 定义

AI 基础设施硬件工程是**以 AI 算力集群的物理硬件与网络基础设施为对象，通过选型评估、配置管理、基准压测、网络运维和故障响应，确保每一层硬件和网络在真实负载下接近物理极限的系统工程实践**。

其核心使命是：

> **让基础设施不成为瓶颈。** 从单块 GPU 的固件参数到跨地域万卡集群的网络拓扑，每一层都以可度量的方式达到硬件设计的理论性能。

AI 基础设施硬件工程位于整个技术堆栈的**最底层**——它管理物理设备，为 [[超算集群工程]] 提供可用的计算节点，为 [[AI集群可靠性工程]] 提供硬件诊断支持，为 [[AI训练推理系统工程]] 提供稳定的运行环境。

## 四大核心域

### 1. 硬件选型与评估

**核心问题**：面对快速迭代的硬件市场，如何为 AI 工作负载选择最优硬件？

#### 关键决策维度

| 维度 | 问题 | 评估方法 |
|------|------|---------|
| **计算** | GPU TFLOPS（FP16/BF16/FP8）是否匹配训练需求？ | 规格书 + 实际 benchmark |
| **显存** | HBM 容量和带宽是否满足模型需求？ | 模型 profiling + Roofline 分析 |
| **互联** | NVLink 带宽、PCIe 代际、网络带宽是否匹配？ | NCCL 测试、拓扑分析 |
| **功耗散热** | TDP 和散热方案是否匹配数据中心能力？ | PUE 建模、功耗测试 |
| **生态** | CUDA/Ascend C 工具链成熟度？ | 软件栈兼容性评估 |
| **总拥有成本** | 采购价 + 功耗 + 运维 + 寿命 = TCO？ | TCO 建模 |

#### 选型流程

```
需求定义（训练什么模型？什么规模？）
    ↓
候选筛选（NVIDIA H100/B200/GB200，华为昇腾 910B/910C）
    ↓
样机测试（跑真实 benchmark，非仅规格书对比）
    ↓
TCO 分析（性能 per dollar、功耗 per TFLOPS）
    ↓
决策 → 批量采购 → 验收 → 部署
```

### 2. 硬件配置管理与固件基线

**核心问题**：在数千台异构硬件中，如何确保每台设备的固件、驱动、配置一致且最优？

#### 配置管理体系

| 层级 | 管理内容 | 工具/接口 |
|------|---------|----------|
| **固件** | BIOS/UEFI、BMC、GPU VBIOS、NIC 固件、SSD 固件 | Redfish API、IPMI、fwupd、NVFlash |
| **驱动** | NVIDIA Driver、CUDA、NCCL、MLNX_OFED | 驱动仓库、版本锁定、升级脚本 |
| **OS 配置** | 内核参数、GRUB、网络配置、CPU 调频策略 | IaC（Ansible/Puppet）、配置即代码 |
| **硬件唯一性** | 序列号、资产标签、物理位置、拓扑关系 | CMDB、自动发现、标签扫描 |

#### 基线管理的最佳实践

```
定义基线：选定"黄金固件版本组合" → 实验室验证 → 锁定为基线
    ↓
部署基线：Redfish 批量查询 → 比对差异 → 自动下发
    ↓
验证基线：固件升级后自动跑压测 → 性能不退化则通过
    ↓
监控基线：持续跟踪固件版本偏离 → 偏离时告警
```

**关键原则**：任何固件/驱动升级都必须有**回滚方案**和**性能基线对比**。没有基线验证的升级是不完整的。

### 3. 基准压测框架

**核心问题**：如何自动化、标准化地验证每一块硬件的性能是否达到预期？

#### 测试覆盖范围

| 组件 | 测试工具 | 关键指标 | 基线对比对象 |
|------|---------|---------|-------------|
| **GPU** | gpu-burn、nvidia-smi、自研 kernel | TFLOPS、显存带宽、温度、功耗 | 同型号其他 GPU 均值 |
| **NVMe SSD** | fio、自研引擎 | 顺序/随机读写 IOPS、延迟 99th%ile | 同批次 SSD 基线 |
| **内存** | memtest86+、stream benchmark | 带宽、延迟、ECC 错误 | 同平台内存基线 |
| **网络** | iperf3、ib_write_bw、NCCL test | 带宽、延迟、PFC 风暴、丢包率 | 同网络链路基线 |

#### 压测流水线

```
硬件上架
    ↓
自动触发压测（无人值守）
    ↓
逐个组件测试 → 汇总数据 → 对比基线
    ↓
── 全部通过 → 交付生产 ──→ 标记为就绪
── 未通过   → 标记异常 → 人工诊断
```

#### 基线数据的工程化

每次压测的数据都应沉淀为**结构化数据**（而非日志文件）：

```
{
  "hostname": "gpu-node-0421",
  "gpu_0": {
    "model": "H100",
    "firmware": "95.02.1C.00.01",
    "tflops_fp8": 1978.3,     // 理论 2000, 达到 98.9%
    "hbm_bandwidth": 3340,    // 理论 3350, 达到 99.7%
    "max_temp": 82.1,         // 正常
    "pcie_gen": 5,
    "pcie_width": 16
  },
  ...
}
```

这些数据可用于：
- **单节点**：与同型号历史均值对比，发现性能退化
- **跨节点**：发现 Silicon Lottery 中的"差片"
- **跨批次**：评估硬件批次间的质量波动

### 4. 物理网络运维

**核心问题**：如何确保跨地域、数万端口的网络基础设施稳定、高效、可扩展？

#### 网络层次

| 网络域 | 技术栈 | 管理重点 |
|--------|--------|---------|
| **数据中心网络（DCN）** | VXLAN、BGP EVPN、Spine-Leaf | 域间路由、负载均衡、故障隔离 |
| **跨数据中心互联（DCI）** | DWDM、OSPF/BGP、双活/主备 | 带宽、冗余、故障切换、延迟 |
| **RDMA 网络** | InfiniBand / RoCEv2 | 拥塞控制、PFC、自适应路由 |
| **带外管理网络** | IPMI / Redfish / iDRAC | 独立物理通道、安全隔离 |
| **办公网络** | 企业 Wi-Fi、VLAN、NAC | 覆盖、漫游、访客隔离 |

#### RDMA 网络运维要点

| 运维场景 | 检查项 | 工具/方法 |
|---------|--------|---------|
| **链路质量** | 光模块功率、CRC 错误、链路降速 | ibdiagnet、ethtool、交换机日志 |
| **拥塞控制** | PFC 风暴、ECN 标记、Back-to-Sender | 网络遥测、流分析、自适应路由配置 |
| **性能退化** | All-Reduce 吞吐下降、延迟抖动 | NCCL 测试、周期性基准对比 |
| **路由均衡** | ECMP 哈希偏斜、多路径负载不均 | 流哈希分析、自适应路由调优 |
| **固件一致性** | 交换机/网卡固件版本 | Redfish 批量检查、固件基线管理 |

#### 网络自动化

```
传统网络运维：    SSH 登录 → 手动输入命令 → 变更风险高 → 文档滞后
NetDevOps：      Git 管理配置 → CI/CD 校验 → 自动下发 → 自动验证
```

## 与相邻概念的关键协作

### 与超算集群工程：硬件供给

```
硬件工程：  选型 → 采购 → 上架 → 压测 → 配置基线 → 交付
                               ↓
超算集群工程：        接收节点 → 加入集群 → 分配角色
```

硬件工程交付的节点质量决定超算集群工程的起点。

### 与 AI 集群可靠性工程：硬件故障响应

```
硬件故障
    ↓
可靠性工程：告警 → 初步诊断 → 确定涉及 GPU/网络
    ↓ 升级
硬件工程：   硬件级诊断 → 固件检查/硬件置换 → 修复验证
```

硬件工程团队是可靠性工程的"后背"——处理需要硬件级技能的问题（固件、PCIe、光模块）。

### 与 IT 桌面：共享的远程管理基础设施

硬件工程搭建的带外管理网络（IPMI/iDRAC/Redfish）也服务于 IT 桌面团队的服务器管理需求。

## 关键挑战

### 1. 异构硬件的"配置漂移"

数千台服务器中，固件版本、驱动版本、BIOS 设置不可避免地出现漂移：

```
理想：  1000 台服务器 → 100% 一致的固件+驱动+配置
现实：  1000 台服务器 → 3 种 GPU 固件、5 种 BIOS 版本、2 种驱动版本
```

漂移的后果：节点间性能差异 → All-Reduce 长尾 → 有效训练吞吐下降。

### 2. RDMA 网络的"可见性"困境

RDMA 网络的数据传输绕过了 CPU 和内核，但也意味着：
- 传统 `tcpdump`、`netstat` 无法看到 RDMA 流量
- 网络丢包难以定位（端到端不可见）
- PFC 反压的传播路径难以追踪

### 3. 压测覆盖的"长尾"

即使通过标准压测，仍有部分故障在生产中才暴露：

```
压测能发现的：     GPU 计算错误、显存故障、网络丢包、SSD 损坏
压测难以发现的：   间歇性链路降速、电源纹波干扰、温度→性能的非线性拐点
```

需要在生产环境中持续**回归基线对比**来捕捉这些"潜伏故障"。

### 4. IT 桌面方向的组织 IT 成熟度

IT 桌面方向反映了组织从"能跑就行"到"零摩擦体验"的跨越：

```
L1：无序响应       ← "坏了来找我"
L2：标准化流程     ← 工单系统、SLA、自助服务
L3：主动优化       ← 数据分析驱动的高频问题消除 ★ AI 公司目标状态
L4：零摩擦         ← 一切自动、员工感知不到 IT 的存在
```
