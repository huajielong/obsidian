# 企业系统集成模式

## 定义

企业系统集成是指**将 AI Agent 平台与组织内外部既有系统（IM 平台、ERP、CRM、OA 等）打通**的架构方法。核心挑战在于：遗留系统往往没有 AI-friendly 的接口，需要在**不改动原系统**的前提下让 Agent 能"理解"和"操作"这些系统。

在 [[智能体研发工程师JD对标分析]] 中，本概念被识别为知识库的关键缺口——本页填补这一空白。

## 集成场景分类

### 按系统类型

| 系统类型 | 典型例子 | 集成难度 | Agent 可赋予的能力 |
|---------|---------|---------|------------------|
| **IM 平台** | 钉钉、飞书、企业微信、Slack、Teams | ⭐⭐ | 消息收发、审批流转、群机器人、日程查询 |
| **办公协作** | 飞书文档、语雀、Notion、Confluence | ⭐⭐ | 文档查询/创建、知识库检索、自动摘要 |
| **企业资源管理** | ERP（SAP/用友/金蝶） | ⭐⭐⭐⭐ | 订单查询、库存检查、采购审批 |
| **客户关系管理** | CRM（Salesforce/纷享销客） | ⭐⭐⭐ | 客户查询、商机更新、报表生成 |
| **项目管理** | Jira、Trello、Teambition | ⭐⭐ | 任务创建/更新、Sprint 状态查询 |
| **人力资源** | HR 系统、考勤系统、绩效系统 | ⭐⭐⭐ | 请假审批、考勤查询、组织架构 |
| **IT 运维** | Zabbix、Prometheus、PagerDuty | ⭐⭐⭐ | 告警响应、运维工单、日志查询 |
| **数据仓库** | Snowflake、ClickHouse、Tableau | ⭐⭐ | SQL 查询、报表生成、数据洞察 |

### 按集成深度

| 深度 | 说明 | 举例 |
|------|------|------|
| **L1: 通知推送** | Agent 向系统单向推送消息 | Agent 完成工作后在钉群发通知 |
| **L2: 查询检索** | Agent 读取系统数据回答问题 | "帮我查一下上周的销售数据" |
| **L3: 事务操作** | Agent 在系统中执行写入操作 | Agent 在 Jira 创建 Ticket |
| **L4: 复杂工作流** | Agent 跨多个系统编排多步骤流程 | "客户投诉 → 查订单 → 查物流 → 发起退款 → 确认" |

## 架构模式

### 1. MCP 标准化接入（推荐）

[[MCP]] 是企业系统接入 Agent 平台的**首选模式**——每个系统暴露一个 MCP Server，Agent 通过标准协议调用：

```
Agent Runtime
     │
     ├── MCP Client
     │     ├── mcp-server-钉钉    (Tools: send_message / create_approval / query_attendance)
     │     ├── mcp-server-飞书    (Tools: create_doc / query_calendar / send_notification)
     │     ├── mcp-server-SAP     (Tools: query_order / check_inventory / create_purchase)
     │     ├── mcp-server-Jira    (Tools: create_issue / query_sprint / transition_status)
     │     └── mcp-server-内部OA  (Tools: leave_apply / query_org_chart / approval_list)
     │
     └── [Skill 层] — Description 自动匹配哪个 MCP Server
```

**优势**：标准化协议、热插拔、开发者只需写一次 MCP Server 即可被所有 MCP Host 使用。

### 2. 适配器模式（Legacy 系统）

对于没有 REST API 的遗留系统，使用适配器模式包装：

```
Agent → MCP Client → [Adapter] → [Legacy System]
                            │
                    格式转换 + 协议适配
                    (SOAP ↔ REST / DB ↔ API / CLI ↔ API)
```

| 适配器类型 | 适用场景 | 示例 |
|-----------|---------|------|
| **数据库适配器** | 系统有 DB 但无 API | 直接读业务库，封装为 Tool |
| **CLI 适配器** | 系统有命令行工具 | 包装 SAP CLI 为 REST 接口 |
| **屏幕抓取适配器** | 纯 GUI 无 API | 使用 [[Agent_Interfaces#Computer Use|Computer Use]] 操作老旧桌面系统 |
| **文件适配器** | 通过文件交换数据 | 监听 FTP 目录或 S3 存储桶 |

### 3. 事件驱动集成（异步模式）

对于不需要即时响应的操作，使用事件驱动架构：

```
Agent → 发出事件 → 消息队列 (Kafka/RabbitMQ) → 各个系统的事件消费者
                                                   │
                                                   ├── 钉钉消费者 → 发消息
                                                   ├── ERP 消费者 → 创建订单
                                                   └── OA 消费者 → 发起审批
```

**适用场景**：批量操作、通知分发、非即时的工作流编排。

## IM 平台集成详解

IM 平台（钉钉/飞书/企微/Slack）是本 JD 中明确要求的外部系统。

### 统一架构模型

```
Agent Platform
    │
    ├── IM 统一适配层
    │     ├── 钉钉 Adapter (DingTalk OpenAPI)
    │     ├── 飞书 Adapter (Feishu OpenAPI)  
    │     ├── 企微 Adapter (WeCom API)
    │     └── Slack Adapter (Slack API)
    │
    ├── 能力抽象:
    │     ├── 消息发送 (text / card / markdown)
    │     ├── 审批处理 (create / approve / reject)
    │     ├── 日历操作 (query / create / update)
    │     ├── 文档操作 (read / write / search)
    │     └── 组织架构 (user / dept / role query)
    │
    └── 回调机制:
          ├── 事件订阅 (消息 / 审批 / @提及)
          └── Webhook → Agent 触发入口
```

### 钉钉集成要点

| 能力 | 接口 | Agent 场景 |
|------|------|-----------|
| **消息机器人** | Webhook 机器人 / Stream 模式机器人 | Agent 主动推送通知和操作卡片 |
| **审批流** | 审批实例创建/查询/审批 | 需要人工介入的 Agent 操作审批 |
| **群会话** | 群消息发送、@提及监听 | Agent 在群内提供交互服务 |
| **通讯录** | 组织架构查询 | Agent 理解企业人员关系 |
| **文档** | 钉钉文档读写 | Agent 知识库与文档协作 |
| **日历** | 日程查询/创建 | Agent 日程管理 |

### 飞书集成要点

| 能力 | 接口 | Agent 场景 |
|------|------|-----------|
| **消息卡片** | 交互式卡片消息 | Agent 富交互卡片（按钮/表单/选择器） |
| **飞书文档** | 文档/知识库 API | Agent 知识库同步、文档生成 |
| **多维表格** | Base/Bitable API | Agent 操作结构化数据 |
| **审批** | 审批流程 API | 需审批的 Agent 操作 |
| **云文档权限** | 权限管理 API | Agent 管理文档访问权限 |
| **机器人事件** | 事件订阅 | @Agent 自动触发交互 |

### 第三方 IM vs 自研 Agent 平台的对接

当自建 Agent 平台需要对接 IM 时，决策树：

```
Agent 需要主动发消息吗？
  ├── 需要 → 使用 Bot API / Webhook
  └── 不需要 → 只处理用户发起的消息
  
Agent 需要互动（按钮点击、表单提交）？
  ├── 需要 → 消息卡片 + 回调 endpoints
  └── 不需要 → 纯文本消息

Agent 需要操作 IM 中的资源（文档/日历/审批）？
  ├── 需要 → 完整的 OpenAPI 集成
  └── 不需要 → 仅消息通道打通
```

## 企业内部 Legacy 系统集成策略

### 六步评估法

```
Step 1: 该系统的 API 成熟度？
    ├── 有 REST API                  → 直接封装为 Tool
    ├── 有 SOAP / XML-RPC           → 适配器模式
    ├── 有数据库访问权限             → DB 适配器
    └── 只有 GUI 界面               → Computer Use 或第三方中间件

Step 2: 数据安全要求？
    ├── 高（财务数据）               → 本地部署 MCP Server，不出内网
    ├── 中（业务数据）               → 加密通道 + 审计日志
    └── 低（公开信息）               → 标准集成

Step 3: 操作需要事务一致性？
    ├── 需要                        → 实现 Compensating Transaction / Saga
    └── 不需要                      → 最终一致性

Step 4: 实时性要求？
    ├── 实时                        → 同步 REST/WebSocket
    └── 非实时                      → 事件驱动 (Kafka)

Step 5: 是否需要人工审批？
    ├── 需要                        → Agent + IM 审批流联动
    └── 不需要                      → 全自动执行

Step 6: 频率和延迟？
    ├── 高频 + 低延迟               → 缓存 + 连接池
    └── 低频 + 可接受延迟            → 标准调用
```

### 安全注意事项

- **凭证管理**：系统 API Key 不要硬编码，使用 Secrets 管理（Vault / AWS Secrets Manager）
- **最小权限**：Agent 用的 API Key 只授予所需操作的最小权限集
- **审计日志**：所有跨系统操作记录 who/what/when，用于事后审计
- **人工闸门**：高影响操作（写财务数据、删除、提权）增加人工审批环节
- **速率控制**：企业系统往往有脆弱的 QPS 上限，Agent 需内置限速
