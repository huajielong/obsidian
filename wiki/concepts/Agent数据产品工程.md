# Agent 数据产品工程（Agent Data Product Engineering）

## 定义

Agent 数据产品工程是**通过评测体系设计、数据生产管线构建和通用场景拆解，系统性地驱动 Agent 产品能力持续迭代的工程与产品混合实践**。它解决的核心问题是：

> **怎么知道 Agent 在真实场景中"好用"了，以及如何让它变得更"好用"。**

Agent 数据产品工程站在 [[AI产品工程]]（定义产品体验）和 [[Agent能力工程]]（构建底层能力）之间——它负责的是**"用数据连接产品体验和模型能力"的桥梁层**。

> **如果说 AI 产品经理回答"做什么"，Agent 能力工程师回答"怎么教会模型"，数据产品经理则回答"做得好不好、怎么更好"。**

## 在工程体系中的位置

### 完整定位图

Agent 数据产品工程是产品层和能力构建层之间的**横向桥梁层**：

```
产品化层     AI产品工程                          ← 怎么让技术变成产品
                ↑ ↓
评测数据层   Agent数据产品工程                   ← ★ 本概念：怎么评测、怎么生产数据（桥梁层）
                ↑ ↓
能力构建层   Agent能力工程                       ← 怎么塑造模型能力（RL 环境/评测任务）
    ↑
控制逻辑层   Harness Engineering / Prompt/Context 三层
    ↑
══════════ 应用层 / 基础设施层 分界线 ══════════
    ↑
框架/计算/集群/存储/硬件/设施/安全等七层基础设施
```

### 三条主线的协作关系

```
产品线：       用户需求 → 产品规划 → 功能设计 → 上线体验
                                              ↑
评测数据线：    场景拆解 → 评测标准 → 数据管线 → 评测反馈 ★ 本概念
                                              ↑
能力线：       能力缺陷 → RL 环境 → 模型训练 → 能力验证
                    
三线交汇点：          让模型在真实场景中更好地完成复杂任务
```

### 与相邻概念的对比

| 维度 | AI产品工程 | Agent数据产品工程 | Agent能力工程 |
|------|-----------|-----------------|--------------|
| **核心问题** | "技术能变成什么产品？" | **"怎么知道变好了？怎么让它更好？"** | "模型怎样学会更多能力？" |
| **主要产出** | 产品路线图、功能设计、原型 | **评测标准、数据管线、场景定义** | RL 环境、能力短板方案 |
| **场景范围** | 全产品线 | **办公/生活/搜索（通用场景）** | Code/安全/科学（专业领域） |
| **与数据的关系** | 用数据做决策 | **构建评测数据和训练数据的管线** | 用数据做训练 |
| **与模型的关系** | 使用者 | **改进的驱动者** | 构建者 |
| **技术深度** | 中 | **中高** | 高 |

## 三大核心职责

### 1. 定义并推进 Agent 在通用场景中的能力

**核心问题**：办公、生活、搜索这些通用场景中，Agent 应该具备什么能力？怎么知道它做到了？

#### 场景拆解框架

```
办公场景（Office）：
    ├── 文档处理（撰写、摘要、格式转换、协作编辑）
    ├── 信息管理（邮件、日历、任务、知识库）
    ├── 数据分析（表格、图表、报告、洞察）
    └── 工作流自动化（审批、通知、跨工具操作）

生活场景（Daily Life）：
    ├── 信息查询（天气、新闻、日程、交通）
    ├── 消费决策（比价、评价、推荐、预订）
    ├── 学习辅助（解释、练习、总结、翻译）
    └── 个人管理（笔记、提醒、健康、理财）

搜索场景（Search）：
    ├── 事实查询（准确、有时效性、可验证）
    ├── 知识探索（系统性、全面性、深度）
    ├── 决策支持（对比、权衡、推荐理由）
    └── 任务前置（搜了之后要做什么？）
```

场景拆解的目的是将抽象的"Agent 能力"转化为具体的、可评测的能力项。

#### 能力瓶颈识别

JD 要求"持续识别能力瓶颈，驱动模型迭代优化"——这需要：

```
用户失败日志 → 聚类分析 → 识别高频失败模式
                            ↓
定位能力瓶颈（如：办公场景中的表格理解能力不足）
                            ↓
驱动模型团队改进（定义具体需要提升的能力维度）
                            ↓
验证改进效果（在对应的评测集上跑分）
```

### 2. 评测体系设计——"能用"与"好用"的区隔

**核心问题**：如何设计一个能真正区分 Agent 表现层级的评测体系？

#### 评测体系的三层金字塔

```
第三层：人工评估（黄金标准）
    ├── 整体体验评级（1-5 分）
    ├── 异常处理评价
    ├── 安全意识评估
    └── 惊喜感/主动性评估
         ↑ 不可规模化，但最准确
         ↓ 校准
第二层：LLM-as-Judge + 半自动评估
    ├── 回答完整性/准确性
    ├── 执行路径合理性
    ├── 错误恢复能力
    └── 用户意图对齐度
         ↑ 可扩展，但需校准
         ↓ 自动化
第一层：自动化可验证指标
    ├── 任务完成率（Success Rate）
    ├── 端到端延迟（Latency）
    ├── Token 消耗（Cost）
    ├── 工具调用成功率
    └── pass^k（连续 k 次成功）
```

#### "能用" vs "好用"的评测维度

| 维度 | "能用"标准 | "好用"标准 | 评测方法 |
|------|-----------|-----------|---------|
| **任务完成** | 最终完成了任务 | **用最少步骤完成** | 自动化（步骤计数） |
| **准确性** | 结果基本正确 | **结果精确 + 无冗余** | LLM-as-Judge |
| **鲁棒性** | 理想输入能工作 | **异常输入优雅处理** | 故障注入测试 |
| **效率** | 能运行完毕 | **计算/Token 消耗最优** | 自动化（成本追踪）|
| **用户体验** | 功能可用 | **交互自然、反馈及时** | 人工评估 |

#### 打通人工与自动评估

JD 要求"打通人工评估与自动化评估"——核心方法论：

```
Step 1：人工标注少量样本（黄金标准）
    ↓
Step 2：用标注样本训练/校准 LLM-as-Judge
    ↓
Step 3：LLM-as-Judge 大规模自动化评估
    ↓
Step 4：定期抽样人工校验（漂移检测）
    ↓
Step 5：发现新维度时更新评测标准
```

### 3. 数据生产管线——从"理想态"到训练数据

**核心问题**：如何源源不断地生产高质量、高多样性的评测数据和训练数据？

#### "理想态"定义

JD 要求"定义回答及执行结果的理想态"——这是数据生产管线的**起点**：

```
理想态 = 对于给定的输入（用户问题 + 场景上下文），
         Agent 应该输出什么样的回答和执行路径才是"最好"的

例：
用户：总结这个季度销售报告
理想态：先理解报告结构 → 提取关键指标 → 对比上季度 → 
        给出有洞察的总结，而不是逐页复述
```

"理想态"是一种**产品决策**——它定义了 Agent 的行为标准，是评测和数据标注的参照系。

#### 数据生产管线的四步流程

```
需求定义：基于用户反馈和能力短板，确定需要生产什么数据
    ↓
场景设计：编写多样化的 Task Prompt（覆盖难度/风格/边界）
    ↓
数据生成：用模型生成 + 人工/自动标注 → 构建"理想态"答案
    ↓
质量验证：在评测集上验证数据效果 → 达标则进入训练/评测数据池
```

#### 数据多样性管理

| 多样性维度 | 说明 | 重要性 |
|-----------|------|--------|
| **难度分布** | 简单/中等/困难 比例合理 | 避免模型只见过简单题 |
| **场景覆盖** | 办公/生活/搜索各场景均衡 | 避免场景偏科 |
| **风格覆盖** | 开放/封闭/创意/事实 多类型 | 避免风格固化 |
| **边界覆盖** | 模糊/歧义/缺失信息/冲突指令 | 提升鲁棒性 |
| **语言覆盖** | 中文为主 + 多语言 | 全球化扩展 |

## 核心技术能力

### 1. 评测判断力——区分能力层级

这是数据产品经理最核心的素质：

> **"对任务质量有清晰的判断标准与内在追求，对'好'有明确的界定能力，能精准辨识输出质量的层级差异，并能将主观感受拆解为可传达、可检验的具体维度。"**

拆解来看：
- **有标准**：不是"感觉好"，而是有明确的评判维度和标准
- **能辨识层级**：能区分 60 分、80 分、95 分的差异，并指出差异在哪
- **可拆解**：能把主观感受转化为可操作的评估维度（如"回答不够简洁"→"回答包含冗余信息 X/Y/Z"）

### 2. 数据与评测的工程化能力

JD 要求"精通 prompt engineering 和（vibe）coding"——这与其他 JD 形成呼应：

| 角色 | 技术能力要求 | 用途 |
|------|------------|------|
| AI 产品经理（JD 9） | Vibe coding（做原型） | 产品原型 |
| Agent 数据产品经理（本 JD） | **精通** vibe coding | **构建评测和数据管线** |
| Code Agent 数据工程师（JD 10） | 传统编程 + AI coding | 构建 RL 环境 |

数据产品经理的技术能力要求高于产品经理，低于能力工程师——但要求"精通"和"丰富的使用自动化手段解决真实问题的经验"。

### 3. 场景理解力

JD 聚焦于办公、生活、搜索三大通用场景，要求数据产品经理具备：

- **对用户工作流的深度理解**：不只是一个功能点，而是用户在办公/生活中的完整任务流
- **对搜索体验的品味**：知道什么搜索结果是"满意的"，什么是"只能凑合用"
- **对 Agent 行为边界的感知**：知道哪些场景 Agent 能做好，哪些还需要人工介入

```

```

## 工程哲学

### 1. "人机共生"——Agent 的终极形态

团队使命的核心精神：Agent 不是工具，而是人的**延伸**。这与之前的 JD 有不同的哲学视角：

> "人与 AI 的关系从对话走向托付，从而真正解放人的行动力与创造力"

这意味着评测的重点不仅是"任务完成率"，更是**人的自由度是否增加了**——这是一个更高维度、更难衡量的标准。

### 2. "能用"是起点，"好用"是追求

"可用"和"好用"之间的差距，正是数据产品经理存在的理由：

```
不可用 → 能用（基础评测通过）
    ↓ 评测数据层
好用（多维评测通过 + 数据驱动迭代）
    ↓
惊喜感（模型做出用户未预期但正确的事）
    ↓
托付（用户信任 Agent 自主完成复杂任务）
```

### 3. 场景拆解是产品化的第一步

与工程角色不同，数据产品经理的工作从**场景拆解**开始——而不是从技术方案开始：

```
工程师思维：这个技术能做什么？→ 怎么实现？
产品思维：用户在什么场景下需要什么？→ 如何评测是否满足？
数据产品思维：用户场景如何拆解？→ 如何定义"好"？→ 如何生产数据让模型变好？
```

### 4. "数据是产品的一部分"

在 Agent 产品中，训练数据和评测数据不是后端的"辅助工作"，而是**产品本身的核心产出**：

> 产品的质量上限 = 评测体系能衡量的质量上限
> 产品的改进速度 = 数据管线迭代的速度

## 知识冲突

### 评测应该自动化到什么程度？
- **全自动派**：一切评测都可自动化，LLM-as-Judge 足够好
- **人工至上派**：只有人工判断才能真正评估"好用"，自动化会丢失关键信号
- **调和**：三层金字塔结构——自动化处理 80% 常规评测，LLM-as-Judge 处理 15%，人工覆盖关键的 5%（安全、创新、惊喜感）

### 数据产品经理需要懂多少技术？
- **产品优先派**：核心是产品判断和评测品味，技术由工程师实现
- **技术优先派**：评测和数据管线需要自己动手，否则效率太低
- **调和**：JD 要求"精通"而非"知晓"——说明公司期望数据产品经理能独立构建评测体系，但不需要做底层工程

### "通用场景"vs"专业场景"的取舍
- **通用派**：办公/生活/搜索是最大市场，做好通用场景覆盖最广
- **专业派**：Code/安全/科学等专业场景的差异化价值更高
- **调和**：通用场景（本 JD）和专业场景（JD 10）由不同团队负责，并行推进
