# AI 存储工程（AI Storage Engineering）

## 定义

AI 存储工程是**为 AI 训练与推理场景设计、构建高性能分布式数据基础设施的系统工程实践**，涵盖 KV Cache 存储系统、分布式文件系统和对象存储系统三大核心领域。

其核心使命是：

> **以零妥协的性能和无限扩展能力支撑 AI 应用的数据获取。** 让每一个训练步骤的数据读取、每一次推理请求的 KV Cache 查找、每一个 checkpoint 的持久化存储都以硬件极限的速度完成。

与前四个基础设施概念不同，AI 存储工程不是垂直堆叠的一层，而是**横向贯穿所有层的存储底座**。

## 在工程体系中的位置

### 存储是横向底座

AI 存储工程的位置是一个**水平层**——它为所有垂直层提供数据存储与访问能力：

```
Prompt Engineering                ← 怎么问
    ↓
Context Engineering               ← 给什么信息
    ↓
Harness Engineering               ← 怎么跑（Agent 控制逻辑）
    ↓
═══════════════════════════════════════  ← 应用层 / 基础设施层 分界线
    ↓
AI训练推理系统工程               ← 怎么训练/推理
    ↓
AI计算引擎工程                    ← 怎么算（算子/通信/编译器）
    ↓                         ↘
超算集群工程                      ← 大规模在哪儿跑
    ↓
Agent沙箱工程                     ← 安全地跑

    ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
    ↕  ↕  ↕  ↕  ↕  ↕  ↕  ↕  ↕  ↕  ↕  ↕  ↕  ←  横向支撑
    ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
    │ AI存储工程（横向底座）                  │ ★ 本概念
    │ KV Cache 存储 | 分布式文件系统 | 对象存储 │
    ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
```

### 各层对存储的具体需求

| 上层 | 存储需求 | 对应存储系统 |
|------|---------|-------------|
| **AI训练推理系统工程** | 训练 Checkpoint 持久化、DataLoader 数据加载、KV Cache 推理缓存 | 分布式文件系统（大文件高吞吐）+ KV Cache 存储（低延迟高频访问）|
| **AI计算引擎工程** | 编译缓存、算子库分发 | 对象存储（低频大文件）|
| **超算集群工程** | 集群共享存储、作业输入输出数据、日志 | 分布式文件系统（强一致、高并发）|
| **Agent沙箱工程** | 沙箱临时存储、Agent 执行环境快照 | 块存储（临时卷）+ 对象存储（持久化快照）|
| **预训练数据工程** | 数据湖、TB-PB 级原始语料 | 对象存储（大规模冷数据）+ 分布式文件系统（训练热数据）|
| **AI搜索工程** | 向量索引存储、倒排索引 | 分布式 KV + 对象存储 |

### 与相邻概念的对比

| 维度 | AI存储工程 | 超算集群工程 | AI计算引擎工程 |
|------|-----------|------------|-------------|
| **定位** | 横向底座（水平层） | 垂直层 | 垂直层 |
| **核心问题** | "数据怎么存、怎么取？" | "资源怎么管？" | "计算怎么快？" |
| **核心度量** | IOPS、延迟、带宽、一致性 | 利用率、PUE、吞吐 | TFLOPS 利用率、带宽 |
| **关键技术** | Raft/Paxos、io_uring、SPDK、RDMA、LSM-Tree | 调度算法、网络拓扑、散热 | CUDA、NCCL、TileLang |
| **用户** | 所有其他基础设施团队（内部）| 训练作业（内部）| 框架工程师（内部）|

> **关键定位**：存储是唯一以"所有其他基础设施团队"为用户的概念——它的好坏被所有层感知。

## 三大存储系统

### 1. KV Cache 存储系统

**核心问题**：如何在分布式集群中为推理请求提供低延迟、高可用、大容量的 KV Cache 服务？

#### 架构层次

```
推理请求 → vLLM/SGLang 推理引擎
              ↓
        本地 KV Cache（GPU HBM）      ← 第一级：显存缓存
              ↓（换出）
        分布式 KV Cache 存储          ← 第二级：本系统 ★
              ↓（持久化）
        分布式文件系统 / 对象存储      ← 第三级：持久化存储
```

#### 关键需求

| 需求 | 指标 | 实现手段 |
|------|------|---------|
| **低延迟** | P99 < 1ms | RDMA 零拷贝 + SPDK 用户态 I/O |
| **高吞吐** | 上亿级 IOPS | 分布式架构 + 多副本并行 |
| **高可用** | 99.99%+ | Raft 共识 + 自动故障切换 |
| **大容量** | TB-PB 级 | 多节点 SSD 聚合 + 分层存储 |
| **一致性** | 最终一致性即可 | 推理场景可容忍少量过时 Cache |

#### 与推理框架的关系

[[AI训练推理系统工程]] 中提到的 KV Cache 优化技术在此落地：

| 推理框架侧 | 存储侧响应 |
|-----------|-----------|
| PagedAttention 管理显存 KV Cache | 换出到存储系统，按 Page 粒度管理 |
| Prefix Caching 识别共享前缀 | 存储系统提供前缀索引和快速查找 |
| KV Cache 磁盘持久化 | 存储系统提供跨 session 的持久化 Bucket |

### 2. 分布式文件系统

**核心问题**：如何为数万 GPU 同时提供高性能、强一致的共享文件存储？

#### 训练场景的 I/O 模式

```
训练节点 1 ──┐
训练节点 2 ──┼──→ 共享文件系统 ←── Checkpoint 写入
训练节点 3 ──┘                    ←── 训练数据读取
...                              ←── 日志写入
训练节点 N ──┘
```

| I/O 类型 | 模式 | 尺寸 | 延迟要求 | 吞吐要求 |
|---------|------|------|---------|---------|
| **训练数据读取** | 顺序大块读 | MB-GB | 不重要 | 极致（GB/s 级）|
| **Checkpoint 写入** | 并发大块写 | GB-TB | 低（每秒同步）| 极致 |
| **日志/指标写入** | 小文件追加 | KB | 中等 | 中等 |
| **代码/配置加载** | 小文件随机读 | KB-MB | 中等 | 低 |

#### 经典 vs 下一代

| 维度 | 经典方案（Lustre/GPFS） | 下一代方案（本 JD） |
|------|----------------------|-------------------|
| **客户端** | 内核态客户端 | 用户态（FUSE 优化/自研）|
| **网络** | TCP/IP + RDMA（可选）| **RDMA 原生** |
| **元数据** | 专用元数据服务器 | 分布式元数据（Raft）|
| **一致性** | close-to-open | 强一致（分布式事务）|
| **扩展性** | 数百客户端 | **数万客户端** |
| **硬件** | HDD/SSD 混用 | **全 NVMe + 存储级内存** |

### 3. 对象存储系统

**核心问题**：如何为海量非结构化 AI 数据（数据集、模型权重、Checkpoint）提供高可靠、低成本、无限扩展的存储？

#### 与文件系统的分工

```
分布式文件系统          对象存储
  高性能场景               大容量场景
  训练数据加载            数据集归档
  Checkpoint 读写         模型权重备份
  共享工作目录            日志/指标的长期存储
  高频 I/O                低频但海量
```

| 特性 | 分布式文件系统 | 对象存储 |
|------|-------------|---------|
| **接口** | POSIX（文件打开/读/写）| RESTful API（PUT/GET/DELETE）|
| **一致性** | 强一致 | 最终一致（或 Read-after-Write）|
| **性能** | 高吞吐、低延迟 | 中等吞吐、高延迟 |
| **扩展性** | 有限（元数据瓶颈）| 无限（扁平寻址）|
| **成本** | 较高 | 较低（EC 编码）|
| **用例** | 训练热数据 | 训练冷数据、备份 |

## 核心技术

### 分布式共识与一致性

| 技术 | 用途 | 在 AI 存储中的角色 |
|------|------|-------------------|
| **Raft** | 强一致 Leader-based 共识 | KV Cache 存储的元数据管理、配置同步 |
| **Paxos** | 经典分布式共识 | 文件系统元数据的一致性保障 |
| **分布式事务** | 跨节点原子操作 | 对象存储的批量操作一致性 |

### 高性能 I/O 技术栈

从应用到 NVMe SSD 的完整 I/O 路径：

```
应用逻辑 (KV Cache 查询/写入)
    ↓
序列化/反序列化 (Protobuf/Cap'n Proto/FlatBuffers)
    ↓
存储引擎 (LSM-Tree / B-Tree，类 RocksDB/WiredTiger)
    ↓
I/O 框架 (io_uring / SPDK)
    ↓
NVMe 驱动 (内核态 / 用户态 SPDK)
    ↓
NVMe SSD (PCIe Gen4/Gen5)
```

**io_uring vs SPDK 的选择**：

| 维度 | io_uring | SPDK |
|------|---------|------|
| **操作模式** | 内核辅助的异步 I/O | 用户态轮询 |
| **延迟** | 低（~1-5μs） | 极低（~0.5-1μs）|
| **CPU 占用** | 低（中断驱动） | 高（轮询占满核心）|
| **适用场景** | 通用存储引擎 | KV Cache、极致低延迟场景 |
| **开发复杂度** | 中等 | 高 |

### RDMA 零拷贝数据链路

加分项中提到的"零拷贝数据链路"：

```
传统路径：  存储节点内存 → CPU → 内核网络栈 → NIC → 网络 → NIC → 内核网络栈 → CPU → 应用内存
RDMA 路径： 存储节点内存 → NIC（RDMA 引擎）→ 网络 → NIC（RDMA 引擎）→ 应用内存
                    ↕               ↕               ↕               ↕
              跳过 CPU 和内核      直接内存访问      直接内存访问      跳过 CPU 和内核
```

### 本地存储引擎

加分项中的"SSD/HDD 上的高性能本地存储引擎"：

| 代表性技术 | 特点 | AI 存储中的用途 |
|-----------|------|----------------|
| **RocksDB** (LSM-Tree) | 写优化、压缩、范围查询 | KV Cache 存储引擎的本地层 |
| **WiredTiger** (B-Tree + LSM) | 读写平衡、多线程 | 元数据存储 |
| **自研引擎** (基于 io_uring/SPDK) | 定制化的访问模式 | 针对 KV Cache 模式的极致优化 |

## 与 Harness Engineering 的关系

### Harness 8 核心元件的存储映射

| Harness 元件 | AI存储工程贡献 | 直接影响 |
|-------------|--------------|---------|
| **① Agent Loop** | Agent 执行状态的持久化存储 | 执行环境快照 → 断点续传 |
| **② Tool Registry** | Tool 产出数据的存储 | Tool 的输出持久化、共享 |
| **③ Context Manager** | 长上下文的外部存储扩展 | Context > 窗口时，溢出到存储系统 |
| **④ Safety Layer** | 数据隔离保障 | 租户间存储隔离、访问控制 |
| **⑤ Retry / Recovery** | Checkpoint 存储与恢复 | **最核心依赖**：没有可靠存储就没有恢复 |
| **⑥ Telemetry / Observability** | 日志/指标数据的存储 | Telemetry 数据的后端存储系统 |
| **⑦ Eval Harness** | Eval 数据集与结果的存储 | 大规模评测的数据分发 |
| **⑧ Cost / Latency** | KV Cache 存储降低推理成本 | Cache 命中减少重复计算 → 直接省钱 |

> **关键发现**：Retry/Recovery（#5）和 Context Manager（#3）是存储系统影响最深远的两个 Harness 元件——没有可靠的持久化存储，整个 Agent 的容错和长上下文能力都会受限。

## 关键挑战

### 1. KV Cache 存储的延迟-容量悖论

KV Cache 需要极低延迟（毫秒级），但又要大容量（TB-PB 级），这两者天然矛盾：

```
低延迟 ←────────────────────────────────→ 大容量
   DRAM（ns 级）    SSD（μs-ms 级）     HDD/对象存储（ms-s 级）
   ↑ 小容量              ↑ 本系统               ↑ 大容量
```

平衡策略：**多级缓存 + 智能预取**——预测哪些 KV Cache 即将被访问，提前从慢层加载到快层。

### 2. 训练与推理的 I/O 模式冲突

训练和推理对存储系统的需求差异巨大：

| 维度 | 训练 | 推理 |
|------|------|------|
| **I/O 模式** | 大块顺序（MB-GB） | 小块随机（KB）|
| **延迟要求** | 宽松（秒级可接受） | 严格（毫秒级）|
| **吞吐要求** | 极高（GB/s） | 高（百万 QPS）|
| **访问模式** | 批量读取，周期性 Checkpoint 写入 | 随机读取，随机写入 |
| **数据量** | 大（TB-PB）| 中（TB 级 KV Cache）|

**挑战**：一个存储系统很难同时完美支持两种模式。通常需要为训练和推理设计独立的存储子系统，但共享底层基础设施。

### 3. 一致性 vs 性能的取舍

| 场景 | 需要的一致性 | 可用的一致性折衷 |
|------|------------|----------------|
| **训练数据读取** | 最终一致即可 | 读旧数据不影响训练正确性 |
| **Checkpoint 写入** | **强一致必须** | 否则恢复时状态不一致 |
| **KV Cache 读取** | 最终一致即可 | 旧 Cache 最多导致重新计算 |
| **元数据操作** | **强一致必须** | 否则目录结构混乱 |

### 4. FUSE 的性能困境

加分项提到"深入解决过 FUSE 在性能与可维护性上的痛点"。FUSE（Filesystem in Userspace）是分布式文件系统的常用实现方式，但性能天坑：

| FUSE 问题 | 原因 | 缓解方案 |
|-----------|------|---------|
| **上下文切换** | 每次文件操作 → 用户态 → 内核态 → 用户态 | 大 I/O 合并、writeback cache |
| **双倍数据拷贝** | FUSE 在内核态和用户态之间拷贝数据 | splice/零拷贝 |
| **元数据延迟** | 每个 lookup 都需经过 FUSE 到后端 | 元数据缓存（attr/dentry timeout）|
| **Page Cache 低效** | FUSE 的 Page Cache 命中率不如内核文件系统 | 用户态缓存 + 预读策略 |

| 类型 | 代表 | 特点 |
