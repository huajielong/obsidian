# AI 产品工程（AI Product Engineering）

## 定义

AI 产品工程是**以系统化的方法将 AI 技术能力转化为用户可感知、可使用的产品体验的工程实践**。它解决的核心问题是：

> **如何让 AI 从实验结果变成改变世界的产品。**

AI 产品工程是整个技术堆栈中最上层的一层——它站在模型与世界之间，定义用户如何与 AI 交互、产品如何传递价值、技术突破如何抵达千万用户。

> 如果说此前所有工程层都在解决"如何造出更强大的 AI 系统"，AI 产品工程则回答"这个强大的系统用来做什么、用户怎么用它"。

## 在工程体系中的位置

### 完整定位图

```
世界（用户/场景/需求）
    ↑
AI产品工程                             ← 怎么让技术变成产品 ★ 本概念（最上层）
    ↑
═══════════════════════════════════════  ← 产品层 / 技术层 分界线
    ↑
Prompt Engineering                     ← 怎么问
Context Engineering                    ← 给什么信息
Harness Engineering                    ← 怎么跑（Agent 控制逻辑）
    ↑
═══════════════════════════════════════  ← 应用层 / 基础设施层 分界线
    ↑
AI训练推理系统工程                     ← 怎么训练/推理（框架层）
AI计算引擎工程                          ← 怎么算（算子/通信/编译器层）
超算集群工程                            ← 怎么建成（集群架构层）
AI集群可靠性工程                        ← 怎么持续运行（运营层）
    ↑
AI存储工程                       （横向底座：数据怎么存）
    ↑
AI基础设施硬件工程                      ← 物理硬件怎么选/测/管
AI数据中心工程                          ← 物理建筑设施怎么建/管
Agent沙箱工程                           ← 安全地跑（沙箱层）
```

### 与相邻概念的关系

| 维度 | AI产品工程 | Prompt Engineering | Harness Engineering |
|------|-----------|------------------|-------------------|
| **核心问题** | "技术能变成什么产品？" | "怎么让模型理解指令？" | "怎么让 Agent 可靠执行？" |
| **用户** | 终端用户/开发者 | 模型 | Agent |
| **输出** | 产品体验、功能、流程 | 提示词（字符串） | Agent Loop、Tool 编排 |
| **成功标准** | 留存率、满意度、转化率 | 任务完成率、指令遵循 | 可靠性、可观测性、成本 |
| **时间尺度** | 周-月（产品迭代） | 分钟-小时（Prompt 调优） | 毫秒-天（Agent 执行） |
| **抽象层级** | 最高（用户意图/场景/行为） | 中等（模型输入/输出） | 较低（执行控制） |

### 产品层与工程层的协作关系

```
产品团队：发现需求 → 定义体验 → 设计功能 → 验证假设
                            ↕ 翻译
工程团队：评估可行性 → 实现功能 → 交付模型 → 优化性能
```

产品层与工程层之间并非简单的"上下级"关系，而是**双向翻译**——产品经理需要理解技术能力边界（才能做出可行的产品决策），工程师需要理解用户需求（才能做出正确的技术选型）。

## 两大产品方向

### 1. AI 产品方向（消费级 AI）

**核心使命**：将模型能力转化为面向亿万用户的消费级产品体验。

#### 产品矩阵

| 产品形态 | 核心挑战 | 关键指标 | 竞争格局 |
|---------|---------|---------|---------|
| **Chat 网页端** | 对话体验的自然度、信息准确性 | 留存率、会话深度 | ChatGPT、Claude、Kimi |
| **APP 端** | 移动场景适配、交互范式创新 | DAU、使用时长 | 豆包、通义、Kimi APP |
| **开放平台（API）** | 开发者体验、文档质量、稳定性 | API 调用量、开发者 NPS | OpenAI API、Anthropic API |

#### 四层能力模型

```
第一层：产品规划
    ↓ 技术跟踪 → 策略制定 → 优先级管理
    → 需要理解模型能力的边界和演进方向
    → 在"做什么"和"不做什么"之间做出战略取舍

第二层：功能设计
    ↓ 数据分析 + 用户反馈 + 定性观察 → 问题拆解
    → 从海量用户行为数据中提炼产品机会
    → 用低门槛的设计降低用户使用 AI 的心理障碍

第三层：项目管理
    ↓ 跨团队协作 → 快速迭代 → 高质量交付
    → 协调模型、算法、研发、运营多团队
    → 在快速迭代和质量控制之间找平衡

第四层：产品品味
    ↓ 对交互细节的敏感度 + 对极致体验的追求
    → "不只是看过，而是自己用过、拆过"的深度用户体感
    → 对异常分支、边界条件和失败场景的敏锐嗅觉
```

### 2. Agent Harness 产品方向（基础设施产品）

**核心使命**：将 [[Harness_Engineering]] 从内部基础设施转化为面向开发者/研究员的优质产品。

#### 与传统产品经理的关键差异

| 维度 | 传统 PM | AI 产品 PM | Agent Harness PM |
|------|---------|-----------|-----------------|
| **用户群体** | 消费者 | 消费者 | 开发者/研究员 |
| **技术深度** | 低-中 | 中 | **高** |
| **成功指标** | DAU、留存、收入 | 使用深度、满意度 | 任务完成率、开发者效率 |
| **迭代周期** | 周-月 | 天-周 | 天-周 |
| **与工程的关系** | 提需求 | 协作探索 | **深度共同构建** |
| **社区管理** | 客服 | 用户反馈 | **开源社区运营** |
| **核心挑战** | 用户获取 | 体验优化 | 定义"好用"的指标 |

#### Harness PM 的独特职责

```
研究员（模型训练团队）                   终端用户（开发者/企业）
    ↑       ↓                                    ↑       ↓
    └─── Harness PM（翻译层/产品化层）──────┘
            ↓
    Harness 产品（Agent Loop / Tool Registry / Safety Layer / ...）
```

1. **指标定义**——如何衡量"Agent 是否真的在更多场景下更深入的帮助到更多的人"？这个开放性问题需要：
   - 分解为可量化的子指标（任务完成率、用户干预率、重试次数等）
   - 设计数据收集方法（日志分析、用户调研、A/B 测试）
   - 在定性洞察和定量数据之间建立连接

2. **路线图规划**——连接研究员的技术愿景和用户的真实需求：
   - 研究员想做什么（技术驱动）↔ 用户需要什么（需求驱动）
   - 在 Harness 的 8 核心元件（Agent Loop、Tool Registry、Context Manager 等）中确定优先级

3. **社区运营**——从海量用户反馈中提取产品迭代信号：
   - 维护开源/开发者社区
   - 跟踪用户任务完成情况
   - 建立反馈→产品的闭环

4. **共同进化**——"模型与 Harness 的共同进化"：
   - 模型能力提升 → Harness 需要适配新能力
   - Harness 使用数据 → 反馈给模型训练团队改进模型

## 核心技能与方法论

### 1. AI 产品经理的技术理解要求

AI 产品经理需要的技术理解深度远超传统产品经理：

| 技术要求 | AI 产品方向 | Agent Harness 方向 |
|---------|-----------|-------------------|
| **LLM API 使用** | 深度使用体验 | 内部机制理解 |
| **Agent 产品** | 深度使用（Claude Code、Cursor 等）| 深度使用 + 架构理解 |
| **模型能力边界** | 知道模型能做什么/不能做什么 | 理解为什么能/不能 |
| **UI/UX 设计** | AI 辅助原型设计 | AI 辅助原型设计 |
| **数据分析** | 用户行为分析、A/B 测试 | 系统性的数据收集 + 统计方法 |
| **编码能力** | — | vibe coding（不要求传统技术背景）|

### 2. Agent Harness PM 的技术知识图谱

JD 中列出的技术要求构成了一幅完整的 Agent 知识图谱：

```
LLM API                     ← 理解模型如何被调用
    ↓
KV Cache                    ← 理解推理成本的核心因素
    ↓
Agent Loop                  ← Agent 的"思考→行动→观察→思考"循环
    ↓
Tool Use                    ← 工具调用的机制与限制
    ↓
Reasoning / Planning        ← 模型的推理与规划能力
    ↓
Skills                      ← Agent 技能系统
    ↓
MCP                         ← 标准化工具调用协议
    ↓
Memory                      ← Agent 记忆系统
    ↓
Subagent / Multi-Agent      ← Agent 的协作与分工
```

同时需要：

| 工程层 | 理解要求 |
|--------|---------|
| **Prompt Engineering** | 提示词如何影响模型行为 |
| **Context Engineering** | 上下文窗口的管理与优化 |
| **Harness Engineering** | Agent 执行环境的完整控制逻辑 |

### 3. 数据驱动决策方法论

JD 强调了系统性的数据方法，这在 AI 产品中尤为关键：

```
定性方法（发现问题）             定量方法（验证假设）
    ↓                              ↓
用户访谈 → 发现使用痛点      →    A/B 测试 → 验证功能效果
用户观察 → 发现行为模式      →    灰度测试 → 验证规模化可靠性
用户反馈 → 发现异常场景      →    统计分析 → 验证统计显著性
```

**核心原则**：不要仅靠直觉做产品决策，也不要完全依赖数据做决策——**要把用户体感、产品直觉和行为数据结合起来做判断**。

### 4. "Vibe Coding"与 UI/UX 的融合

两个方向都强调：

> "具备 UI/UX 设计素养，能够在 AI 辅助下完成产品原型图设计、UI 设计"

这里的隐含信息：
- **不要求是专业设计师**——但要求能独立完成产品原型
- **AI 是加速器**——用 AI 工具（如 Claude、Midjourney、Figma AI）跨过"从想法到原型"的鸿沟
- **设计素养而非设计技能**——理解设计原则（一致性、可访问性、信息层级），而不是精通 Photoshop

## 工程哲学

### 1. "站在模型与世界之间"——桥接而非指挥

产品团队的角色不是"指挥工程师做什么"，而是"理解模型能力的边界，找到技术与用户需求的交汇点"。这要求：

- **技术谦逊**：承认自己不是最懂模型的人，但最懂用户
- **用户共情**：将用户的语言翻译为工程需求，将工程的能力翻译为用户的体验
- **双向翻译**：让研究员理解用户痛点，让用户理解能力边界

### 2. "从第一性原理出发"的产品思维

此前的工程 JD 强调"从第一性原理出发"的工程哲学（如"行业惯例只是历史经验"、硬件压测要"逼近物理极限"）。产品方向的等价物是：

> **不是"竞争对手做了什么所以我们也要做"，而是"用户在这个场景下真正需要什么，技术能为此做什么"。**

### 3. 产品与技术是一枚硬币的两面

十份 JD 覆盖从数据中心到产品体验的完整链条，揭示了一个核心认知：

> **一个 AI 公司的竞争力不是模型单点优势，而是从物理设施到用户体验的全链条能力。** 产品经理负责最上层的一环，但如果对下层一无所知，就无法做出好的产品决策。

## 知识冲突

### AI 产品经理是否需要技术背景？
- **技术派**：AI 产品经理必须懂技术，否则无法与技术团队有效沟通、无法理解产品能力边界
- **用户派**：AI 产品经理最重要的是用户同理心和产品品味，技术可以学习、可以用 AI 工具弥补
- **调和**：AI 产品方向偏向"用户派"，Agent Harness 方向偏向"技术派"——取决于产品的领域和形态

### 数据驱动 vs 直觉驱动
- **数据派**：所有产品决策必须有数据支撑，直觉不可靠
- **直觉派**：真正的创新来自产品直觉，数据只能验证已知问题
- **调和**：JD 中明确提到"把用户体感、产品直觉和行为数据结合起来做判断"——两者缺一不可

### "Vibe Coding"是否足够？
- **支持方**：AI 时代产品经理不需要会编码，vibe coding + AI 辅助足够完成原型和验证
- **反对方**：理解代码的产品经理才能与工程师深度协作，vibe coding 只能做表面原型
- **调和**：取决于产品方向——AI 产品方向可以接受 vibe coding，Agent Harness 方向需要更深的技术理解
