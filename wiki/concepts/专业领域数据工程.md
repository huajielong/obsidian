# 专业领域数据工程（Professional Domain Data Engineering）

## 定义

专业领域数据工程是**凭借领域专家级别的知识和判断力，通过评测体系构建、优化策略设计和高质量数据生产，系统性地将人类专业领域深度知识注入模型的工程实践**。它解决的核心问题是：

> **如何让模型在专业领域（医学、法律、小语种等）达到人类专家级别的精确度和判断力。**

专业领域数据工程是 [[Agent数据产品工程]]（通用场景评测数据方法论）在**高知识密度专业领域**的特定展开——方法论相同，但要求**领域专家级别的判断力**作为工作基础。

> **通用数据产品经理 ≈ 评测工程师 + 数据工程师**
> **专业领域数据产品经理 ≈ 领域专家 + 评测设计师 + 数据策展人**

## 在工程体系中的位置

### 完整定位图——从"广度"到"深度"的完整覆盖

```
产品化层     AI产品工程                           ← 消费级产品体验
                ↑
评测数据层   Agent数据产品工程（通用场景）         ← 办公/生活/搜索
                  ↑                              ★ 本概念（专业领域）
            专业领域数据工程                       ← 小语种/医学/法律
                ↑
能力构建层   Agent能力工程                         ← Code/安全/科学
    ↑
控制逻辑层   Harness Engineering / Prompt/Context 三层
    ↑
══════════ 应用层 / 基础设施层 分界线 ══════════
    ↑
框架/计算/集群/存储/硬件/设施/安全等七层基础设施
```

### 与 Agent 数据产品工程（JD 11）的关系

```
Agent数据产品工程（方法论框架）            专业领域数据工程（专业展开）
    ┌───────────────┐                    ┌───────────────┐
    │ 场景拆解       │  ← 方法论 →        │ 领域知识建模   │
    │ 评测体系       │  ← 方法论 →        │ 专业评测规范   │
    │ 数据管线       │  ← 方法论 →        │ 专家质量把控   │
    │ Vibe Coding    │  ← 技能 →          │ Vibe Coding    │
    └───────────────┘                    └───────────────┘
        通用场景                             专业领域
        产品+技术背景                        领域专家背景
        用户行为数据                         专家知识资源
```

两者共享同一套方法论框架（评测体系 + 数据管线），但所需的核心能力不同——前者需要产品+技术理解，后者需要**领域深度 + 评测判断力**。

| 维度 | Agent 数据产品工程（JD 11） | 专业领域数据工程（本 JD） |
|------|-------------------------|-----------------------|
| **发力方向** | 广度（覆盖最多用户场景） | **深度（覆盖最高价值领域）** |
| **核心能力** | 产品判断 + 数据工程 | **领域专家判断 + 数据工程** |
| **质量定义者** | 用户需求和产品品味 | **领域专业标准** |
| **数据来源** | 用户交互数据 | **专家知识和专业资源** |
| **评测挑战** | 区分"能用"vs"好用" | **确保知识精确性和推理严谨性** |
| **规模化方式** | 场景分解 + 数据管线 | **领域专家网络 + 知识体系建模** |

## 三大核心职责

### 1. 建评测体系——领域专家定义质量标准

**核心问题**：在专业领域，什么才是"好"的回答？如何系统性地衡量？

#### 三维度评测框架

```
知识准确性（Knowledge Accuracy）
    ├── 事实正确性：关键事实是否有误？
    ├── 引用准确性：参考资料是否准确引用？
    ├── 时效性：知识是否过时？
    ├── 完整性：是否覆盖了关键信息？
    └── 一致性：前后回答是否逻辑一致？

推理质量（Reasoning Quality）
    ├── 逻辑严密性：推理链是否有漏洞？
    ├── 鉴别诊断：是否考虑了替代可能性？
    ├── 证据支撑：结论是否有充分依据？
    └── 偏差识别：是否存在认知偏差/刻板印象？

表达水准（Expression Standard）
    ├── 术语规范：专业术语使用是否正确？
    ├── 语体适当：表达是否符合领域惯例？
    ├── 清晰度：复杂概念是否阐释清楚？
    └── 专业性：整体是否达到专业水准？
```

#### 学科差异的评测适配

| 领域 | 知识准确性重点 | 推理质量重点 | 表达水准重点 |
|------|--------------|------------|------------|
| **医学** | 诊断标准、用药指南、解剖知识 | 鉴别诊断、临床推理 | 医学术语、病历写作 |
| **法律** | 法条、司法解释、判例 | 法律适用、论证逻辑 | 法律文书、正式语体 |
| **小语种** | 词汇、语法、文化背景 | 语境理解、语用推理 | 语体、礼貌层级、方言 |
| **工程** | 技术规范、参数标准 | 设计决策、权衡分析 | 技术文档写作 |
| **金融** | 法规、产品条款、风险参数 | 投资分析、风险评估 | 报告规范 |

#### 从个案到共性问题

JD 要求"从个案中归纳出共性的体验问题与优化方向"——这需要：

```
单个错误回答（如：误诊了某种疾病）
    ↓
分析：知识错误？推理错误？表达误导？
    ↓
定位根因：模型对该疾病的诊断标准理解不足
    ↓
归纳共性：在类似疾病（同样需要鉴别诊断）上都表现不佳
    ↓
方向：构建"鉴别诊断"专项评测集 + 补充训练数据
```

### 2. 设计优化策略——从定义"好"到实现"好"

**核心问题**：给定评测发现的问题，如何设计最优的改进路径？

#### 优化策略设计

```
现状评测 → 短板定位 → 优化路径设计 → 数据/训练 → 验证
                                                    ↑
                                            可能需要多轮迭代
```

三种优化路径：

| 路径 | 适用场景 | 本层贡献 | 优点 | 缺点 |
|------|---------|---------|------|------|
| **数据路径** | 知识覆盖不足 | 标注高质量领域数据 | 精准、可控 | 成本高、规模化慢 |
| **训练路径** | 推理质量不高 | 设计训练任务和评测 | 从根本上改进 | 周期长、效果不确定 |
| **Prompt 路径** | 表达/格式问题 | 设计 system prompt | 快速见效 | 深度有限、不治本 |

#### "专家在回路"的优化循环

```
模型初始回答
    ↓
领域专家评估（三维度评分）
    ↓
定位具体缺陷（如：推理链条不完整）
    ↓
设计针对性方案（如：构造需要多步推理的示例数据）
    ↓
模型 DPO/RL 训练
    ↓
再次专家评估
    ↓（循环直到达标）
```

### 3. 建设高质量数据——"兼具实用与审美价值"

**核心问题**：如何生产出能让模型真正"变聪明"的高质量领域数据？

#### 数据质量层级

```
L3：审美价值          不仅能回答问题，还能展示领域专家的最佳实践
    ↑                 例：一份法律意见书 → 不仅是正确的，更是优雅的论证范例
L2：高质量            推理严谨、表达专业、覆盖边界
    ↑                 例：一个医学诊断 → 包含鉴别诊断、推理过程、文献支撑
L1：能用              事实正确、格式规范
    ↑                 例：一个翻译 → 语法正确、意思准确
    ↓（不可接受）
错误数据              事实错误、推理混乱、表达不当
```

#### 数据制作流程

```
需求定义：基于评测短板 + 用户需求 → 确定需要什么数据
    ↓
数据设计：定义"理想态"标准 → 设计数据规范和模板
    ↓
数据生产：领域专家按规范生产标注数据
    ↓
质量审核：多层专家审核（同行评议）
    ↓
数据交付：进入训练/评测数据池
```

与 JD 11 的数据管线相比，本层的核心差异在于：

| 环节 | JD 11（通用） | 本 JD（专业） |
|------|-------------|-------------|
| **数据生产者** | 自动化生成 + 人工校验 | **领域专家亲自生产** |
| **质量标准** | 用户满意度 + 任务完成度 | **领域专业标准 + 同行评审** |
| **质量控制** | 自动化验证 + 抽样人工 | **多层专家审核** |
| **数据规模** | 大规模（自动化+扩展） | **小批量高质量** |
| **知识深度** | 中等（通用知识） | **深度（专业领域）** |

## 核心技术能力

### 1. 领域专业判断力

这是最核心的能力。JD 要求：

> "能凭借自身专业积累，从知识准确性、推理质量、表达水准等多个维度对模型回答形成清晰的优劣判断，说明问题所在及原因"

这意味着：
- 不是"感觉模型回答不对"，而是能**精确定位**问题（是知识错了还是推理错了？）
- 不是"这个回答不好"，而是能**说明原因**（为什么不好？哪里不好？）
- 不是个案判断，而是能**归纳共性问题**（这类问题模型普遍表现不佳？）

### 2. 将专业判断转化为可度量的评测维度

领域专家的难点在于——如何将自己的直觉判断转化为**可操作、可复现的评测规范**：

```
"这个回答不好"                     ← 直觉判断，别人无法复用
    ↓
"回答在第二个推理步骤上跳跃了结论，没有考虑 X 和 Y 的替代解释"
    ↓                                  ← 可理解的诊断
"在这个领域的评测中，应加入'推理步骤完整性'维度"
    ↓                                  ← 可复用的评测维度
"评分标准：推理每缺失一个关键步骤 -1 分，最多 -3 分"
                                        ← 可度量的标准
```

### 3. 跨领域学习能力

JD 不限专业方向，意味着数据产品经理需要具备：

- **本领域深度**：在自己的专业方向有扎实积累
- **跨领域迁移**：将评测方法论从自己的领域迁移到其他领域
- **快速学习**：能快速理解一个陌生领域的基础知识和质量判断

## 关键挑战

### 1. 专家判断的主观性 vs 评测的可复现性

领域专家的判断本身可能存在差异：

```
专家 A：这个法律论证很好，逻辑严谨
专家 B：这个论证有缺陷，忽略了某个判例
    ↓
挑战：如何建立"专家间的共识标准"？
方法：
    1. 显式化评判维度（不是"好/不好"，而是三维度打分）
    2. 建立标杆样本（公认的"好"和"不好"的示例）
    3. 定期的专家校准会议
```

### 2. 专业数据的高成本 vs 规模化需求

专业领域数据的成本远高于通用数据：

| 成本项 | 通用数据 | 专业数据 |
|--------|---------|---------|
| **标注者** | 普通用户/标注员 | **领域专家** |
| **单价** | 低 | **高（10-100×）** |
| **产出速度** | 快 | **慢（需专家投入）** |
| **质量验证** | 自动化 | **专家交叉审核** |

这意味着专业领域数据工程需要**精打细算**——不能像预训练数据那样靠规模取胜，而是靠**每一条数据的质量**。

### 3. "审美价值"的客观化

JD 提出了"审美价值的数据"——这是一个难以客观化的概念：

```
如何定义一条医学回答具有"审美价值"？
    - 不仅仅是诊断正确（这是基本）
    - 还包含优雅的鉴别推理、清晰的解释、合适的预后建议
    - 让其他医生看了也会说"这个回答写得好"
    
挑战：审美标准因人而异，如何建立一致的"审美标准"？
```

这需要明确：**"审美价值"不是主观喜好，而是领域最佳实践的范式化**。

### 4. 模型 vs 专家——谁更权威？

当模型输出与专家判断不一致时：

```
模型：某种疾病的诊断标准是 X
专家：根据最新指南，诊断标准已经更新为 Y
    ↓
模型的知识落后了（训练数据截止日期问题）
    ↓
解决：通过最新领域的评测数据发现知识的时效性问题
```

但有时模型可能是正确的——专家可能没有跟上最新研究。这要求专家保持**开放心态**和**持续学习**。

## 工程哲学

### 1. "标尺"哲学——专家是质量的最后防线

> "领域专家的判断力是校准模型的标尺"

这背后是一种**质量优先**的工程哲学：
- 不信任自动评测在所有场景下都可靠
- 极端重要的领域（医学、法律）必须以人类专家判断为最终标准
- 自动化评测的目标是接近人类专家的判断力，而非替代它

### 2. "深度"优先于"广度"

与预训练数据工程追求大规模不同，专业领域数据工程信奉：

> **每条数据的质量 × 知识密度 = 对模型能力的贡献**

一条高质量的专家标注数据，可能比一万条自动爬取的数据对模型能力的提升更大——尤其是在高知识密度的专业领域。

### 3. "智识能力"——超越"知识问答"

JD 中使用了"智识能力"而非"知识"——这暗示了更高的要求：

| 概念 | 含义 | 评测方式 |
|------|------|---------|
| **知识** | 事实性信息（What） | 知识准确性评分 |
| **智识** | 理解、判断、推理、应用（Why/How） | 推理质量 + 表达水准评分 |

模型不仅在专业领域"知道得多"，还要**理解得深、推理得准**。

### 4. "每一个领域都可能是突破口"

不限专业方向——这意味着公司认为 AGI 的涌现可能来自于**任何一个领域的高质量数据注入**，而不是只有常见的大领域（如代码/数学）。

> 冷门领域的专家可能贡献了模型下一次"突破性能力"的关键数据——这是一种**探索型**的数据策略。

## 知识冲突

### 专家 vs 自动化——评测应该由谁主导？
- **专家派**：专业领域必须由领域专家评判，自动化不靠谱
- **自动化派**：专家不可规模化，必须用 LLM-as-Judge 替代
- **调和**：专家定义标准 + 校准标杆集，自动化评估运行在标杆校准的框架内，专家定期校验漂移

### 小批量高质量 vs 大批量中质量——哪个策略更好？
- **质量派**：每一条数据都有专家把关，宁缺毋滥
- **数量派**：先覆盖更多领域和场景，质量问题后续迭代修复
- **调和**：专业领域采用"质量派"，但通过数据设计（自动化生成+专家审核）提高效率

### 模型是否应该在某些领域"不回答"？
- **安全派**：模型在医学/法律等高风险领域应该免责声明，不提供具体建议
- **能力派**：既然用户来问，就应该尽可能准确地回答，否则模型的价值大打折扣
- **调和**：模型应该尽最大努力给出高质量回答，同时在不确定时明确说明局限性和置信度
