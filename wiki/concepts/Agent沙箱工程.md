# Agent 沙箱工程

## 定义

Agent 沙箱工程（Agent Sandbox Engineering）是设计、构建和运维 Agent 安全执行环境的系统工程体系。它解决的核心问题是：**如何在不可信、无监督的 Agent 自主运行时，保证宿主系统的安全、隔离、可靠与高效。**

> **Agent 沙箱 = 为 Agent 量身定制的"安全牢笼"——Agent 可以在里面自由执行，但不能破坏笼子本身。**

## 与 Harness Engineering 的关系

在 [[Harness_Engineering]] 的 8 核心元件框架中，Agent 沙箱工程是 **Safety Layer（#4）** 和 **Sandboxed Execution** 的物理实现层：

```
Harness 抽象层
┌─────────────────────────────────────┐
│ Agent Loop    │ Tool Registry        │  ← 应用层抽象
│ Context Mgr   │ Retry/Recovery       │
├─────────────────────────────────────┤
│ Safety Layer  │ Sandboxed Execution  │  ← 这一层的接口定义
├─────────────────────────────────────┤
│            Agent 沙箱工程            │  ← 物理实现层（DSec）
│  虚拟化  │  容器  │  存储  │  网络   │
└─────────────────────────────────────┘
```

> 如果把 Harness Engineering 比作"建筑设计图"，Agent 沙箱工程就是"施工队"——把 Safety Layer 设计落地为可运行的基础设施。

## Agent 沙箱 vs 传统云平台

| 维度 | 传统云平台 | Agent 沙箱平台 |
|------|----------|--------------|
| **工作负载** | 长稳服务（数周-数月） | 短生命周期（秒-小时）、频繁创建销毁 |
| **用户** | 人类运维（可预期行为） | Agent（不可预测、可能尝试逃逸） |
| **安全模型** | 防外部攻击 | 防内部 Agent 逃逸 + 防 Agent 间干扰 |
| **资源模式** | 固定配额、可规划 | 突发性、动态波动、释放后迅速回收 |
| **网络策略** | 基于租户的静态规则 | Agent 动态发现与连接、临时通信通道 |
| **数据隔离** | 租户间完全隔离 | Agent 间可能需受控共享（多 Agent 协作场景）|
| **故障模式** | 已知模式（硬件、OS、应用） | Agent 行为引入新故障类型（资源耗尽、无限循环等）|
| **可观测性** | 服务的健康监控 | Agent 行为轨迹 + 资源使用 + 安全事件 |
| **生命周期管理** | 手工或半自动部署 | 全自动创建/销毁、无人工介入 |

## 核心技术领域

### 1. 虚拟化环境

两种主要技术路线的取舍：

| 维度 | 虚拟机（VM） | 容器 |
|------|------------|------|
| **隔离强度** | ★★★★★（硬件级） | ★★★（OS 级） |
| **启动速度** | 秒级 | 毫秒级 |
| **资源开销** | 大（完整 OS 栈） | 小（共享内核） |
| **适用场景** | 高安全需求、多 OS、硬件直通 | 轻量级沙箱、高频创建销毁 |
| **逃逸面** | 小（Hypervisor 隔离） | 相对大（共享内核风险）|

> **混合策略**：大多数 Agent 沙箱平台采用 VM + Container 混合方案——VM 提供硬隔离边界，容器提供快速弹性。

### 2. 容器安全加固

| 技术 | 作用 | 威胁缓解 |
|------|------|---------|
| **Seccomp** | 限制系统调用 | 防内核逃逸 |
| **AppArmor/SELinux** | 强制访问控制(MAC) | 防文件系统逃逸 |
| **Capabilities 裁剪** | 移除不必要的 Linux Capability | 降权运行 |
| **Read-only RootFS** | 容器根文件系统只读 | 防写入恶意文件 |
| **User Namespace** | 容器内 root ≠ 宿主机 root | 提权防护 |
| **Cgroups v2** | 资源限制（CPU/内存/IO） | 防资源耗尽 |

### 3. 虚拟化网络

```
Agent A 沙箱                  Agent B 沙箱
┌────────────┐                ┌────────────┐
│  veth0     │                │  veth1     │
└────┬───────┘                └────┬───────┘
     │                             │
     ├──── OVS Bridge ─────────────┤
     │         │                   │
     │    VPC Overlay (VXLAN/Geneve)
     │         │                   │
     ├── eBPF / XDP ───────────────┤
     │         │                   │
     └───── 物理网络 ──────────────┘
```

核心挑战：
- **超大规模虚拟网络**：数千节点的 OVS/VXLAN 网络，控制面压力巨大
- **Agent 间通信**：允许受控的 Agent-to-Agent 通信（协作 Agent），但防止恶意连接
- **RDMA 支持**：多 Agent 训练场景需要高吞吐直接通信（NVLink/RoCE）
- **eBPF/XDP**：在虚拟网络中实现高性能包处理和安全策略

### 4. 临时存储

Agent 沙箱的存储需求与传统持久化服务截然不同：

| 需求 | 传统存储 | Agent 沙箱存储 |
|------|---------|--------------|
| **持久性** | 强持久（数据不可丢） | 沙箱生命周期内有效，销毁即清理 |
| **性能** | 均衡（读写兼顾） | 写密集型（Agent 生成大量临时数据）|
| **隔离** | 租户间隔离 | Agent 间完全隔离或受控共享 |
| **容量** | 固定配额 | 弹性、短时间可迸发大量写入 |
| **回收** | 手动或定期 | 沙箱销毁时快速回收空间 |

关键组件：分布式块存储（Ceph RBD）、共享文件系统（Lustre/GPFS）、本地 NVMe 缓存。

### 5. 可观测性与高可用

| 子领域 | 关键能力 | 典型工具/技术 |
|--------|---------|-------------|
| **指标监控** | 沙箱级资源指标（CPU/内存/IO/网络） | Prometheus + Node Exporter |
| **事件日志** | Agent 行为审计、安全事件、系统日志 | Loki / ELK |
| **分布式追踪** | 跨沙箱 Request 链路追踪 | OpenTelemetry + Jaeger |
| **告警** | 异常检测、资源过载、逃逸尝试 | Alertmanager + 自定义规则 |
| **自愈** | 节点自动上下线、沙箱重建、故障迁移 | Kubernetes Operator + 自定义控制器 |
| **应急响应** | 自动隔离受损 Agent、回滚配置 | Policy Engine + GitOps |

## 范式挑战：无监督 Agent 带来的安全新维度

### 传统云安全的假设（在 Agent 时代不成立）

```
传统：人类用户操作 → 行为可预测、可审计、可审批
     ↓
Agent：无监督自动执行 → 行为不可预测、执行极快、无法逐条审批
```

### 具体的新威胁面

| 安全挑战 | 描述 | DSec 解法方向 |
|---------|------|-------------|
| **沙箱逃逸** | Agent 利用内核漏洞突破隔离 | 多重隔离层（VM + 容器）+ Seccomp |
| **资源耗尽** | Agent 故意或意外消耗所有资源 | Cgroups 配额 + 动态限流 + 自动 kill |
| **数据偷渡** | Agent 试图将内部数据发送到外部 | 网络出口限制 + 数据防泄漏（DLP）|
| **Agent 间攻击** | 一个 Agent 入侵另一个 Agent 的沙箱 | 强网络隔离 + 最小权限通信 |
| **推理侧信道** | Agent 通过时序/缓存侧信道窃取信息 | 资源独立 + 定时随机化 |
| **无限递归** | Agent 不断 Fork 子 Agent 耗尽资源 | 进程树限制 + 沙箱嵌套深度限制 |
| **持久化 Rootkit** | Agent 在沙箱内安装持久化后门 | 沙箱只读文件系统 + 每次创建全新从镜像启动 |

## 工程哲学

### 1. "全系统栈"思维

JD 描述的能力路径：**自旋锁 → OS → 虚拟化 → 网络/存储 → 调度/控制面 → 跨区域一致性**

意味着 Agent 沙箱工程师需要具备**从 CPU 指令级到分布式系统级**的全栈理解，因为问题可能出现在任何一个层面。

### 2. 极致的资源效率

Agent 训练规模的扩张速度远超硬件增长。沙箱工程需要在保证隔离强度的前提下，**把每个沙箱的资源占用量压到最低**——包括减少内存开销、共享内核页缓存、使用精简镜像等。

### 3. 从"防人"到"防 Agent"的范式转移

传统云安全专注于防外部攻击和防用户误操作；Agent 沙箱需要**同时防 Agent 主动逃逸和 Agent 无意识的自我毁灭行为**——后者因为 Agent 的"智能性"和"自主性"，比传统场景更难预测。

## 知识冲突

### VM vs Container：哪种隔离更好？
- **VM 派**：硬件级隔离是唯一可信的沙箱方案，不能信任共享内核
- **容器派**：容器加 Seccomp/AppArmor 足够安全，且资源效率远超 VM
- **调和**：生产环境多采用**嵌套方案**——VM 提供首层隔离，VM 内部用容器提供弹性；或者用 MicroVM（Firecracker）兼顾两者优势

### 沙箱应该多大？
- **大沙箱派**：给 Agent 充足资源，避免限制模型能力发挥
- **小沙箱派**：轻量沙箱可高频创建、快速销毁，适合大规模并行训练
- **调和**：动态分级的沙箱对象池——根据 Agent 任务的"信用分"分配不同规格

### 可观测性的成本
- **全量派**：记录 Agent 所有行为，便于事后审计和调试
- **采样派**：全量日志成本过高，只应采样关键事件
- **调和**：自适应采样——正常行为按 1% 采样，异常行为 100% 记录+实时告警
