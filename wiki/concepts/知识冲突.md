# 知识冲突管理（Knowledge Conflict Management）

## 定义

知识冲突管理是**在个人知识库的长期演进过程中，系统化识别、记录和消解新旧知识之间的矛盾与张力的工程实践**。它解决的核心问题是：

> **当新知识挑战旧认知时，我们如何不盲目覆盖，也不固执保守，而是保留双方的合理性？**

在 LLM Wiki 知识库中，知识冲突不是需要消除的"错误"，而是**认知成长的证据**——每次冲突都意味着知识体系正在吸收新的、更深刻的见解。

> **知识的进步不是新知识替换旧知识，而是在保留旧知识合理性的同时，容纳新知识的洞察。**

每个概念页面以 `## 知识冲突` 节作为标准结构，记录该概念领域的核心争议。目前有 **18 个概念页面**包含此节。

### 冲突记录格式

每个冲突使用三段式结构：

```markdown

- **一方观点**：[立场 A 的论据]
- **另一方观点**：[立场 B 的论据]
- **调和**：[综合视角或临时结论]
```

### 冲突的价值

```
无冲突的知识库：
    所有观点一致 → 没有反思 → 停止成长
    → 适合"教科书式"的知识传播，但不适合前沿领域

有冲突的知识库：
    保留多方立场 → 触发反思 → 促进认知深化
    → 适合 AI/LLM 这样的快速演进领域
    → 每个冲突都是下一次突破的起点
```

## 核心方法论

### 三阶段冲突管理

```
阶段一：发现（Detection）
    场景：新知识在某个问题上与已有页面立场不同
    行动：不静默覆盖，标注冲突
    产出：在页面中新增 ## 知识冲突 节

阶段二：分析（Analysis）
    场景：冲突双方都是合理的，如何共存？
    行动：清晰陈述双方的核心论据
    产出：结构化的冲突记录（一方/另一方/调和）

阶段三：演进（Evolution）
    场景：随着对领域理解的加深，冲突可能消解
    行动：更新冲突记录——标记已解决、添加上下文
    产出：冲突的生命周期管理（活跃 / 搁置 / 解决）
```

### 冲突的类型学

| 类型 | 描述 | 处理方式 | 示例 |
|------|------|---------|------|
| **学术争论** | 尚未定论的前沿分歧 | 并存，追踪发展 | Scaling Law 的局限 |
| **工程策略** | 不同场景下的最优选择差异 | 并存，区分适用条件 | 去重粒度选择 |
| **路线之争** | 根本性的技术路线选择 | 并存，记录优劣 | 多模态 vs 文本扩展 |
| **哲学/价值观** | 不可调和的价值判断 | 明确立场差异 | AI 是否该有情感 |

## 工程哲学

### 1. "冲突不是债，是财富"

> 传统知识管理追求"一致性"——所有答案应该指向同一个方向。但在快速演进的前沿领域（如 AI/LLM），**冲突是认知深度的标志**。2024 年的"正确答案"在 2025 年可能被颠覆，保留冲突就是保留未来修正的余地。

### 2. "保留，而非覆盖"

本知识库最核心的冲突处理原则：

> **新旧知识冲突时，绝不停默覆盖。保留旧知识的合理性，同时为新知识留出空间。**

这意味着：
- 不因为一篇新文章就删除旧页面的内容
- 在冲突节中并置双方，而不是选择一方
- 为每方标明其适用条件和局限

### 3. "调和不是和稀泥"

三段式的"调和"不是必须消除分歧，而是：

```
好的调和：指出双方各自成立的场景（"在这个条件下 A 更优，在另一个条件下 B 更优"）
差的调和：含糊地说"两者都对"而不给出判断依据
```
