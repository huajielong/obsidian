# 预训练数据工程

## 定义

预训练数据工程（Pretraining Data Engineering）是围绕大语言模型预训练阶段的数据全生命周期管理工程体系。其核心任务是从人类已有的知识沉积中"开采"数据富矿，经过采集、清洗、加工、组织，最终转化为模型训练可用的高质量、规模化、多模态数据燃料。

> 预训练数据是驱动模型强大能力的核心燃料，数据质量直接决定模型能力的上限。

## 与三层工程模型的关系

预训练数据工程横跨三层工程模型的多个层次：

| 工程层 | 对应工作 | 说明 |
|-------|---------|------|
| **Context Engineering** | 语料清洗、去重、质量筛选、多模态数据组织 | 决定"喂给模型什么数据"——数据的纯度与多样性 |
| **Harness Engineering** | 数据采集环路、分布式处理框架、KV/消息队列、调度与容错 | 决定"数据怎么跑"——Pipeline 的吞吐与可靠性 |
| **底层系统层** | 数据湖、对象存储、分布式文件系统、性能调优 | 决定"数据存在哪、怎么存"——基础设施的 Scale |

> 三层工程模型原本以 [[Agent_Loop]] 为典型场景定义，而预训练数据工程将其应用于**非 Agent 的数据处理管道**，证明了三层模型在 LLM 基础设施中的普适性。

## 数据全生命周期

### 1. 数据发现与采集

- **全网数据选取策略**：链接质量预估、属性聚合、站点抓取配额、数据优先级
- **时效性管理**：面向 [[RAG]] 等场景的时效种子挖掘与调度
- **采集环路工程**：高吞吐、可容错的分布式采集系统（下载、任务调度、流式数据处理、DNS、连通性）
- **链接规模控制**：低质站群识别、镜像去重、URL 参数清洗、链接规范化

### 2. 数据清洗与加工

- **语料清洗框架**：构建稳定、高效、可扩展的大规模数据处理平台
- **核心去重技术**：
  - **MinHash 去重**：基于 Jaccard 相似度的近似去重，适合大规模文本
  - **向量去重**：基于 Embedding 的语义去重，精度更高但计算成本也更高
- **质量筛选**：语言模型质量评分、困惑度过滤、规则过滤
- **生产流水线**：增量处理、任务调度、数据统计、质量回归

### 3. 多模态数据处理

- **数据结构设计**：图像、视频、图文结合的结构化存储方案
- **全生命周期管理**：安全管控、自动化校验、质量治理
- **清洗与标注链路**：结合内容理解技术（CV 模型等）的去重、结构化、质量筛选、打标分析
- **分阶段适配**：为 Pretrain / SFT / RL 不同训练阶段设计差异化的数据组织方式

### 4. 数据存储与基建

- **核心数据组件**：键值数据库、消息队列
- **数据湖方案**：面向多模态海量数据的管理与存储架构（对象存储、分布式文件系统）
- **分布式处理框架**：支撑 PB 级数据的高效处理
- **性能调优**：CPU/内存/网络维度的吞吐与资源利用率优化

## 关键技术栈

| 技术领域 | 具体技术/工具 |
|---------|-------------|
| **数据处理框架** | Spark、DuckDB、Pandas、Ray |

| **分布式系统** | 分布式 KV、分布式文件系统、任务调度系统 |
| **模型框架（多模态）** | PyTorch、Megatron、vLLM |
| **内容理解** | CV 模型（用于多模态数据的去重/质量筛选） |
| **编程语言** | Python（主力）、C++/Rust（性能关键路径） |

## 核心挑战

### 1. 质量与规模的平衡
数据规模越大，噪声越多；但过度清洗可能损失长尾多样性。需要在召回率与精确率之间找到最优平衡点。

### 2. 去重中的语义保留
MinHash 以 token 级特征去重，无法理解语义相似但形式不同的内容；向量去重能补足这个缺陷，但计算成本高、调参复杂。

### 3. 多模态数据的异构性
文本、图像、视频的数据结构差异大，难以用统一的 Pipeline 处理。需要为每种模态设计专门的清洗链路，同时保持整体架构的一致性。

### 4. 多阶段训练的数据适配
Pretrain、SFT、RL 对数据的需求截然不同：
- Pretrain：海量、多样、覆盖广泛
- SFT：高质量、指令遵循、人机对齐
- RL：奖励信号明确的偏好数据

如何让同一套数据体系同时支持三个阶段，是架构设计的核心难题。

### 5. 数据 Pipeline 的可观测性
TB/PB 级数据的 Pipeline 中，任何一个环节出问题都难以排查。需要全面的日志、指标、监控、测试与调试工具。

## 知识冲突

### 去重粒度：精确 vs 近似
- **传统经验**：MinHash 在大规模文本去重中效果足够，效率最优
- **新趋势**：向量去重能捕获格式不同但语义重复的内容（如维基百科的不同翻译版本），未来方向是混合方案
- **共识**：两种方法互补而非替代，实际 Pipeline 中通常串联使用

### 数据量的"通货膨胀"
- **一方观点**：更大规模的数据始终能提升模型能力（Scaling Law）
- **另一方观点**：数据质量比规模更重要，"少而精"可能优于"多而杂"
- **调和**：数据质量筛选 + 规模化清洗并重，高质量数据的"规模化"才是关键
