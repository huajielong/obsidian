# 超算集群工程（Supercomputing Cluster Engineering）

## 定义

超算集群工程是**以大规模 AI 训练与推理为目标，设计、构建、运营超大规模（万卡→数十万卡）异构计算集群的系统工程实践**。

其核心使命是：

> **将电力高效转换为智能。** 通过系统性的硬件选型、架构设计、软件栈调优与运维自动化，最大化每一瓦电力到有效算力的转换效率。

超算集群工程是 [[Harness_Engineering]] 的**物理基础设施层在大规模分布式场景下的独立展开**——如果说 Harness Engineering 设计的是 Agent 运行的"规则"，那么超算集群工程设计的就是 Agent 运行的"场地"。

---

## 三层定位

在三层工程模型（[[Prompt_Engineering]] → [[Context_Engineering]] → [[Harness_Engineering]]）之外，超算集群工程属于**基础设施底座层**，与 [[Agent沙箱工程]] 构成互补：

```
Prompt Engineering                ← 怎么问
    ↓
Context Engineering               ← 给什么信息
    ↓
Harness Engineering               ← 怎么跑（控制逻辑）
    ↓
═══════════════════════════════════════  ← 应用层 / 基础设施层 分界线
    ↓
AI训练推理系统工程               ← 怎么训练/推理（并行策略/训练循环/推理调度）
    ↓
AI计算引擎工程                    ← 怎么算（算子/通信/编译器）
    ↓                         ↘
超算集群工程                      ← 怎么建成（集群架构/调度/网络）★ Build
AI集群可靠性工程                  ← 怎么持续运行（监控/故障/自愈）★ Run
    ↓
Agent沙箱工程                     ← 安全地跑（沙箱容器/VM）
```

### 与相邻概念的对比

| 维度 | AI训练推理系统工程 | AI计算引擎工程 | 超算集群工程 |
|------|-----------------|-------------|------------|
| **焦点** | 分布式训练/推理的**系统层面**编排 | 单卡/多卡间的**计算效率** | 集群级**资源编排** |
| **核心度量** | 训练吞吐（TFLOPs/utilization）、推理延迟/吞吐 | TFLOPS 利用率、通信带宽利用率 | 有效训练吞吐、集群利用率、PUE |
| **抽象层级** | GPU 节点 / 训练作业 / 推理请求 | CUDA 线程块 / Warp / Tensor Core | 物理节点 / 网络拓扑 / IDC |
| **技术栈** | Megatron-LM, DeepSpeed, vLLM, PyTorch | CUDA/NCCL/TileLang/MLIR | Kubernetes/Slurm/RDMA/散热 |
| **硬件距离** | 较远（通过框架抽象调用硬件） | 最近（直接与 GPU/NPU 指令集打交道） | 较远（与集群节点打交道） |

> 五者构成完整的**基础设施工程体系**：AI训练推理系统工程让计算**更协调**，AI 计算引擎工程让计算**更快**，超算集群工程让计算**更稳**，AI集群可靠性工程让计算**更可靠**，Agent 沙箱工程让计算**更安全**。

### 与 Agent 沙箱工程的对比

| 维度 | Agent沙箱工程 | 超算集群工程 |
|------|-------------|------------|
| **粒度** | 单 Agent 沙箱环境 | 万卡级集群系统 |
| **核心问题** | "Agent 在哪安全地跑一次？" | "大规模训练如何高效地跑完？" |
| **用户** | Agent（单个任务） | 训练作业/分布式推理（多任务） |
| **技术焦点** | 隔离/安全/资源配额 | 调度/网络/吞吐/可靠性 |
| **生命周期** | 秒-小时级（沙箱创建销毁） | 天-月级（集群持续运营） |
| **规模** | 单个 VM/容器 | 数千节点互联 |

### Build vs Run：超算集群工程与 AI 集群可靠性工程的互补

| 维度 | 超算集群工程（Build） | AI集群可靠性工程（Run） |
|------|------------|----------------|
| **核心问题** | "集群怎么设计、建成、上线？" | "集群怎么持续稳定运行？" |
| **时间焦点** | 建设期、架构设计、硬件选型 | 运营期、持续保障、故障响应 |
| **核心动作** | 设计、开发、选型、部署 | 监控、排查、修复、优化、预防 |
| **成功指标** | 系统是否能工作？吞吐是否达标？ | MTTR、可用性、自愈覆盖率 |
| **思维模式** | 建设者（Builder） | 守卫者（Guardian） |

> 一个优秀的超算集群中心需要同时具备这两种能力——但通常需要不同的团队和不同的思维方式。

## 五大技术域

### 1. 异构资源调度系统

将 CPU/GPU/NPU 等异构计算资源抽象、池化，实现拓扑感知的智能调度。

| 子领域 | 核心问题 | 关键技术 |
|--------|---------|---------|
| **资源抽象** | 异构算力统一表示 | 资源描述模型（YAML/CRD）、设备插件框架 |
| **拓扑感知** | 任务如何分配到最优位置 | NUMA 感知、NVLink 拓扑、PCIe Switch 拓扑 |
| **调度算法** | 吞吐/排队/利用率三角平衡 | Gang Scheduling、Backfilling、Binpacking、碎片整理 |
| **混部调度** | 训练与推理任务共存 | QoS 分级、抢占、cgroup 资源隔离 |
| **弹性调度** | 节点增减时任务自动适配 | Elastic Training Framework、动态重分配 |

### 2. 集群管理平台

覆盖任务和节点全生命周期的集群运营系统。

- **生命周期管理**：节点上线/下线/维修/退役全流程
- **日志与可观测性**：日志采集与召回、关键性能指标收集与可视化
- **故障管理**：故障发现 → 告警 → 自动容灾 → 根因分析
- **硬件导入**：新 GPU/网卡/交换机接入流程与验证
- **软件栈管理**：驱动（NVIDIA Driver/CUDA/HCCL/NCcl）/容器镜像/训练框架的版本管理与导入

### 3. 端到端性能调优

从集群视角进行全系统栈性能优化。

```
┌─────────────────────────────────────────────────────┐
│  应用层优化                                          │
│  Kernel Fusion / Flash Attention / Operator Tuning   │
├─────────────────────────────────────────────────────┤
│  通信优化                                            │
│  All-reduce 算法选择 / Overlap 计算通信 / 拓扑优化    │
├─────────────────────────────────────────────────────┤
│  内核/OS 优化                                        │
│  内核参数调优 / I/O 协议栈 / 中断亲和性 / 内存大页     │
├─────────────────────────────────────────────────────┤
│  网络优化                                            │
│  RDMA 配置 / 拥塞控制算法 / 多路径负载均衡             │
├─────────────────────────────────────────────────────┤
│  故障诊断                                            │
│  性能抖动分析 / 长尾延迟追踪 / 全链路排障              │
└─────────────────────────────────────────────────────┘
```

核心挑战：
- **性能抖动**：同构集群中节点间性能差异（Silicon Lottery）
- **长尾延迟**：All-reduce 同步中慢节点拖慢全体
- **性能不均**：租户间干扰、拓扑位置不公平
- **全链路排障**：训练变慢 → 是网络/存储/GPU/框架的问题？

### 4. 超大规模网络架构

AI 集群的"血管系统"——连接数千 GPU 的网络基础设施。

| 组件 | 技术选型 | 关键指标 |
|------|---------|---------|
| **网络拓扑** | Fat-Tree / Dragonfly+ / Dragonfly-Torus | 对分带宽、跳数、等分带宽 H&B |
| **通信协议** | InfiniBand NDR400 / RoCEv2 | 带宽（400G+）、延迟（<1μs）|
| **拥塞控制** | DCQCN / TIMELY / HPCC | PFC 风暴控制、ECN 标记阈值 |
| **路由策略** | 自适应路由 / Multi-path / ECMP | 负载均衡度、流量倾斜控制 |
| **多路径** | 多轨通信 / 路径冗余 | 有效带宽、故障容灾切换 |

并行策略的网络依赖：

| 并行策略 | 通信模式 | 网络需求 |
|---------|---------|---------|
| **数据并行** | All-Reduce（梯度同步） | 高带宽、低延迟 |
| **张量并行** | 点对点（P2P） | 低延迟（NVLink 优先）|
| **流水线并行** | 点对点（stage 间通信） | 带宽不敏感、延迟敏感 |
| **专家并行（MoE）** | All-to-All | 带宽密集型、拓扑灵活 |
| **序列并行** | Ring Attention | 高带宽、环形拓扑 |

### 5. 下一代架构规划

对新兴硬件的前瞻性评估与集群架构演进规划。

- **GPU 评估**：NVIDIA H100/B200/GB200/后续代际
- **国产加速器**：华为昇腾、寒武纪、百度昆仑、海光 DCU 等
- **网络硬件**：400G→800G→1.6T 演进、智能网卡/DPU 引入、P4 可编程交换机
- **存储分级**：Hot/Warm/Cold 三级存储、并行文件系统（Lustre/GPFS/WEKA）
- **IDC 协同**：供电密度（10kW/rack → 100kW/rack+）、液冷散热、布线规划

## 与 Harness Engineering 的深层关系

### Harness 8 核心元件的集群级映射

| Harness 元件 | 超算集群对应 | 说明 |
|-------------|------------|------|
| **① Agent Loop** | 作业调度生命周期 | 训练作业的"调度→执行→完成/失败"循环 |
| **② Tool Registry** | 资源配额与访问管控 | GPU/CPU/内存/网络的资源策略 |
| **③ Context Manager** | 分布式共享上下文 | 训练数据分发、模型状态同步 |
| **④ Safety Layer** | 作业隔离与容灾 | 作业间隔离、节点故障自动迁移 |
| **⑤ Retry / Recovery** | Elastic Training + Checkpoint | 节点故障时自动 checkpoint 还原 |
| **⑥ Telemetry / Observability** | 集群可观测性平台 | 节点/网络/GPU 性能指标全链路追踪 |
| **⑦ Eval Harness** | 大规模评测集群 | 多节点并行评测的调度与编排 |
| **⑧ Cost / Latency** | 集群能效优化 | 电力封顶、动态调频、利用率提升、PUE 优化 |

### 关键公式

> **有效训练吞吐 = 理论峰值 × 框架效率 × 通信效率 × 故障损失率 × 调度利用率**

超算集群工程的每个技术域都在优化这个公式中的一个或多个因子。

### Model-Harness-Fit 的集群维度

[[Harness_Engineering]] 中的 Model-Harness-Fit 概念可以扩展到集群层面：

| Model 能力演进 | 集群需求变化 | Harness 联动调整 |
|--------------|------------|----------------|
| 参数规模增长（100B→1T+） | 更多节点、更高带宽 | 并行策略重选（DP→TP→PP→EP） |
| MoE 架构普及 | All-to-All 通信暴增 | 网络拓扑优化、通信调度调整 |
| 长序列训练（128K→1M+） | Ring Attention 带宽需求 | 序列并行 + 通信 Overlap |
| 推理经济化（Speculative Decoding） | 推理集群与训练集群解耦 | 两级调度的协同编排 |

## 核心挑战

### 1. 规模复杂度（万卡 → 数十万卡）

集群规模每增长一个数量级，系统复杂度以超线性增长：

| 问题 | 万卡 | 数十万卡 |
|------|------|---------|
| MTBF（平均无故障时间） | 小时级 | 分钟级（必然有节点/网络故障）|
| 网络拓扑层级 | 2 层 | 3-4 层 |
| 负载均衡难度 | 单路径可控 | 多路径拥塞不可预测 |
| 调试难度 | 单任务可复现 | 多任务干扰+Heisenbug |
| 能耗管理 | 被动散热 | 主动功耗封顶+动态调度 |

### 2. 异构调度困境

单个集群中混有不同代际/不同厂商的加速器，需要考虑：
- **性能不均衡**：H100 vs B200 混部时，慢卡拖慢快卡
- **软件栈分裂**：NVIDIA CUDA vs 华为 CANN vs 寒武纪 BANG
- **通信协议不兼容**：NVLink vs HCCS 无法直接互联
- **人力投入**：每种加速器需独立的驱动/框架适配与性能调优

### 3. RDMA 网络的复杂性

RDMA 网络（无论是 InfiniBand 还是 RoCEv2）的部署和运维难度远高于传统 TCP/IP 网络：

- **PFC 死锁**：RoCEv2 依赖 PFC 流控，配置不当导致死锁
- **拥塞扩散**：单点拥塞通过 PFC 反压传播影响全局
- **Sharing Domain 隔离**：多租户场景下的 RoCE 网络隔离
- **调试困难**：RDMA 丢包/延迟问题难以定位（端到端不可见）

### 4. 故障管理的范式转变

在十万卡规模下，"故障不是异常，而是常态"：

```
传统运维思维：    集群健康 → 故障发生 → 人为修复 → 恢复
十万卡运维思维：   集群常态降级 → 自动容灾 → 弹性适应 → 持续优化
```

这要求从"反应式运维"转向**预测式运维**：
- 提前预测节点/GPU/网络故障
- 训练作业自动 checkpoint 迁移
- 拓扑自动重配置避开故障区域

## 行业格局

### 主要参与者

| 类型 | 代表 | 特点 |
|------|------|------|
| 云厂商 | AWS（EKS/SageMaker）、GCP（GKE/TPU）、Azure（ND-series） | 弹性、按需、多租户 |
| 超算中心 | 国家超算（无锡/广州/天津）、ORNL/LLNL | Linpack 传统强、AI 转型中 |
| AI 公司自建 | DeepSeek、字节跳动、Meta（AI Research SuperCluster） | 定制化、极致优化、全栈自建 |
| 硬件厂商 | NVIDIA（DGX SuperPOD）、华为（昇腾集群） | 交钥匙方案、生态锁定 |

### 与云原生 Stack 的关系

超算集群工程与云原生（Kubernetes 生态）正在走向融合：

| 维度 | 传统 HPC | 云原生 | AI 集群（融合态） |
|------|---------|-------|----------------|
| **调度器** | Slurm / LSF / PBS | Kubernetes | Volcano / Yunikorn / 自研 |
| **网络** | InfiniBand / OmniPath | Calico / Cilium / eBPF | RDMA over K8s + 智能网卡卸载 |
| **存储** | Lustre / GPFS | Ceph / Rook | 并行文件系统 + 对象存储分级 |
| **容器** | Singularity / Charliecloud | Docker / containerd | GPU-aware containerd（nvidia-container-toolkit）|
| **可观测性** | Prometheus / Ganglia | Prometheus + Grafana + OpenTelemetry | DCGM + 网络遥测 + Profile |
