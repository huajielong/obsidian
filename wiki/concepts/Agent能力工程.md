# Agent 能力工程（Agent Capability Engineering）

## 定义

Agent 能力工程是**通过设计 RL 训练环境、构建高质量评测任务、定位并补齐能力短板，系统性地提升模型在真实复杂场景中的 Agent 能力的工程实践**。它解决的核心问题是：

> **如何让模型从"能答对选择题"变成"能独立解决真实的工程问题"。**

Agent 能力工程站在 [[预训练数据工程]]（提供静态语料）和 [[AI训练推理系统工程]]（提供 RL 训练系统）之间——它负责的是"**喂给 RL 训练系统的内容是什么**"，以及"**怎么知道模型真的变强了**"。

> **预训练数据工程 ≈ 教材编写员**
> **Agent 能力工程 ≈ 教练 + 考官**
> 前者决定模型"知道什么"，后者决定模型"会做什么"。

## 在工程体系中的位置

### 完整定位图

Agent 能力工程是一个**跨层角色**——它不像其他工程概念那样只负责一个层次，而是横跨数据层、训练层和评估层：

```
产品化层     AI产品工程                          ← 怎么让技术变成产品
    ↑
能力构建层   Agent能力工程                       ← ★ 本概念：怎么塑造模型能力（跨多层）
    ↑
控制逻辑层   Harness Engineering                ← 怎么让 Agent 可靠执行
            Prompt Engineering / Context Engineering
    ↑
══════════ 应用层 / 基础设施层 分界线 ══════════
    ↑
框架层       AI训练推理系统工程                  ← 怎么训练（RL 训练框架）
计算层       AI计算引擎工程
集群层       超算集群工程 / AI集群可靠性工程
存储层       AI存储工程（横向底座）
硬件层       AI基础设施硬件工程 / AI数据中心工程
安全层       Agent沙箱工程
```

### 与其他概念的关系

```
                    预训练数据工程（静态语料）
                          ↑ 不同阶段
Agent能力工程（RL 环境 + 评测 + 训练数据）
    ↑ 使用训练框架
AI训练推理系统工程（RL 训练系统：GRPO/PPO）
    ↑ 能力被评估
Eval_Harness（评测执行框架）
    ↑ 推动产品上限
AI产品工程（Agent 产品体验）
```

| 维度 | 预训练数据工程 | Agent 能力工程 | Eval_Harness |
|------|-------------|--------------|-------------|
| **核心问题** | "模型该读什么？" | "模型该会什么？" | "模型会了吗？" |
| **数据形态** | 静态文本/图文 | **动态交互环境** | 评测题目 + 评分标准 |
| **时间点** | 预训练阶段 | **后训练/RL 阶段** | 全生命周期 |
| **输出** | 训练语料 | RL 环境 + 评测 + 能力数据 | 评测结果 + 能力报告 |
| **迭代方式** | 数据质量提升 | 环境设计实验 | 题目更新 |

## 三大核心职责

### 1. RL 环境构建——将真实场景转化为可训练的环境

**核心问题**：如何把真实世界中的问题（写代码、挖漏洞、做实验）变成一个模型可以在其中通过试错学习的环境？

#### 环境构建的三步抽象

```
真实场景（复杂、有噪音、不可重复）
    ↓ 1. 问题空间定义
保留核心挑战维度，去掉无关噪音
（例：不用真正的 GitHub 仓库，而是构造简化但保留关键难度的代码问题）
    ↓ 2. 交互接口设计
定义模型可以执行的动作空间
（例：读文件、写代码、运行测试、查看结果……）
    ↓ 3. 奖励信号设计
什么行为被鼓励、什么被惩罚
（例：代码通过测试 +1，引入安全漏洞 -5，运行失败 -2）
```

#### 三种策略的奖励设计

| 策略 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| **自动可验证** | 代码正确性、测试通过 | 客观、可复现 | 仅适用于结果可验证的场景 |
| **LLM-as-Judge** | 代码质量、设计合理性 | 灵活、覆盖更广 | 可能有偏见、成本高 |
| **人工反馈** | 安全策略、科学假设 | 最准确 | 成本极高、不可扩展 |

#### 环境设计实验

JD 特别提到"与研究员紧密合作探索不同环境设计对模型 Agent 能力的影响"——这意味着环境本身是一个**超参数**：

```
环境 A：只有通过/不通过两种结果
    vs
环境 B：给出多级分数（性能、正确性、代码风格）
    vs
环境 C：引入随机失败（模拟真实世界的不确定性）
        ↓
哪个环境设计能让模型学到更强的泛化能力？
```

### 2. 评测任务设计——量化模型在真实场景中的表现

#### 三方向评测框架

```
Code 评测：
    ┌─ 功能正确性（Unit Test 覆盖）
    ├─ 代码质量（可读性、性能、安全性）
    ├─ 工程能力（架构设计、模块化、错误处理）
    └─ 工具使用（Git、CI/CD、Debugger）

网络安全评测：
    ┌─ 漏洞发现（给定代码库找到漏洞）
    ├─ 漏洞利用（编写 PoC/Exploit）
    ├─ 安全加固（修复漏洞、加固系统）
    └─ 攻击链构建（多步组合攻击）

Science 评测：
    ┌─ 实验设计（从假设到方案）
    ├─ 数据分析（处理真实数据集）
    ├─ 工具使用（专业数据库、计算工具）
    └─ 可复现性（独立验证结果）
```

#### 评测设计的工程原则

| 原则 | 说明 | 反例 |
|------|------|------|
| **真实场景优先** | 问题来自真实用户数据/真实工程实践 | 从教科书中抄题目 |
| **可复现** | 相同输入应得到可验证的结果 | 依赖外部 API 或实时数据 |
| **渐进难度** | 从简单到复杂覆盖能力光谱 | 全是极端难题或全都太简单 |
| **抗 Reward Hacking** | 模型不能通过取巧绕过真正能力测试 | 只测正确率不看过程 |
| **自动化** | 评测可以大规模自动化执行 | 需要人工逐条判断 |

### 3. 能力短板补齐——从用户失败信号到训练数据

```
真实用户使用
    ↓
发现失败/低效场景
    ↓
定位模型能力缺陷（代码能力不足？安全理解不够？）
    ↓
针对性构建训练环境（模拟这类场景）
    ↓
模型在环境中RL训练
    ↓
验证能力是否提升
    ↓（循环）
```

#### 数据闭环

```
用户行为日志 → 失败模式聚类 → 能力缺陷定位
                                    ↓
                               构建训练场景
                                    ↓
模型 RL 训练 ←── 场景转化为 RL 环境
                                    ↓
                            Eval 验证能力提升
                                    ↓
                           部署 → 用户使用 → 再次收集日志
```

> **关键理念**："用真实世界的失败信号驱动能力提升"——不是从教科书或 benchmark 出发定义"应该会什么"，而是从用户实际使用中发现问题并针对性地解决。

## 三个专项方向的深度分析

### Code 专项

**核心能力要求**：
- AI 编程工具的重度使用经验（Claude Code、Codex 等）
- "有代码品味"——不仅会写，还知道什么是好代码
- 自驱力：独立发现并定义问题

**与本知识库的关联**：
- 与 [[Agentic_Coding]] 的关系：Code 专项负责让模型学会 Agentic Coding，而 [[Agentic_Coding]] 描述的是人类开发者使用 AI 的范式——两者从不同角度推动同一件事
- 与 [[Developer_Agentic_Workflow]] 的关系：模型需要学会的正是这个工作流

**典型工作示例**：
```
发现：模型在重构大型函数为小函数时经常引入 bug
定位：模型理解代码语义但缺乏"重构的安全边界"意识
方案：构建"安全重构"RL 环境——给出代码和重构要求，奖励正确的重构，惩罚引入 bug
验证：在 SWE-bench 重构类任务上的表现提升
```

### 网络安全专项

**核心能力要求**：
- 扎实的安全研究能力：漏洞发现、分析、验证、利用
- 覆盖 Web 安全、二进制安全、内核安全、浏览器安全、虚拟化安全等多个方向
- 攻击者视角 + 工程能力：自动化漏洞发现和 PoC 编写

**与现有知识体系的关系**：
- 这是十份 JD 中**唯一涉及网络安全**的角色
- 安全能力的 RL 训练是一个前沿方向——传统安全 benchmark（如 CyberSecEval）远不如代码 benchmark 成熟
- 安全场景的特殊挑战：**奖励信号设计极其困难**——"没有发现漏洞"不一定是因为安全，也可能是因为没找到

**典型工作示例**：
```
发现：模型在编写网络应用时经常引入 XSS 漏洞
定位：模型理解 XSS 概念，但在真实编码中不会主动考虑安全
方案：构建"安全代码评审"RL 环境——模型先写代码，然后被要求找到自己代码中的漏洞
验证：安全 benchmark 分数 + 真实代码中的安全缺陷率
```

### Science 专项

**核心能力要求**：
- 生命科学、计算生物学、化学、计算化学、基础医学相关背景
- 两年以上科研工作经验（实验或计算方向均可）
- 熟悉完整科研工作流：假设 → 实验 → 数据分析 → 结论
- 熟练使用 Python/R/SQL + 专业数据库（GEO/ChEMBL/PDB/UniProt）

**独特价值**：
- 这是十份 JD 中**唯一涉及科学研究领域**的角色
- 科学场景的 RL 训练是最前沿的方向之一——比代码和安全更早期
- 科学工作流的完整性使它可以作为评测"长链条推理"的天然场景

**典型工作示例**：
```
发现：模型在回答科学问题时能给出正确的理论知识，但不会设计实验验证假设
定位：模型缺乏"从理论到实验"的实践推理能力
方案：构建"实验设计"RL 环境——给出科学问题，模型需提出假设、设计实验、预测结果
验证：在科学 benchmark 上提升 + 专家评估实验设计质量
```

## 与 Harness Engineering 的深层关系

### Harness 8 核心元件的映射

| Harness 元件 | Agent 能力工程贡献 | 说明 |
|-------------|------------------|------|
| **① Agent Loop** | 训练环境定义 Agent 的行为空间 | RL 环境的好坏决定 Agent Loop 能学到什么 |
| **② Tool Registry** | 环境中的工具设计 | 模型在训练环境中使用的工具种类影响其泛化 |
| **③ Context Manager** | — | 间接（上下文管理不依赖本层） |
| **④ Safety Layer** | 安全专项直接贡献 | 模型的安能力本身就是 Safety Layer 的一部分 |
| **⑤ Retry / Recovery** | 环境中的失败体验 | 模型在 RL 环境中学会从失败中恢复 |
| **⑥ Telemetry / Observability** | 训练/评估数据的采集 | 模型行为数据本身就是能力评估的来源 |
| **⑦ Eval Harness** | **最直接的关系**——内容供给 | Eval Harness 是框架，评测任务是本层的产出 |
| **⑧ Cost / Latency** | 间接 | 能力提升不影响成本，但决定了"值这个价吗" |

### 与三层工程模型的关系

```
Prompt Engineering：         控制模型在单次交互中的行为
Context Engineering：        控制模型能看到什么信息
Harness Engineering：        控制 Agent 执行框架
Agent 能力工程：             ← 决定模型本身"会什么"——在更底层
```

Agent 能力工程不是在 Harness 之上或之下，而是**在 Harness 的侧翼**——它不是在执行层面优化，而是在能力层面塑造。

## 关键挑战

### 1. RL 环境的"真实性-可训练性"权衡

```
完全真实：   真实的 GitHub 仓库、真实的安全靶场、真实的实验室设备
    ↓ 太复杂、训练慢、噪音大、奖励信号难定义
高度简化：   简化的代码问题、模拟的攻防场景、虚拟的实验数据
    ↓ 太简单、学到的东西不泛化到真实场景
理想平衡点：  保留核心挑战复杂度 × 确保可训练性
```

### 2. 评测任务的"幸存者偏差"

如果评测任务只覆盖模型已经会的内容，就测不出真正的问题：

```
误区：用公开 benchmark → 模型刷题过拟合 → 分数高 ≠ 能力强
解决：基于真实用户失败数据 → 构建未知难度的评测任务
```

### 3. 三个方向的能力评估难以统一

| 方向 | 客观评估 | 自动化难度 | 领域专家依赖度 |
|------|---------|-----------|-------------|
| Code | 高（测试驱动） | 低（可自动） | 低 |
| 安全 | 中（PoC 可验证） | 中 | 中 |
| Science | 低（专家判断） | 高 | **极高** |

科学方向是三个中最依赖人工评估的——这使其规模化比代码和安全困难得多。

### 4. "奖励设计"的科学

奖励设计是 RL 训练中最微妙的部分：

- **奖励太稀疏**：模型探索很久才得到一次反馈，难以学习
- **奖励太密集**：模型学会"刷分数"而不是真正能力
- **奖励被 Hack**：模型找到奖励捷径（如输出更长的答案而非更正确的答案）
- **奖励冲突**：多目标（正确性 + 效率 + 安全）之间需要权衡

## 工程哲学

### 1. "能力"不是教出来的，是从试错中长出来的

预训练给模型知识（know what），RL 训练给模型能力（know how）。Agent 能力工程的核心信念是：

> **模型在真实环境的试错中学会的，远比从教科书中学到的更可靠。**

### 2. Code 是最务实的 AGI 度量标准

团队使命中的核心命题：**"模型在真实复杂场景中解决工程问题的能力是通用智能最务实的度量标准。"**

这背后的工程哲学：
- 学术 benchmark 可以被刷分 → 真实工程问题无法被刷分
- 代码执行有客观的正确/错误 → 不需要人工评判
- 工程问题的解决链条足够长 → 能测试模型的完整推理能力

### 3. 三个方向覆盖"人与数字世界交互"的全部维度

| 方向 | 交互维度 | 能力体现 |
|------|---------|---------|
| Code | 创造 | 在数字世界中**构建**东西 |
| Security | 保护 | 发现数字世界中的**脆弱点** |
| Science | 理解 | 探索**物理世界**的规律 |

三者合在一起，覆盖了一个通用智能体需要具备的全部核心能力。

### 4. 数据工程的"高阶形态"

如果说 [[预训练数据工程]] 是在已有的文本中"淘金"，Agent 能力工程是在：

> **设计金矿的开采方式 + 建造冶炼设备 + 检验黄金纯度。**

这不是数据 Pipeline 工程师，而是"能力塑造师"。

## 知识冲突

### Agent 能力 vs Harness 能力——谁更重要？
- **能力派**：模型能力是上限，Harness 是辅助——模型不会的事，再好的 Harness 也做不了
- **Harness 派**：Agent = Model + Harness——好 Harness 可以弥补模型能力的不足（Retry、Tool Use、Orchestration）
- **调和**：两者是乘法关系——能力 × Harness = Agent 效果，任何一方为 0 结果都是 0

### 三个方向是否需要同一团队管理？
- **统一派**：方法论相同（RL 环境 + 评测 + 数据闭环），只是领域不同
- **分拆派**：Code/安全/科学需要的领域知识差异太大，应该由不同团队负责
- **调和**：JD 显示放在同一团队下——说明公司认为**能力构建的方法论比领域知识更重要**

### 科学方向是否太超前？
- **实用派**：当前模型在科学上的 Agent 能力还很弱，投入产出比低
- **前瞻派**：科学能力是 AGI 的关键检验标准，现在不布局未来会落后
- **调和**：JD 中科学方向要求"有科研工作经验"而不是"顶尖科学家"——说明目前还在探索阶段，先做最核心的代码方向，科学是扩展
