# 多租户 SaaS 架构

## 定义

多租户（Multi-Tenancy）是一种**单份软件实例服务多个租户（客户/组织）**的架构模式。核心挑战是在**共享资源效率**与**租户隔离安全性**之间找到平衡——让每个租户感觉"这是我的专属系统"，而实际运行在同一套基础设施上。

在 AI Agent 平台场景中，多租户不仅仅是传统 SaaS 的隔离需求，还增加了 Agent 实例隔离、Memory 数据隔离、Skill 权限范围等 Agent 原生维度。

## 数据隔离模型

### 三种经典模式

| 模式 | 架构 | 隔离度 | 资源效率 | 运维复杂度 | 适用场景 |
|------|------|--------|---------|-----------|---------|
| **Database-per-Tenant** | 每个租户独立数据库 | ⭐⭐⭐ | ⭐ | ⭐⭐ | 对数据隔离要求高的企业客户 |
| **Schema-per-Tenant** | 同库不同 Schema | ⭐⭐ | ⭐⭐ | ⭐⭐⭐ | 中等隔离需求，MySQL 常见 |
| **Shared Schema** | 全共享，Tenant ID 字段区分 | ⭐ | ⭐⭐⭐ | ⭐ | 海量小租户，SaaS 标准模式 |

选择标准：**租户数据密度 × 合规要求 × 运维成本**。

### 混合策略（行业最佳实践）

```
策略: Database-per-Tenant for 大客户 + Shared Schema for 小客户

大租户（企业版）→ 专属数据库 → 最高隔离 + 自定义备份策略
小租户（免费版）→ 共享数据库 → Tenant ID 字段隔离 → 成本最优
```

## Agent 平台多租户的额外维度

传统 SaaS 多租户主要管"数据"，Agent 平台还要管**"行为"和"能力"**：

### 1. Agent 实例隔离

| 隔离级别 | 说明 | 适用场景 |
|---------|------|---------|
| **进程级** | 每个 Agent 独立进程/容器 | 高风险 Agent（可执行代码）、企业定制 Agent |
| **线程级** | 共享进程，独立 Context Window | 标准 Agent，资源效率优先 |
| **轻量级** | 同一 Context，按 Tenant ID 路由 | 读多写少的查询 Agent |

### 2. Memory 数据隔离

| Memory 类型 | 隔离策略 | 说明 |
|------------|---------|------|
| Working Memory | 全隔离 | Agent 当前对话上下文不允许跨租户访问 |
| Long-term Semantic | Schema-per-tenant | 知识库数据按 Tenant 前缀或 Collection 隔离 |
| Episodic Memory | 全隔离 | 执行历史/反思记录为租户私密数据 |
| Procedural Memory | 部分共享 | Skills/Tools 定义可全局共享，但可限制可见范围 |

### 3. Skill 权限范围

| 可见范围 | 谁可以创建 | 谁可以使用 | 示例 |
|---------|-----------|-----------|------|
| **全局（平台级）** | 平台管理员 | 所有租户 | 内置搜索 Skill、数学计算 Skill |
| **租户级** | 租户管理员 | 该租户所有成员 | 企业定制审批 Skill、内部知识库 Skill |
| **用户级** | 用户自己 | 仅创建者 | 个人快捷指令、私人 API 集成 |
| **角色级** | 租户管理员 | 特定角色成员 | 财务分析 Skill（仅财务角色可用） |

### 4. LLM API 资源配额

| 配额维度 | 控制方式 | 说明 |
|---------|---------|------|
| **Token 配额** | 日/月 Token 上限 | 防止单个租户耗尽预算 |
| **速率限制** | RPM/TPM 控制 | 确保多租户公平使用 |
| **模型访问控制** | 按 Plan 限制可用模型 | 基础版只用 Haiku，专业版可用 Opus |
| **并发限制** | 最大 Agent 实例数 | 控制资源占用 |
| **预算门控** | 超自动停或审批 ([[Cost_Aware_Budget_Gates]]) | 防止预算意外超支 |

## 认证与授权

### 多租户认证流程

```
用户登录
  → 识别 Tenant（通过 subdomain / 自定义域名 / API Key 前缀）
  → 加载该 Tenant 的认证策略（内部账密 / SSO / LDAP / OAuth）
  → 返回 Tenant-scoped JWT（含 tenant_id + roles + permissions）
  → 后续请求 Gateway 从 JWT 提取上下文
```

### 授权模型

| 模型 | 说明 | Agent 平台适配 |
|------|------|---------------|
| **RBAC** | 角色→权限映射 | 平台管理员 / 租户管理员 / 开发者 / 最终用户 |
| **ABAC** | 属性→权限策略 | Skill 可见范围、数据访问级别、模型可用性 |
| **ReBAC** | 关系→权限图 | Agent ↔ Tool ↔ Memory 之间的复杂关系（Google Zanzibar 范式） |

## 与同类概念的区分

| 概念 | 区别 |
|------|------|
| **多用户（Multi-user）** | 同一组织内多个用户，共用资源但权限不同 |
| **多租户（Multi-tenant）** | 多个组织的完全隔离，各自管理自己的用户 |
| **工作空间（Workspace）** | Dify 实现的多租户抽象，每个工作空间独立数据集/应用/配置 |

## 常见挑战与解决方案

| 挑战 | 解决方案 |
|------|---------|
| 租户间噪音干扰（Noisy Neighbor） | 资源配额 + 限流 + 独立 Pod/容器 |
| 数据泄露风险 | Tenant ID 强制注入 + SQL WHERE 层自动过滤 + 定期渗透测试 |
| Schema 升级复杂度 | 共享数据库时需向后兼容迁移；Database-per-Tenant 时需并行迁移 |
| 个性化配置膨胀 | 配置继承 + 覆盖模型（全局 → 租户 → 用户） |
| 计费复杂度 | 按 Token / 按 Agent 实例 / 按存储量 / 按用户数的多维计价模型 |
