# 后训练研究（Post-training Research）

## 定义

后训练研究是**在预训练基座模型之上，通过强化学习算法（RLHF/RLVR/PPO/GRPO）、高质量后训练数据和系统性评测的三元组驱动，最大化释放模型在复杂真实场景中的潜力的工程-研究混合实践**。它解决的核心问题是：

> **如何让基座模型从"渊博但笨拙"变成"既聪明又好用"？**

后训练研究位于 [[预训练研究]]（产生基座模型）和产品/能力层（使用对齐后的模型）之间——它是**基座模型释放层**，将预训练阶段积累的"知识势能"转化为"能力动能"。

> **预训练决定模型"知道什么"，后训练决定模型"能用什么"。一个知识渊博但无法对话的模型，其价值远低于一个知识相当但能协作的模型。**

### 模型能力链——从知识到价值的完整路径

```
数据（原始语料） → 预训练 → Base 模型（知识渊博但未对齐）
    │                              ↓
    │                    后训练（RLHF/GRPO 等）
    │                              ↓
    │                    对齐模型（可用、可靠、可控）
    │                    ┌────┬────┬────┬────┐
    │                    │SFT │RL  │DPO │... │
    │                    └────┴────┴────┴────┘
    │                              ↓
    │                    Agent 能力工程（RL 环境训练）
    │                              ↓
    │                    产品级模型（在真实场景中好用）
    ↓
最终价值：用户感受到的智能体验
```

### 与相邻概念的对比

#### 后训练研究 vs 预训练研究

| 维度 | 预训练研究 | 后训练研究 |
|------|----------|-----------|
| **输入** | 海量原始语料 | **Base 预训练模型** |
| **核心方法** | 架构设计、优化器、Scaling Law | **RLHF/RLVR/PPO/GRPO** |
| **产出** | Base 模型（知道很多） | **对齐模型（能做很多）** |
| **核心结构** | 双核：算法 + 数据 | **三元组：算法 + 数据 + 评测** |
| **评测定位** | Eval 是外部工具 | **评测是内部核心组件** |
| **Scaling Law** | 研究数据/模型/算力的缩放关系 | **研究奖励信号/训练步数的缩放关系** |
| **典型问题** | "怎么让模型知道更多？" | **"怎么让模型更听话、更能干？"** |

#### 后训练研究 vs Agent 能力工程

| 维度 | Agent 能力工程 | 后训练研究 |
|------|-------------|-----------|
| **角色** | "训练营建造者" | **"教练"** |
| **核心产出** | RL 环境、评测任务、弱点数据 | **训练算法、训练数据、评测体系** |
| **关注** | "模型在什么场景下训练？" | **"用什么算法训练？用什么数据？怎么知道效果？"** |
| **与模型的关系** | 提供训练内容（环境 + 任务） | **提供训练方法和数据（算法 + 数据集）** |
| **相互依赖** | 环境需要算法来驱动 | **算法需要环境来实现** |
| **类比** | 设计考试题目和考场 | **设计教学方法、出题策略、评分标准** |

#### 后训练研究 vs AI 训练推理系统工程

| 维度 | AI 训练推理系统工程 | 后训练研究 |
|------|-----------------|-----------|
| **核心关注** | "训练/推理怎么跑得快、跑得稳？" | **"训练/推理什么算法、什么数据效果好？"** |
| **技术栈** | 分布式并行、通信优化、显存管理 | **RLHF/PPO/GRPO、奖励设计、数据策略** |
| **产出** | 训练框架、推理引擎 | **训练算法方案、数据策略、评测标准** |
| **关系** | 后训练算法跑在训练框架之上 | **训练框架是算法工程化的基础设施** |

### 1. 算法方向——RL 算法的创新者

**核心使命**：在预训练模型和数据既定的条件下，通过更好的 RL 算法释放模型潜力。

#### 主流 RL 算法光谱

```
RLHF（OpenAI 经典范式）：
    Step 1: 人类标注偏好 → 训练奖励模型（Reward Model）
    Step 2: 用奖励模型打分 → PPO 优化策略
    ↓
    优点：可以建模复杂、主观的偏好
    缺点：奖励模型本身有偏差、训练不稳定、成本高

RLVR（可验证奖励学习）：
    不需要奖励模型，直接用可验证结果作为奖励
    └── 代码测试通过/不通过 → +1/0
    └── 数学答案正确/错误 → +1/0
    ↓
    优点：客观、无需奖励模型、训练稳定
    缺点：仅适用于结果可验证的场景

PPO（Proximal Policy Optimization）：
    策略梯度方法的核心算法
    └── Clip 机制防止策略更新过大
    └── Advantage 估计引导策略改进
    └── 价值网络（Critic）辅助训练
    ↓
    优点：经过广泛验证、稳定
    缺点：需要价值网络，计算量大

GRPO（Group Relative Policy Optimization）：
    DeepSeek 提出的改进方案
    └── 对同一 prompt 采样多个 response
    └── 在组内计算相对奖励
    └── 不需要价值网络（Critic）
    ↓
    优点：更简单、更高效（省去 Critic 大模型）
    缺点：对采样质量敏感
```

#### 后训练算法的关键挑战

```
Reward Hacking：
    模型找到奖励信号的捷径，而非真正学会能力
    → 示例：模型学会写"看起来正确"的代码，而非真正能运行的代码
    → 解法：对抗性奖励设计、多维度奖励组合

奖励稀疏：
    大多数训练步骤得不到有意义的奖励信号
    → 示例：长代码任务，只有最终测试通过/不通过
    → 解法：过程奖励模型（PRM）、课程学习

遗忘灾难：
    后训练 RL 过程中，模型可能遗忘预训练学到的知识
    → 示例：模型 RL 后代码能力提升，但通用知识下降
    → 解法：KL 散度约束、经验回放、多任务训练

泛化 vs 特化：
    在特定环境上训练过度 → 无法泛化到新场景
    → 示例：在 Codeforces 题目上 RL 后，不能解决真实 GitHub Issue
    → 解法：环境多样性、域随机化
```

### 2. 数据方向——后训练数据的设计者

**核心使命**：构建高质量、高多样性的后训练数据集，包括清洗、筛选、合成等全流程。

#### 后训练数据 vs 预训练数据

| 维度 | 预训练数据 | 后训练数据 |
|------|----------|-----------|
| **规模** | TB/PB 级 | **GB/TB 级** |
| **质量要求** | 海量、覆盖广泛 | **高质量、目标导向、多样性** |
| **标注** | 无需标注（自监督） | **需要标注/合成/筛选** |
| **数据类型** | 文本、图文、视频 | **指令对、偏好对、交互轨迹** |
| **来源** | 互联网爬取 | **人工编写、合成、用户反馈** |
| **价值密度** | 低（需大规模筛选） | **极高（每条都有明确目标）** |
| **与模型的关系** | 构建知识基础 | **塑造行为模式** |

#### 后训练数据的三类形态

```
指令数据（SFT Data）：
    格式：指令 + 理想回答
    用途：教会模型遵循指令、格式化输出
    示例：{"instruction": "用 Python 写一个快速排序", "response": "..."}
    关键挑战：覆盖场景的广泛性，回答质量的深度

偏好数据（Preference Data）：
    格式：指令 + 好的回答 + 差的回答（或排名）
    用途：训练奖励模型（RLHF）
    示例：{"prompt": "解释量子计算", "chosen": "...", "rejected": "..."}
    关键挑战：标注者的判断一致性，边界的清晰度

交互轨迹数据（Trajectory Data）：
    格式：指令 + 模型的多步思考/行动轨迹 + 最终结果
    用途：GRPO/PPO 训练，Agent 场景
    示例：{"task": "部署一个 Web 应用", "trajectory": [{"step": 1, "action": "...", "observation": "..."}]}
    关键挑战：轨迹长度控制、探索与利用的平衡
```

#### 自动化数据管线

JD 要求"设计并落地自动化数据清洗、评测及合成管线"——这不是一次性工作，而是持续运行的工程系统：

```
原始数据源（人工编写 / 用户反馈 / 模型生成）
    ↓
质量清洗（去重、过滤低质、格式标准化）
    ↓
数据合成（模型生成 → 筛选 → 质量检验）
    │
    ├── Self-Instruct：用强模型生成指令数据
    ├── Evol-Instruct：逐步复杂化指令
    ├── Rejection Sampling：采样多候选 → 筛选最优
    └── Data Augmentation：变换现有数据生成新变体
    ↓
数据评估（自动检查 + LLM-as-Judge + 人工抽检）
    ↓
训练数据集（准备好进入训练流程）
```

#### 数据方向的工程哲学

> **后训练数据的价值不在于"多"，而在于"精准"。一条精准暴露模型缺点的数据，胜过一万条模型已经会了的数据。**

### 3. 评测方向——能力诊断的设计者

**核心使命**：设计能真正反映模型能力边界与短板的评测方案，精准定位问题驱动优化。

#### "刷 Benchmark" vs "真正评测"

```
"刷 Benchmark" 的典型特征：
    ✓ 在标准化测试集上运行
    ✓ 得到一个漂亮数字
    ✓ 报告 "我们的模型在 X 上 SOTA"
    ✗ 但不清楚这个数字在真实场景中意味着什么
    ✗ 不知道模型在哪些场景下系统性地失败

"真正评测" 的要求（本 JD）：
    ✓ 设计目标场景的评测维度
    ✓ 构建能暴露模型短板的测试
    ✓ 关注模型在哪失败，而不只是在哪成功
    ✓ 将评测结果与产品体验关联
    ✓ 评测本身是可迭代、可改进的
```

#### 后训练评测的三层体系

```
L1：标准化 Benchmark（"体检"）
    业务：MMLU, HumanEval, GSM8K, MATH 等
    目的：快速了解模型的基础能力基线
    频率：每个训练 checkpoint
    自动化程度：完全自动

L2：场景化评测（"专科检查"）
    业务：Agent 场景（工具调用/多步推理）、写作场景、专业领域
    目的：评估模型在目标场景中的表现
    频率：每次重大训练迭代
    自动化程度：自动 + LLM-as-Judge 混合

L3：能力边界探测（"极限测试"）
    业务：设计极端/边界/对抗性测试案例
    目的：发现模型的系统性短板
    频率：迭代中持续进行
    自动化程度：高度依赖人工设计
```

## 后训练三阶段方法论

### 阶段一：SFT（Supervised Fine-Tuning）

**目标**：让 Base 模型学会遵循指令和格式化输出。

```
输入：Base 模型（只会续写文本）
输出：SFT 模型（能听懂指令并作答）

核心数据：高质量指令-回答对
关键挑战：
    - 指令覆盖的广泛性
    - 回答的质量和深度
    - 避免"过拟合"到固定格式

与预训练的关系：SFT 是"教模型听懂人话"，不改变模型的知识量。
```

### 阶段二：RL（Reinforcement Learning）

**目标**：让模型在开放任务中通过试错提升能力。

```
输入：SFT 模型（能听懂指令但能力有限）
输出：RL 模型（能在复杂任务中探索和改进）

核心方法：PPO / GRPO / RLVR
关键挑战：
    - 奖励设计（客观+主观的平衡）
    - 训练稳定性
    - 探索与利用的平衡

与 Agent 能力工程的关系：RL 训练需要环境（Agent 能力工程提供），
本层提供 RL 算法和数据策略。
```

### 阶段三：对齐（Alignment）

**目标**：让模型的行为符合人类价值观和期望。

```
输入：RL 模型（能力强但可能不听话）
输出：对齐模型（能力强且符合要求）

核心方法：RLHF 偏好训练、Constitutional AI
关键挑战：
    - 什么是对的"价值观"？
    - 安全与自由的平衡
    - 过度对齐导致能力下降

与安全的关系：对齐是 AI 安全的第一道防线。
```

> **SFT 教模型"说人话"，RL 教模型"解决问题"，对齐教模型"做正确的事"。三者共同构成完整的后训练流程。**

## 核心能力

### 1. 从算法到工程的完整经验

JD 要求"具备从算法设计到工程落地的完整经验"——这不是纯学术角色：

```
纯学术：在论文中提出新算法，在 toy 设置下验证
    ↓
工程+学术（本角色）：提出新算法 → 在数千 GPU 上跑通 → 验证效果 → 上线
    ↓
纯工程：实现他人设计的算法 → 优化性能 → 上线
```

### 2. 数据直觉——"从复杂多源的数据中提炼信号"

后训练数据不像预训练数据那样"多即是好"——质量是核心。数据直觉意味着：

```
看一条数据：能判断"这条数据对模型有用吗？"
看一个数据集：能判断"这个数据集覆盖了哪些能力？缺什么？"
看一个训练过程：能判断"模型是从哪条数据学到了这个能力？"
看一个失败案例：能判断"是算法问题还是数据问题？"
```

### 3. 评测的方法论思维

> "对模型评测有方法论层面的思考"

这意味着**不把评测当成跑分工具，而是当成理解模型的窗口**：

```
跑分工具视角：
    评测 → 跑分 → 报告结果

方法论视角（本 JD）：
    设计评测 → 分析结果 → 理解模型行为 → 定位短板
    → 改进数据/算法 → 重新评测 → 验证改进
    → 评估评测本身是否需要改进
```

## 关键挑战

### 1. Reward Hacking——RL 训练的根本难题

```
RL 训练的本质：模型优化奖励函数
但如果奖励函数设计不当，模型会走捷径：
    - 代码场景：模型学会输出"看起来"正确的代码，而非能运行的
    - 写作场景：模型学会用华丽辞藻打分高，但空洞无物
    - Agent 场景：模型学会"假装"完成任务
    
解法：
    - 对抗性奖励设计
    - 多维度奖励组合
    - 过程奖励模型（PRM）
    - 持续更新评测数据集
```

### 2. 后训练阶段的数据"毒化"

```
预训练数据污染后训练评测是一个已知问题：
    - 模型在预训练阶段"见过"评测题目
    - 后训练评测时得到虚高的分数
    - 看似能力提升，实则是记忆匹配
    
解法：
    - 使用非公开评测集
    - 动态生成评测题目
    - 交叉验证不同来源的评测结果
```

### 3. 通用性 vs 专业性的平衡

```
后训练的一个核心矛盾：
    要让模型在某场景表现出色 → 需要在该场景大量训练
    但大量训练可能导致模型在其他场景能力下降
        ↓
    解法：
    - 多任务联合训练
    - 课程学习（先通用、后专业）
    - 持续监控全场景能力
```

### 4. GRPO/PPO 的超参数敏感性

```
RL 训练对超参数极其敏感：
    - 学习率：太大训练不稳定，太小训练慢
    - KL 散度系数：太大限制探索，太小导致遗忘
    - 奖励缩放：影响策略更新的剧烈程度
    - 批次大小：影响梯度估计的方差
    
    每个超参数都需要针对具体场景调优。
```

## 工程哲学

### 1. "释放"而非"构建"

后训练研究的核心理念与预训练不同：

> **预训练研究是"构建知识"——从零开始让模型知道更多。**
> **后训练研究是"释放潜力"——让模型已经知道的变得可用。**

这意味着后训练不是在教模型新知识，而是在激活和引导已有知识。这种视角影响所有设计决策：

```
如果模型不理解用户的指令 → 不是它不知道，是它没学会"怎么回答"
→ 后训练 SFT 数据教它"格式"

如果模型不会写代码 → 不是它不懂编程语言，是它没学会"怎么在交互环境中写代码"
→ 后训练 RL 环境让它"练习"

如果模型输出不安全内容 → 不是它不知道安全规则，是它没学会"什么该说什么不该说"
→ 后训练对齐数据告诉它"边界"
```

### 2. "算法 × 数据 × 评测"的三元组不可分割

JD 将三条职责放在同一职位下，反映了一个深刻的认知：

> **算法、数据、评测不是三个独立的环节，而是互相依赖、互相定义的同一件事的三个面。**

```
没有好的评测：不知道算法改进是否有效
没有好的数据：算法有再好的设计也缺乏训练材料
没有好的算法：再好的数据也无法充分利用
    ↓
三者必须在一个团队中紧密耦合
```

这正是与 [[预训练研究]]"双核"结构的演进关系——后训练增加了"评测"作为第三极，因为评测在后训练阶段比预训练阶段**更关键、更场景化**。

### 3. "模型既能解决最难的问题，也能读懂每一个意图"

这是本 JD 使命中最打动人的一句话：

> **"使得模型既能解决最难的问题，也能读懂每一个意图，更能作为自主 Agent 在复杂真实世界中可靠地行动。"**

```
"解决最难的问题" → 能力的上限（算法 + 数据的质量）
"读懂每一个意图" → 理解的深度（对齐 + 评测的精准度）
"在复杂真实世界中可靠地行动" → 信用的保证（安全 + 稳定性）

三者缺一不可——这正是后训练研究的使命。
```

## 知识冲突

### RLHF vs GRPO——哪种方法更优？
- **RLHF 派**：奖励模型可以建模复杂偏好，PPO 是经过广泛验证的稳定框架
- **GRPO 派**：无需价值网络更高效，组内相对奖励减少了 reward hacking
- **调和**：不同场景选择不同方法——有明确答案的场景用 RLVR/GRPO，主观偏好场景用 RLHF

### 后训练应该多"重"？
- **轻量派**：后训练只是"微调"，核心能力在预训练阶段已经确定
- **重量派**：后训练的效果极其显著，投入更多资源在后训练上可以带来质的飞跃（如 DeepSeek-R1 的 RL 训练）
- **调和**：预训练决定能力天花板，后训练决定能达到天花板多近——两者同等重要

### 评测应该由后训练团队还是独立团队做？
- **内嵌派**（本 JD 的立场）：评测是后训练的核心内部组件，必须与算法/数据深度耦合
- **独立派**：评测应该由独立团队做，避免后训练团队"既当运动员又当裁判"
- **调和**：日常迭代评测由后训练团队内嵌，重大版本评测由独立团队或外部交叉验证

### 自动化数据合成 vs 人工数据——哪个质量更高？
- **合成派**：大规模+自动筛选能产生足够高质量的数据，成本低、速度快
- **人工派**：人工编写的质量更高、更可控，自动化合成的数据有"模型偏见"问题
- **调和**：合成数据做规模和多样性，人工数据做质量和标杆——两者混合最优
