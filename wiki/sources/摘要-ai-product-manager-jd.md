# AI 产品经理职位描述

## 来源概览

- **职位**: AI 产品经理
- **公司信号**: DeepSeek（产品团队）
- **招聘方向**: AI 产品方向、Agent Harness 产品方向
- **团队使命**: "站在模型与世界之间，将最前沿的智能能力转化为千万用户触手可及的产品体验"

---

## 关键洞察

### 洞察 1：产品团队——"站在模型与世界之间"的最上层

这是十份 JD 中**唯一一个非工程/非基础设施角色**，但它在整个堆栈中占据了**最顶端的位置**：

```
世界（用户/场景/需求）
    ↑
AI产品工程                     ← ★ 本 JD：将技术转化为产品
    ↑
Prompt Engineering            ← 怎么问
Context Engineering           ← 给什么信息
Harness Engineering           ← 怎么跑（Agent 控制逻辑）
    ↑
══════════════════════════════  ← 应用层 / 基础设施层分界线
    ↑
AI训练推理系统工程            ← 怎么训练/推理
...                            ← 以下全为基础设施层
```

> 如果把此前八份 JD 比作"造火箭的工程师"，这份 JD 就是"决定火箭飞向哪里的产品负责人"——两者缺一不可。

### 洞察 2：两个方向——面向用户 vs 面向 Harness

| 维度 | AI 产品方向 | Agent Harness 产品方向 |
|------|-----------|----------------------|
| **用户** | 终端用户（亿万级） | 开发者/研究员/开源社区 |
| **产品形态** | Chat 网页端、APP 端、开放平台 | Harness 产品（Agent 执行环境）|
| **核心指标** | 留存率、使用深度、转化率 | 任务完成率、开发者满意度 |
| **技术深度要求** | 深度使用 AI 产品 | 理解 Agent 完整技术栈 |
| **工作模式** | 数据驱动 + 用户洞察 | 社区驱动 + 研究员协作 |
| **竞争参照** | ChatGPT、豆包、Kimi 等消费级产品 | Claude Code、Manus 等 Agent 产品 |

两个方向共享"产品经理"这一头衔，但所需的领域知识和工作方式截然不同——前者是**消费级 AI 产品经理**，后者是**基础设施产品经理**（在 AI 公司语境下）。

### 洞察 3：AI 产品方向——数据驱动的四层能力模型

```
第一层：产品规划
    ↓  技术跟踪 → 策略制定 → 优先级管理
第二层：功能设计
    ↓  数据分析 + 用户反馈 + 定性观察 → 问题拆解
第三层：项目管理
    ↓  跨团队协作 → 快速迭代 → 高质量交付
第四层：产品品味
    ↓  对交互细节的敏感度 + 对极致体验的追求
```

核心要求中的"是 AI 产品深度用户，不只是看过，而是自己用过、拆过"——这与其他 JD 的"素质优先"招聘哲学一致，强调**真实体感**胜过**简历经历**。

### 洞察 4：Agent Harness 产品方向——行业罕见的角色定位

这是十份 JD 中**最具独特性的角色之一**。其职责范围远超传统产品经理：

1. **定义无法量化的核心指标**——"如何衡量'Agent 是否真的在更多场景下更深入的帮助到更多的人'"是一个开放性问题，需要从用户行为中提取可操作信号
2. **产品化基础设施**——将 [[Harness_Engineering]]（一个内部系统基础设施）转化为面向开发者的产品，需要同时理解技术深度和用户体验
3. **连接研究员与用户**——"与模型训练团队的研究员深度沟通与合作，实现模型与 Harness 的共同进化"——产品经理成为模型研究员和终端用户之间的翻译层
4. **构建社区**——维护 Harness 产品用户社群，从海量用户反馈中提取产品迭代信号

> 这个角色的存在本身就是一个强烈的信号：**该公司认为 Agent 基础设施不是一个内部工具，而是一个需要产品化的核心资产。**

### 洞察 5：Harness PM 需要理解完整 Agent 技术栈

JD 中列出的技术要求构成了一幅完整的 Agent 知识图谱：

```
LLM API → KV Cache → Agent Loop → Tool Use → Reasoning
    → Planning → Skills → MCP → Memory → Subagent → Multi-Agent
```

同时要求对**三层工程模型**有第一手实践：
- [[Prompt_Engineering]] — 提示词工程
- [[Context_Engineering]] — 上下文工程
- [[Harness_Engineering]] — 夹具工程

这意味着 Harness PM 不是"懂点技术的产品经理"，而是**具备工程思维的产品领导者**——能理解技术细节、能参与技术讨论、能做出有技术根据的产品决策。

### 洞察 6："Vibe Coding"与系统性数据方法论的矛盾统一

JD 同时要求两种看似矛盾的能力：

| 能力 | 描述 | 体现什么 |
|------|------|---------|
| **Vibe Coding** | "能够使用 vibe coding 写代码，不一定需要技术背景" | 拥抱 AI 辅助编程，降低技术门槛 |
| **系统性数据方法论** | "设计系统性的收集数据的方法（包括问卷、访谈、A/B测试、灰度测试等），并使用统计学的工具严谨科学的分析数据" | 严谨的科学思维，不靠直觉拍脑袋 |

这种"矛盾统一"恰恰是 AI 时代产品经理的典型画像——**既要有创造力（用 AI 快速原型），又要有科学性（用数据验证决策）**。

### 洞察 7：产品方向与 Harness 方向的 UI/UX 共性

两个方向都要求：

> "具备 UI/UX 设计素养，能够在 AI 辅助下完成产品原型图设计、UI 设计等相关的产品设计工作"

这揭示了一个重要信号：**该公司要求产品经理具备设计能力，但接受"在 AI 辅助下完成"**——不要求是专业设计师，但要求能用 AI 工具跨过"从想法到原型"的鸿沟。这与其他 JD 中的"vibe coding"要求一脉相承。

### 洞察 8：跨 JD 分析——产品层完成知识体系拼图

将此 JD 放入十份 JD 的全景对比中：

```
产品化层     AI产品经理（AI产品方向 / Agent Harness方向）      ← ★ 新增
    ↑
应用层       Harness 团队（研究/工程/产品/PM）
服务端工程   AI 搜索工程   预训练数据工程
    ↑
═══════════════ 分界线 ═══════════════
    ↑
框架层       训练推理框架工程
    ↑
计算层       算子/通信/编译器工程
    ↑
集群层       超算集群工程 / AI集群可靠性工程 / AI平台运维
    ↑
存储层       AI存储工程（横向底座）
    ↑
硬件层       AI基础设施硬件工程
    ↑
设施层       AI数据中心工程
    ↑
安全层       Agent沙箱工程
```

**完整堆栈终于闭合**——从最底层的物理基础设施（数据中心/硬件/沙箱）到最上层的产品体验（产品经理），十个 JD 构建了一个纵向贯穿的 AI 公司技术-产品全栈体系。

> **核心启示**：一个 AI 公司的竞争力不仅仅是模型能力，而是从数据中心到产品体验的完整链条——链条上的每一个环节都需要专门的工程/产品团队来运营。
