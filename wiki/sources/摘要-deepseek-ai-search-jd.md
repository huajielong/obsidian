# DeepSeek AI 搜索算法 / 架构工程师职位描述

> **来源**：DeepSeek 官方招聘信息（2026 年），由 Claude 获取并整理。
>
> **核心定位**：**搜索是 AGI 感知世界的原生感官。** 我们正在打造下一代 AI 原生的通用搜索引擎——不再只是返回十条蓝色链接，而是让每一次提问都被真正理解。

---

## 团队使命

探索 LLM 与搜索引擎的深度融合，让搜索成为 LLM 理解世界的重要基础设施：任何语言、任何模态都能被平等地检索。

> **构建极致简洁 & 轻量的 LLM-based 检索系统，承载全球海量的搜索请求，在成本、延迟与效果之间寻找最优平衡。**

---

## 两个招聘方向

| 方向 | 角色定位 | 核心 Pipeline 环节 |
|------|---------|-----------------|
| **AI 搜索算法方向** | 核心搜索算法优化 | Query 理解 → 召回 → 排序 → 索引筛选 → 质量评估 |
| **AI 搜索架构方向** | 大规模检索系统构建 | 在线召回/排序/策略模块 → 离线索引构建管线 → 系统调优 |

## 各方向详情

### 1. AI 搜索算法方向

**工作职责：**
- 核心搜索算法优化：query 理解、召回、排序、索引筛选、质量评估，让搜索结果更准确、权威、实时，更适合 LLM 理解
- 多语言、多场景、多模态搜索能力建设
- 搜索质量评估体系构建：自动化评估、badcase 分析
- 与架构、数据、产品、posttrain 等团队紧密协作
- 跟进搜索、RAG、LLM、多模态等方向技术进展

**硬性要求：**
- 至少熟悉机器学习、深度学习、信息检索中的一个方向
- 对搜索和大模型结合感兴趣
- 良好的工程落地能力与意识，在成本与延时约束下落地算法
- 沟通直接、协作意识强

**加分项：**
- 有搜索、推荐、广告、RAG、chatbot 等大规模在线系统经验

### 2. AI 搜索架构方向

**工作职责：**
- 构建支持千亿级数据、千万日活用户的大规模信息检索系统
- 开发并优化在线召回、排序、策略模块，以及离线索引构建管线——在性能、成本与效果之间做系统性权衡
- 与算法、数据团队协作，支持新模型、新算法快速落地
- 探索搜索新架构、新范式，为 Agent 提供更强大的信息检索工具

**硬性要求：**
- 熟练掌握 C++ 或 Rust，熟悉 Python、SQL 等常用语言，工程能力强
- 熟悉数据密集型系统开发和性能优化，有良好的系统设计品味

**加分项：**
- 熟悉大规模分布式向量检索技术（稠密/稀疏/多向量），熟悉 RDMA 高性能通信技术
- 熟悉 PyTorch、CUDA 等模型推理技术，了解 KVCache 存储系统，有模型部署和推理加速经验
- 有大规模搜索/推荐/广告/多模态检索系统研发经验
- 有个人开源项目作品，深度参与过大型开源软件研发，发表过高水平学术论文

## 关键洞察

### 1. "AI 原生搜索" vs "传统搜索+AI"

| 维度 | 传统搜索 + AI | AI 原生搜索 |
|------|-------------|------------|
| **核心范式** | 检索→排序→呈现（十条蓝链接） | 理解→检索→推理→直接回答 |
| **Query 理解** | 关键词匹配 + LTR 排序 | LLM 深度语义理解 + 意图识别 |
| **排序依据** | 文本相关性 + 点击信号 | 语义相关性 + 时效性 + 权威性 + LLM 偏好 |
| **输出形式** | 搜索结果列表 | 结构化答案 / 综合摘要 / 可直接供给 LLM |
| **模态支持** | 以文本为主 | 文本 + 图片 + 视频 + 代码全模态平等检索 |
| **成本模型** | 索引 + 检索的计算成本 | 检索成本 + LLM 推理成本（需系统性权衡）|

### 2. 三层工程模型中的定位

| 工程层 | AI 搜索中的对应 |
|-------|---------------|
| **Context Engineering** | 检索结果是 LLM 生成回答的**上下文来源**——搜索是 Select 子问题的极致实现 |
| **Harness Engineering** | 搜索系统本身的 Pipeline（召回→排序→过滤）是一个**离线/在线混合的 Harness**：容错、调度、性能监控 |

> AI 搜索本质上是 **Context Engineering 的 Select 子问题在搜索引擎尺度下的独立成体系**——从 RAG 的"从有限文档库中检索相关信息"，
> 放大到"从整个互联网/全球知识中找到最相关的信息"。

### 3. 成本、延迟与效果的三元平衡

JD 反复强调"在成本、延迟与效果之间寻找最优平衡"——这构成了 AI 搜索工程的核心模型：

```
效果（Accuracy/Recall）
    ↑
    |    ↗ 理想点
    |   /
    |  /  
    | /   
    0 ──────────→ 延迟（Latency）
    |
    ↓
  成本（Cost）
```

三个约束相互制约：
- 更好的效果通常需要更复杂的模型 → 更高的延迟和成本
- 更低的延迟需要剪枝/量化 → 可能牺牲效果
- 更低的成本需要更轻量的模型 → 在效果和延迟上让步

### 4. 搜索即 RAG 的超集

| 维度 | RAG | AI 原生搜索 |
|------|-----|-----------|
| 数据源规模 | 有限文档库（MB-GB） | 全网/全互联网级（PB） |
| 索引管理 | Vector DB | 分布式搜索引擎 + 向量索引 |
| 排序策略 | 向量相似度 + 简单重排 | 多层排序 LTR + LLM reranker |
| 实时性 | 取决于索引更新频率 | 秒级/分钟级时效性要求 |
| 离线处理 | Embedding + Chunk | 索引构建 + 预处理 + 质量评分 |

AI 搜索解决的是 **RAG 在互联网规模下遇到的 Scale 问题**。

### 5. "极致简洁 & 轻量"的设计哲学

JD 强调系统要"极致简洁 & 轻量"，这与 DeepSeek Harness 团队的"不做过度设计"一脉相承，反映了 DeepSeek 整体工程文化：

- **轻量** = 少依赖、低开销、快速迭代
- **简洁** = 清晰的数据流、可 Debug 的 Pipeline、不堆复杂度
- 在深度学习的"大力出奇迹"时代保持系统工程的克制
