# 大模型训练/推理框架工程师职位描述

> **来源**：互联网公开招聘信息（2026 年），由 Claude 获取并整理。公司来源未明确标注。
>
> **核心定位**：AGI 的基础设施工程——构建高效、稳定、可扩展的分布式训练推理系统，支撑模型从能力验证到大规模落地的全生命周期。

---

## 团队使命

AGI 的成功，离不开对极致工程系统的追求。致力于为 AGI 构建基础设施，打造高效、稳定、可扩展的工程系统，支持前沿探索的持续迭代、先进模型的大规模训练、以及面向全社会的极致性价比服务。产出陪伴着每一代模型从能力验证一路走向大规模落地，长期、持续为下一代智能应用提供坚实基础设施底座。

---

## 工作职责（五个方向）

| 方向 | 核心工作 | 关键技术 |
|------|---------|---------|
| **分布式训练推理系统** | 模型并行、长上下文、MoE、低精度训练推理 | Megatron-LM / DeepSpeed / FSDP，并行策略（TP/PP/DP/EP/SP），混合精度（FP8/BF16）|
| **RL 训练系统** | 异步 RL、Agent RL、OPD（Online Policy Distillation）| RLHF / GRPO / PPO，Reward Model 训练，在线策略蒸馏 |
| **多模态训练系统** | 模型结构搭建、性能显存优化、数据读取加速 | 多模态融合架构、Activation Checkpointing、数据预取管线 |
| **推理服务系统** | KV Cache 磁盘缓存、负载均衡策略 | vLLM / TGI / SGLang、PagedAttention、Prefix Caching |
| **算法协作** | 与算法团队协作，支持新模型、新算法快速落地 | 快速原型、性能 Profiling、模型-框架 Co-design |

- 基础扎实，学习能力强，愿意深入复杂系统和底层细节
- 代码能力强，代码品味好，**擅长与 Code Agent 合作**

## 关键洞察

### 1. 框架层是"调度指挥官"——与 AI 计算引擎的明确分工

本 JD 与 [[摘要-hpc-operator-comm-compiler-jd]] 的分工可以这样理解：

```
AI计算引擎工程              → 提供"武器"（算子/通信/编译器）
    ↓ 调用
训练/推理框架工程（本 JD）  → 决定"战术"（并行策略/训练循环/推理调度）
    ↓ 服务
算法团队 & 上层应用         → 决定"战略"（模型架构/产品形态）
```

| 维度 | AI计算引擎工程 | 训练/推理框架工程 |
|------|-------------|-----------------|
| **关注点** | 单个 kernel 的物理极限 | 全系统训练/推理的吞吐和稳定性 |
| **抽象层级** | CUDA 线程块 / Warp / Tensor Core | GPU 节点 / 分布式拓扑 / 训练作业 |
| **核心问题** | "这个算子怎么跑最快？" | "整个训练怎么在 1000 张卡上跑稳跑快？"|
| **时间尺度** | 微秒级（kernel 延迟） | 天-周级（训练作业）、毫秒级（推理延迟）|
| **输出形态** | 算子库 / 通信库 / 编译器 | 训练框架 / 推理引擎 / RL 系统 |

> **两层的边界**：AI计算引擎工程优化的是 `torch.matmul(A, B)` 这个调用有多快；训练/推理框架工程优化的是 "整个 training step 怎么在 1024 张 GPU 上 3 秒内完成"。

### 2. "RL 训练系统"——后训练时代的核心战场

JD 中专门列出 RL 训练系统（异步 RL、Agent RL、OPD），这是一个**强烈的行业信号**：

| 训练阶段 | 传统方法 | RL 增强方法 |
|---------|---------|------------|
| **Pre-training** | 语言建模 (next token prediction) | 同上，框架成熟 |
| **Post-training** | SFT（监督微调）| RLHF / GRPO / 在线策略优化 |
| **推理时扩展** | 静态推理 | 推理时 RL Search / MCTS / Self-Refine |

RL 训练系统相比传统训练有独特的工程挑战：

| 挑战 | 说明 | 框架需求 |
|------|------|---------|
| **异步性** | Policy 模型和 Reward 模型异步执行 | 异步调度、队列管理 |
| **Agent RL** | Agent 在环境中交互生成训练数据 | 环境管理、轨迹录制、Reward 计算 |
| **OPD** | 在线策略蒸馏：大模型知识蒸馏到小模型 | 双模型管理、KL 散度计算 |
| **训练稳定性** | RL 训练比 SFT 更容易发散 | 梯度裁剪、Reward Normalization |
| **计算效率** | 每个 step 需同时跑 inference + training | 资源调度、计算-通信 Overlap |

### 3. "MoE" 和 "低精度"——模型架构驱动的框架创新

JD 强调 MoE 和低精度训练推理：

**MoE 对框架的影响**：
- 传统的 DP/TP/PP 不够用了，需要 **Expert Parallelism (EP)**
- All-to-All 通信成为瓶颈，需要通信调度和负载均衡
- 动态 Expert 加载/卸载需要框架支持
- 辅助 Loss 和容量因子需要框架原生支持

**低精度（FP8/INT8/INT4）对框架的影响**：
- 训练：FP8 训练需要框架管理 Scale 参数、Overflow 检测
- 推理：INT4/INT8 量化需要 Calibration 和 Kernel 选择
- 混合精度策略（哪些层 FP16、哪些层 FP8）需要框架自动决策

### 4. "KV Cache 磁盘缓存"——推理系统的前沿课题

JD 中推理方向特别提到 KV Cache 磁盘缓存和负载均衡，反映了推理系统的最新趋势：

| 推理优化技术 | 解决的问题 | 框架支持 |
|------------|-----------|---------|
| **PagedAttention** | KV Cache 显存碎片 | vLLM 核心创新 |
| **Prefix Caching** | 共享前缀的 KV Cache 复用 | vLLM / SGLang 支持 |
| **KV Cache 量化** | 压缩 Cache 降低显存 | FP8/INT8 KV Cache |
| **KV Cache 磁盘缓存** | 跨 session 持久化，冷启动加速 | **前沿方向**（本 JD 重点） |
| **负载均衡** | 多推理节点间的请求分配 | Request Router、Copy-Async 策略 |

KV Cache 磁盘缓存是一个**非常前沿**的方向——它让推理服务器可以在磁盘上持久化 KV Cache，跨请求复用。这对长对话、RAG 场景的推理延迟有巨大改善，但涉及高速 I/O、缓存一致性、LRU 淘汰策略等系统难题。

### 5. "擅长与 Code Agent 合作"——Harness Engineering 的文化渗透

JD 在核心要求中明确写入了 **"擅长与 Code Agent 合作"**，这是一个值得注意的信号：

- AI 辅助开发不再只是"加分项"，而是**核心要求**
- 团队的工作方式预期已经包含 Agent（Claude Code / Cursor / Copilot 等）
- 这与 [[Harness_Engineering]] 中的"高强度用户"门槛完全一致
- 队员不仅要用 Agent，还要**擅长**——能写出 Agent-friendly 的代码、能设计 Agent 能安全修改的系统

### 6. 竞赛信号延续——与前面 JD 属于同一招聘体系

加分项中出现了与之前两份 JD **完全一致的竞赛列表**——SC/ASC/ISC/PAC + IOI/NOI/ICPC：

| 本 JD | [[摘要-hpc-cluster-engineer-jd]] | [[摘要-hpc-operator-comm-compiler-jd]] |
|------|------|------|
| SC / ASC / ISC / PAC | SC / ASC / ISC / PAC | SC / ASC / ISC / PAC |
| IOI / NOI / ICPC | CMO / NOI / IOI / ICPC | NOI / IOI / ICPC |
| 排名前 10% | — | — |

这**几乎可以确定三份 JD 来自同一家公司**——它们共享同一套人才过滤器和文化信号。

### 7. "复现过论文/项目"——从读者到创造者的门槛

加分项中的"复现过 LLM、RL、多模态相关论文/项目"不同于其他 JD 的"开源贡献"。它寻找的是：

- 能**从论文到代码**的人（不是只会读论文或只会写代码）
- 能处理论文中没写清楚的细节（这是最难的）
- 能对比原始实现和复现版本的性能差异
- 对模型训练/推理有**动手经验**而非只有理论知识
