# AI 算力集群性能与可靠性工程师职位描述

> **来源**：互联网公开招聘信息（2026 年），由 Claude 获取并整理。公司来源未明确标注。
>
> **核心定位**：AGI 时代的大规模算力系统可靠性工程师——不是传统运维，而是让十万卡集群成为稳定可靠生产力的系统思考者。

---

## 团队使命

AGI 时代的竞争，本质上也是大规模算力系统性能与可靠性的竞争。团队的价值，是让巨大而复杂的 AI 超算集群成为稳定可靠的生产力，让模型的每一次训练和推理都建立在坚实可靠的算力底座之上。

---

## 四个核心挑战

| 挑战 | 核心目标 | 关键能力 |
|------|---------|---------|
| **1. 守护"吞金兽"** | 让十万卡集群像一台电脑一样稳定；缩短 MTTR；每次故障交出根因分析 | 硬件故障定位、全生命周期管理、根因分析 |
| **2. 新算力"开箱即巅峰"** | 高质量交付每一批资源；新节点基线检查、性能调优与生产验证 | 性能基准测试、差距分析、验证流程设计 |
| **3. 把"黑盒"变透明** | 建设面向 AI 超算集群的可观测性体系；告警先于业务投诉到达 | 监控体系设计、信号降噪、核心指标提炼 |
| **4. 用自动化替代人工** | 让集群自己救自己；自动化运维工具链；故障自愈能力 | 自动化方案设计、故障场景覆盖、边界判断 |

## 关键洞察

### 1. "Build" vs "Run"——与超算集群工程的互补分工

本 JD 与 [[摘要-hpc-cluster-engineer-jd]] 的关系是**同一个硬币的两面**：

```
超算集群工程（设计/建设）         ≠     本 JD（运营/可靠性）
  设计网络拓扑                           让网络持续可靠
  实现调度系统                           让调度不崩溃
  选择硬件                               让硬件持续输出峰值
  开发集群管理平台                        运营集群管理平台
  ↑ "造"                                ↑ "守"
```

| 维度 | 超算集群工程 | 本 JD（可靠性工程） |
|------|------------|-------------------|
| **时间焦点** | 建设期、架构设计 | 运营期、持续保障 |
| **核心动作** | 设计、开发、选型 | 监控、排查、修复、优化 |
| **成功标准** | "系统能不能工作" | "系统能不能持续可靠工作" |
| **面对的问题** | 如何构建 | 坏了怎么办、怎么不再坏 |
| **思维模式** | 建设者思维 | 守卫者思维 |

> 一个优秀的超算团队需要同时具备这两种能力——但通常需要不同的人、不同的思维方式来完成。

### 2. "十万卡规模下，故障不是异常，而是常态"

JD 中最震撼的数据推演：

> 如果每张卡每年发生一次故障，平均每天就有近三百个故障需要处理。

这重新定义了运维的思维范式：

```
传统运维假设：        集群是健康的，故障偶尔发生
十万卡运维现实：      集群是常态降级的，没有"全健康"的时刻

MTBF 推导：    100,000 张卡 × 1 次/年 ÷ 365 天 ≈ 274 次/天
                     ≈ 11 次/小时
                     ≈ 每 5 分钟一次
```

这要求：
- **自动化不是加分项，是生死线**——人工无法响应 300 次/天的故障
- **告警必须降噪**——300 次/天中哪些是真正的业务影响？
- **故障模式库**——积累常见故障的诊断-修复脚本
- **自愈而非告警**——让系统自己修复大多数故障

### 3. 三个核心指标选三个——可观测性的哲学

JD 中有一个独到的问题：

> 如果只给你 3 个核心指标来判断集群整体健康度，你选哪三个？

这是一个极好的练**抽象能力**和**系统性思考**的面试题。可能的答案（非标准）：

| 核心指标 | 为什么选它 | 它揭示什么 |
|---------|-----------|-----------|
| **有效训练吞吐（Effective TFLOPs Utilization）** | 训练是集群的核心产出，这个指标直接反映产出效率 | 框架、通信、计算、I/O 的综合表现 |
| **节点级健康度（Node Health Score）** | 聚合所有硬件/OS 层面的报警和性能偏离 | 硬件故障、驱动问题、固件缺陷 |
| **作业完成率（Job Completion Rate）** | 训练作业是否正常运行到终点 | 综合可靠性表现（网络/存储/计算） |

### 4. "开箱即巅峰"——新硬件验收的工程化

JD 第二个挑战聚焦于新硬件验收流程——这是集群工程中最容易被忽视但极其重要的环节。

一个理想的验收流程：

```
到货 → 物理检查（外观/固件版本） → 基准测试（节点级/网络级/存储级）
    → 稳定性测试（72h 压力） → 生产模拟（跑典型训练任务）
    → 对比分析（实际 vs 理论峰值） → 交付生产
```

每个环节回答不同的问题：
- 节点级：GPU 性能是否达标？内存带宽是否正常？
- 网络级：RDMA 延迟和带宽是否符合规格？有没有链路丢包？
- 存储级：并行文件系统的读写吞吐是否达到预期？
- 训练模拟：在实际训练负载下，新节点的性能与已有节点是否一致？

### 5. "值班工程师不再被半夜电话吵醒"——自动化的四个层次

```
L1：告警通知             ← 故障发生了，告诉人
L2：自动化诊断           ← 故障发生了，系统自动定位原因
L3：自动化修复           ← 故障发生了，系统自动修复（重启/切换/降级）
L4：自动预防             ← 系统在故障发生前自动调整避免故障
```

本 JD 的目标是达到 L3-L4——让集群自己救自己。

### 6. "从比特到砖块"的完整系统地图

JD 要求的"系统性地技术视野"本质上是一幅完整的硬件→软件映射图：

```
加速卡 (GPU/NPU)         ← 性能数据、温度、功耗、NVLink 带宽
    ↓
NVSwitch / NVLink        ← 拓扑、带宽、错误率
    ↓
网卡 (NIC / HCA)         ← RDMA 带宽、延迟、丢包率
    ↓
网络交换机                ← 拥塞、降速、链路状态
    ↓
存储 (分布式文件系统)      ← I/O 带宽、延迟、元数据性能
    ↓
调度系统 (Slurm/K8s)     ← 排队延迟、调度效率、资源分配
    ↓
训练框架 (PyTorch/等)     ← GPU 利用率、通信效率、Checkpoint 耗时
```

### 7. 与其他五份 JD 的文化信号对比

| 维度 | 前五份 JD | 本 JD |
|------|----------|-------|
| **语言风格** | 技术、系统、结构清晰 | **感召、反传统、面向新人的叙事** |
| **核心要求** | Rust/C++/Python、系统能力 | **素质（热情/逻辑/视野）> 经验** |
| **对经验的态度** | 硬性技术栈要求 | "不太在意你有没有经验" |
| **竞赛信号** | SC/ASC/ISC/PAC + NOI/IOI/ICPC | **无**（完全不同的招聘体系）|
| **学术信号** | OSDI/FAST/VLDB（部分）| **无** |
| **面试流程** | 技术面试 | **项目深挖 + 故障推演 + AGI 愿景** |

> **结论**：这份 JD 与前五份的文化信号完全不同——它很可能来自**同一个公司的不同团队**（团队文化差异）或**完全不同的公司**。
