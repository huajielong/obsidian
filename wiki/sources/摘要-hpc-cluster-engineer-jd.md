# 超算集群研发工程师职位描述

> **来源**：互联网公开招聘信息（2026 年），由 Claude 获取并整理。公司来源未明确标注。
>
> **核心定位**：定义并建设下一代超算基础设施，将电力高效转换为智能的引擎。从万卡到数十万卡集群的系统架构演进。

---

## 团队使命

在 AI 时代，超算集群就是把电力转换为智能的引擎。它是通向 AGI 的坚实底座。

从万卡集群去往数十万卡的路上，丛生着系统复杂度的荆棘。团队目标，是跑满所有计算资源，榨干每一份算力。这牵涉着异构算力的调度权衡、全系统栈的性能调优、超大规模网络的互联承载，以此探索下一代集群的更优架构。

每一层设计都需要从第一性原理出发，重新审视，重新思考。让每一瓦电力都尽可能多地转换为有效算力，让每一次模型训练和推理都扎根于极致高效的算力底座之上。

## 工作职责（五个方向）

| 方向 | 核心工作 | 关键技术 |
|------|---------|---------|
| **异构调度系统** | 大规模异构计算资源调度，CPU/GPU/NPU 抽象、池化与拓扑感知调度，在吞吐/排队/利用率间取得均衡 | 资源调度算法、拓扑感知、异构资源抽象 |
| **集群管理平台** | 任务与节点生命周期管理、日志采集与召回、性能数据收集与可视化、故障发现与自动容灾、新硬件/新软件栈导入 | 集群管理、可观测性、自动化运维 |
| **端到端系统性能调优** | 从集群视角的 CPU/GPU/NPU 计算优化、Linux 内核、I/O 和网络协议栈优化；构建可观测性体系；大规模训练故障诊断、性能回归、全链路排障；分析性能抖动、长尾延迟、性能不均等系统性瓶颈 | 内核调优、性能工程、可观测性、全链路排障 |
| **超大规模网络架构** | 集群网络拓扑设计、路由策略、多路径负载均衡、拥塞控制；RDMA（RoCEv2 / InfiniBand）协议栈深入研究；与通信算子和存储团队配合提供最优底层支撑 | RDMA、RoCEv2、InfiniBand、拥塞控制、自适应路由 |
| **下一代集群架构规划** | 参与新一代超算集群规划与建设；评估 GPU、国产 AI 加速器、网卡、新兴硬件；与 IDC 团队配合驱动数据中心规划交付 | 硬件评估、IDC 协同、架构规划 |

## 岗位要求

### 基本要求
- 扎实的计算机体系结构基础：深刻理解计算机组成、操作系统、计算机网络等核心原理
- 从硬件微架构、访存子系统到片上和网络互联架构，从数理逻辑到进程指令再到操作系统执行，能够系统性地思考和定位计算规模大幅提升后带来的性能优化与复杂性问题
- 熟悉 C / C++ / Rust / Python 之一，具备优秀的设计能力、极强的动手和工程能力以及代码质量意识
- 对分布式系统有深刻理解与实践经验（分布式任务调度与资源编排，或高性能、高可用的系统架构设计）
- 对性能优化有较高热情，发自内心地想要跑满所有计算资源，榨干每一份算力
- 良好的中文沟通能力和团队协作能力

### 加分项
- 熟悉容器运行时（cgroup、namespace 等）与容器编排系统（Kubernetes 等）的底层实现原理，理解资源隔离、调度（sched_ext）与 QoS 等机制
- 精通 RDMA 编程及 RoCEv2 / InfiniBand 网络架构，深入理解拥塞控制、多路径、自适应路由等机制，具备大规模 RDMA 网络排障与调优经验
- 有 HPC 和超算全机规模应用经验，熟悉并行编程范式，或有超大规模集群组网设计、部署建设、调优经验
- 有智能网卡 DPU、可编程交换机 P4 开发经验，或熟悉 GPU、国产 AI 加速器或国产 CPU 的体系结构和性能特性
- 在大型开源软件中有深度贡献，或有优秀个人开源项目，或有 HomeLab 经历
- 在 HPC 赛事（SC / ASC / ISC / PAC）、数学或算法竞赛（CMO / NOI / IOI / ICPC）中取得优异成绩

## 关键洞察

### 1. 从"万卡到数十万卡"——超算集群的系统复杂度跃迁

JD 开篇就点出了核心挑战：**从万卡到数十万卡不是线性扩展，而是系统性复杂度的阶跃**。这涉及：

| 维度 | 万卡规模 | 数十万卡规模 | 核心挑战 |
|------|---------|------------|---------|
| **调度** | 单任务独占为主 | 多任务异构混部 | 拓扑感知调度、资源碎片管理 |
| **网络** | 单层 Fat-Tree | 多级 Clos/ Dragonfly+ | 拥塞控制、多路径负载均衡 |
| **故障** | 节点级故障常态化 | 链路/交换机/网络分区多重故障 | 故障预测、自动容灾、梯度保存恢复 |
| **性能** | 单节点优化即可见效 | 全系统协同优化 | 长尾延迟、性能抖动、All-reduce 不均 |
| **能耗** | kW 级 | MW 级 | PUE、功耗封顶、动态调频 |

### 2. "异构"是核心主题，不是口号

JD 中反复出现的 **CPU/GPU/NPU 异构** 反映了真实趋势：
- **GPU**：NVIDIA H100/B200/GB200 主导训练，CUDA 生态成熟
- **NPU**：国产 AI 加速器（华为昇腾、寒武纪等）快速追赶，需深度适配
- **CPU**：数据预处理、控制面、通信调度仍有大量 CPU 计算需求
- 异构不仅仅是硬件堆叠，还需要**统一的资源抽象层 + 拓扑感知调度**才能发挥协同优势

### 3. RDMA 网络已成为 AI 集群的"血管"

JD 将 RDMA 网络单独列为一个方向，这反映了分布式训练对网络基础设施的极致要求：

```
模型并行             → 通信密集，需微秒级延迟
数据并行 + All-Reduce → 带宽密集，需 400G/800G 互联
专家并行 (MoE)       → 全对全通信，需灵活拓扑
序列并行             → 长序列下 Ring Attention 通信模式
```

RDMA（无论是 RoCEv2 还是 InfiniBand）提供的**低延迟、高带宽、CPU 旁路**特性，是支撑这些并行策略的基石。

### 4. "榨干每一份算力"——性能优化驱动力

JD 中多次强调的"跑满所有计算资源，榨干每一份算力"反映了一个工程现实：

```
理想峰值算力
    ↓ × 框架开销（~10-20%）
框架可达算力
    ↓ × 通信开销（~20-40%）
实际训练吞吐
    ↓ × 故障/重启（~5-15%）
有效产出算力
```

性能优化就是不断缩小每一层差距的过程：

| 瓶颈层 | 典型优化手段 |
|--------|------------|
| **计算** | Kernel fusion、Flash Attention、算子调优、Tensor Core 利用 |
| **通信** | 拓扑感知调度、All-reduce 算法选择（Ring/Rabbit/2D-Torus）、overlap 计算通信 |
| **I/O** | 预取流水线、内存映射、NVMe Direct、存储分级 |
| **调度** | 拓扑亲和性、Gang Scheduling、Backfilling、碎片整理 |
| **可靠性** | 异步 checkpoint、Elastic Training、故障预测、快速恢复 |

### 5. 与 Harness Engineering 的双向映射

这 JD 描述的是 **Harness Engineering 的物理实现层** 与 **Agent 的执行底座** 的交叉点：

| 超算集群方向 | 对应 Harness 概念 |
|------------|-----------------|
| **异构调度系统** | Agent Loop 的资源调度依赖、Task Registry 的执行环境分配 |
| **集群管理平台** | Harness Runtime 的部署/运维/可观测性基础设施 |
| **系统性能调优** | Harness Cost/Latency 优化（#8）的物理层实现 |
| **网络架构** | 分布式 Agent 间通信、多 Agent 协同的基础设施依赖 |
| **下一代架构规划** | Model-Harness-Fit 中"硬件→Harness→Model"的底层支撑 |

> 如果说 [[Agent沙箱工程]] 是单 Agent 沙箱的物理实现，那么**超算集群工程就是大规模 Agent/HPC 工作负载的集群级物理底座**。

### 6. 六大赛事信号——HPC 行业的"学术信用分"

加分项中列出了 SC / ASC / ISC / PAC（HPC 赛事）和 CMO / NOI / IOI / ICPC（数学/算法竞赛）。这传递了明确的信号：

| 赛事 | 领域 | 信号含义 |
|------|------|---------|
| **SC** (Supercomputing Conference) | HPC | 全球最大超算会议的学生集群竞赛 |
| **ASC** (Asia Supercomputer Community) | HPC | 亚洲超算大赛，Linpack + 应用优化 |
| **ISC** (International Supercomputing Conference) | HPC | 欧洲超算会议的学生集群挑战 |
| **PAC** (全国并行应用挑战) | HPC | 中国并行计算应用大赛 |
| **CMO** | 数学 | 中国数学奥林匹克 |
| **NOI / IOI** | 算法 | 全国/国际信息学奥林匹克 |
| **ICPC** | 编程 | 国际大学生程序设计竞赛 |

这反映该团队寻找有"硬竞赛背景"的系统人才——不仅是能写代码，更是能在**极端约束下做最优决策**的人。

### 7. 全系统栈能力——从"自旋锁"到"跨区域一致性"

与 [[摘要-deepseek-agent-infra-jd]] 类似，这份 JD 也要求全系统栈能力：

```
硬件微架构（Cache/Memory/Interconnect）
    ↓
操作系统（内核/调度/I/O 协议栈）
    ↓
网络（RDMA/RoCE/InfiniBand/拥塞控制）
    ↓
分布式系统（调度/编排/容错）
    ↓
应用层（训练框架/通信算子/数据管线）
    ↓
基础设施（IDC/供电/散热/布线）
```

一个人/团队需要理解从 CPU 晶体管到 IDC 数据中心的全链路，这种"从比特到砖块"的能力栈是超算集群工程师的核心竞争力。
