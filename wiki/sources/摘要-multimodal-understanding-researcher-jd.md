# 多模态理解（数据/算法）研究员职位描述

## 来源概览

- **职位**: 多模态理解（数据/算法）研究员
- **公司信号**: DeepSeek
- **团队使命**: "为 AI 注入'看懂并理解世界'的多模态灵魂，让模型在面对复杂的真实世界时，依然能展现出精准的感知与推理能力"
- **覆盖方向**: 视觉编码器优化 / 多模态预训练 / 多模态后训练 / 评测体系建设（四方向可选）
- **核心关键词**: 视觉编码器、VLM、多模态预训练、SFT/RL/OPD、图文数据、多模态 Agent、多模态评测

## 关键洞察

### 洞察 1：多模态——文本范式之外的"第二维度"

此前全部 17 份 JD 隐含地假设模型的输入/输出都是**文本**。本 JD 打破了这一假设：

```
17 份文本假设：AGI = 语言模型 + 工具 + 对齐
    模型"理解世界"的方式是通过文字学到的
    ↓ 局限
    文本是世界的"投影"，不是世界本身
    模型通过文字"知道"猫有毛，但没有"见过"猫

本 JD（多模态）：AGI 必须能直接感知物理世界
    模型不仅通过文字了解猫
    还能通过图像/视频直接"看"猫
    ↓
    多模态感知 = 通往 AGI 的必经之路
```

> **"AGI 绝不会止步于文本，迈向 AGI 的关键一环，在于让模型拥有理解物理世界的能力。"**

### 洞察 2：全链路结构——从像素到智能的完整路径

本 JD 覆盖多模态理解基座模型的**全链路**：

```
像素（Raw Pixels）
    ↓
视觉编码器（CLIP/SigLIP 等）
    ├── 将图像/视频转化为 Token 序列
    ├── 与文本 Token 对齐到同一语义空间
    └── ★ 本 JD 的第一个方向
    ↓
多模态预训练（VLM Pretraining）
    ├── 图文对比学习 / 图文匹配 / 图文生成
    └── ★ 本 JD 的第二个方向
    ↓
多模态后训练（SFT / RL / OPD）
    ├── 指令遵循 / 多步推理 / 对齐
    └── ★ 本 JD 的第三个方向
    ↓
多模态评测体系
    ├── 从公开 Benchmark 到真实场景
    └── ★ 本 JD 的第四个方向
    ↓
多模态 Agent（GUI / 办公 / 搜索等）
    ├── 将多模态能力转化为具体场景的 Agent 能力
    └── ★ 本 JD 第五项工作职责
```

| 阶段 | 核心问题 | 与文本体系对照 |
|------|---------|--------------|
| **视觉编码** | "像素如何变成 Token？" | 文本的 Tokenizer |
| **多模态预训练** | "视觉和语言如何对齐？" | 文本的预训练 |
| **多模态后训练** | "如何让多模态模型更好用？" | 文本的后训练 |
| **多模态评测** | "怎么看多模态模型有没有变好？" | 文本的 Eval |
| **多模态 Agent** | "多模态能力如何在真实场景生效？" | Agent 能力工程 |

### 洞察 3：OPD——超越 SFT 和 RL 的第三种后训练方法

JD 提到了三种后训练方法：SFT、RL 和 **OPD（Online Preference Distillation）**。

```
SFT（Supervised Fine-Tuning）：
    用人工标注的指令-回答对训练
    → 教会模型遵循指令

RL（Reinforcement Learning）：
    通过试错和奖励信号训练
    → 教会模型自我改进

OPD（Online Preference Distillation / Online Policy Distillation）：
    在线策略蒸馏——模型在训练过程中实时生成候选回答
    通过某种偏好信号（人工/模型/规则）在线筛选
    将筛选结果蒸馏回模型自身
    → 结合 SFT 的监督信号和 RL 的在线探索
    → 不需要奖励模型，比 RLHF 更轻量
```

OPD 的引入丰富了后训练方法的理解：

| 维度 | SFT | RL | OPD |
|------|-----|----|-----|
| **数据来源** | 静态标注数据 | 环境交互 | **模型在线生成** |
| **是否需要奖励模型** | 否 | 是 | **否（直接用偏好信号）** |
| **探索性** | 无 | 高 | **中** |
| **训练成本** | 低 | 高 | **中** |
| **典型应用** | 指令遵循 | 复杂任务 | **偏好对齐** |

### 洞察 4：多模态 Agent——GUI 操作、白领办公、多模态搜索

JD 第五项工作职责指向了多模态 Agent 的三个核心场景：

```
场景一：GUI Agent
    模型"看"屏幕截图 → 理解界面 → 操作界面
    → 不再依赖 HTML/API，直接像人类一样操作 GUI
    → 这是计算机使用（Computer Use）的核心能力
    → 与 [[Agent_Interfaces]] 中的 Computer Use 层直接关联

场景二：白领办公 Agent
    模型"看"文档/表格/图表 → 理解内容 → 执行操作
    → 多模态格式的办公自动化
    → 与 [[Agent数据产品工程]] 的办公场景互补

场景三：多模态搜索 Agent
    用户可以"用图搜"、"用视频搜"、"图文结合搜"
    → 搜索不仅限于文本
    → 与 [[AI搜索工程]] 融合
```

这些场景与 [[Agent能力工程]] 的关系：

```
Agent 能力工程（JD 10）：
    聚焦文本/代码 Agent——环境是 shell/编译器/代码仓库
    输入是文本指令，输出是代码或文本

多模态理解（本 JD）：
    拓展到视觉 Agent——环境是 GUI/文档/真实世界
    输入可以是图像/视频+文本，输出包含基于视觉的操作
```

### 洞察 5：多模态数据体系——"海量且充满噪声"

JD 强调多模态数据处理的难度远超纯文本：

```
文本数据的"噪声"：
    语法错误、格式混乱、内容低质
    但文本是离散的、结构化的（有语法规则）
    清洗相对标准化

多模态数据的"噪声"：
    图像质量（模糊/过曝/过暗）
    图文不对齐（图和文本描述不匹配）
    格式多样性（png/jpg/gif/video/PDF……）
    内容多样性（自然图像/屏幕截图/文档扫描/图表……）
    标注难度（描述一幅图比写一段文本难得多）
```

#### 与已有 [[预训练数据工程#3. 多模态数据处理]] 的关系

```
预训练数据工程（已有）：
    关注多模态数据的"工程层面"
    └── 怎么存、怎么清洗、怎么去重
    └── 数据基础设施层

多模态理解研究（本 JD）：
    关注多模态数据的"研究层面"
    └── 什么样的数据提升多模态能力？
    └── 图文配比的策略是什么？
    └── 合成数据怎么设计？
    └── 数据质量与模型能力的映射关系

两者是"工程实现"与"研究策略"的互补。
```

### 洞察 6："反刷榜"文化与评测哲学

JD 的加分项提出了明确的"反刷榜"立场：

> "对模型自身真实能力的提升有极致追求，反感且不愿意通过'刷榜'来换取指标的虚假繁荣。"

这与此前的多个 JD 形成共鸣：

```
JD 17（后训练研究）：
    "不满足于'刷 benchmark'，而是能设计出真正反映模型能力边界与短板的评测方案"

JD 11-14（评测四象限）：
    构建从真实场景出发的评测体系

本 JD（多模态理解）：
    "反感且不愿意通过'刷榜'来换取指标的虚假繁荣"
```

> **"反刷榜"不是不要评测，而是拒绝用评测数字替代真实能力评估。这在多模态领域尤其关键——因为多模态的 Benchmark 更多、更杂、更容易被操纵。**

### 洞察 7：在完整体系中的定位——与文本体系平行的"多模态维度"

```
文本体系                          多模态体系
                      │
┌──────────────────┐  │  ┌───────────────────────────┐
│ Frontier 研究     │  │  │ （多模态前沿探索——未来）       │
│ (JD 15)          │  │  │                            │
└──────────────────┘  │  └───────────────────────────┘
       ↓              │         ↓
┌──────────────────┐  │  ┌───────────────────────────┐
│ 预训练研究         │  │  │ 多模态理解研究 ★（本 JD 18） │
│ (JD 16)          │  │  │ 视觉编码 / 多模态预训练     │
└──────────────────┘  │  │ 多模态后训练 / 评测 / Agent │
       ↓              │  └───────────────────────────┘
┌──────────────────┐  │
│ 后训练研究         │  │  ← 本 JD 的多模态后训练与此并行
│ (JD 17)          │  │
└──────────────────┘  │
       ↓              │
┌──────────────────┐  │  ┌───────────────────────────┐
│ 产品/数据/能力层    │  │  │ GUI Agent / 多模态搜索    │
│ (JD 9-14)        │  │  │ 白领办公 Agent（多模态版）  │
└──────────────────┘  │  └───────────────────────────┘
       ↓              │
┌──────────────────┐  │
│ 工程/基础设施层     │  │  ← 共享同一套基础设施
│ (JD 1-8)         │  │
└──────────────────┘  │
```

> **多模态不是文本体系之上的"附加层"，而是与文本体系平行的"第二维度"。两者共享基础设施，但在训练方法、数据策略、评测标准上各有独特性。**

| 层次 | 文本体系 | 多模态体系 |
|------|---------|-----------|
| **预训练** | Transformer + Next Token Prediction | VLM + Contrastive Learning |
| **后训练** | SFT → RLHF/GRPO → Alignment | SFT → RL/OPD → Alignment |
| **数据** | 文本语料 | 图文/视频/文本混合 |
| **评测** | MMLU/HumanEval | MMMU/ChartQA/TabMWP |
| **Agent** | 代码/搜索/办公 Agent | GUI Agent/多模态搜索/文档 Agent |
| **编码** | Tokenizer | 视觉编码器（CLIP/SigLIP） |
