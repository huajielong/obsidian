# 高性能分布式存储工程师职位描述

> **来源**：互联网公开招聘信息（2026 年），由 Claude 获取并整理。公司来源未明确标注。
>
> **核心定位**：大模型时代的数据基础设施——让任何 AI 应用都能以零妥协的性能和无限扩展能力持续获取数据。

---

## 团队使命

在大模型时代，数据与存储是智能的基石。让任何 AI 应用都能以零妥协的性能和无限扩展能力持续获取数据——存储不仅是承载信息的底座，更是决定 AI 成本与极致体验的核心引擎。

---

## 岗位职责（两大方向）

| 方向 | 核心工作 | 关键技术 |
|------|---------|---------|
| **KV Cache 存储系统** | 设计开发支撑大模型推理的高性能、高可用 KVCache 存储系统，在毫秒级延迟和上亿级 IOPS 下保障系统稳定，用底层技术突破直接颠覆公司核心产品的性价比与用户体验 | 分布式 KV、PagedAttention 衍生、RDMA、持久化缓存、LRU 淘汰 |
| **分布式文件/对象存储** | 设计开发下一代分布式文件系统和对象存储系统，为超大规模训练提供极致带宽与稳定数据链路，为分布式数据处理提供强一致的共享存储层 | 分布式文件系统( Lustre/GPFS/CephFS )、对象存储( S3 )、强一致性协议、Raft/Paxos |

---

## 岗位要求

### 核心要求
- 扎实的 Rust/C++ 编程能力，精通多线程、异步编程及系统性能分析
- 熟悉分布式系统原理，深入理解并实践过分布式事务、Paxos/Raft 等共识机制
- 能灵活运用经典文献及顶尖开源系统中的设计范式和经验（如 RocksDB、FoundationDB、ClickHouse 等），而非简单套用与组合。致力于从第一性原理出发，用简单优雅的方案解决问题

### 加分项
- 具备开发和应用零拷贝数据链路的经验，熟悉基于 RDMA 实现高吞吐、低延迟的通信方法，致力于榨干现代网络硬件的性能极限
- 设计并开发过 SSD/HDD 上的高性能本地存储引擎，熟练掌握基于 io_uring/SPDK 的实现，能在存储硬件极限上做文章
- 实现过 Linux 内核文件系统，或深入解决过 FUSE 在性能与可维护性上的痛点，对内核 I/O 栈有第一手经验
- 在 OSDI/FAST/VLDB 等高水平系统会议发表过论文，或者有同等水准的工程影响力

## 关键洞察

### 1. 存储是 AI 基础设施的"横向底座"

本 JD 与前四份 JD 的根本区别在于：**存储不是一层垂直堆叠，而是横向贯穿所有层的底座**。

```
AI训练推理系统工程    ← 需要存储：CKPT 存储、DataLoader 读取、KV Cache 推理
    ↓
AI计算引擎工程        ← 需要存储：编译器缓存、算子库下载
    ↓
超算集群工程          ← 需要存储：并行文件系统、集群共享存储
    ↓
Agent沙箱工程         ← 需要存储：沙箱临时/持久化存储
    ↓
预训练数据工程        ← 需要存储：数据湖、对象存储（最直接的存储需求）
```

这份 JD 的独特之处也正在于此——**存储工程师的服务对象不是最终用户，而是所有其他基础设施团队**。存储做得好，整个系统的性能和可靠性都会受益。

### 2. KV Cache 存储系统——推理场景的"第二级内存"

JD 中最引人注目的是 KV Cache 存储系统方向：

```
推理请求
    ↓
GPU 显存中的 KV Cache（第一级） ← PagedAttention 管理
    ↓（换出）
KV Cache 存储系统（第二级）      ← ★ 本 JD 构建
    ↓（持久化）
磁盘/对象存储（第三级）          ← 跨 session 缓存
```

| 级别 | 介质 | 延迟 | 容量 | 用途 |
|------|------|------|------|------|
| **L1** | GPU HBM | <100ns | GB 级 | 当前请求的活跃 KV Cache |
| **L2** | DRAM + 高速 SSD（本系统）| 微秒-毫秒 | TB 级 | 跨请求共享的 Prefix / 长上下文 Cache |
| **L3** | 磁盘 / 对象存储 | 毫秒-秒 | PB 级 | 跨 session 持久化、冷启动加速 |

KV Cache 存储系统本质上是**推理场景的分布式缓存层**，需要：
- 毫秒级延迟（不拖累推理 P99 延迟）
- 上亿级 IOPS（数千并发推理请求的 Cache 查找/写入）
- 高可用（单点故障不导致 Cache 失效）
- RDMA 接入（零拷贝数据传输，不经过 CPU）

### 3. 两大"存储硬件极限"技术

加分项中的两个硬件极限技术：

| 技术 | 定位 | 解决的问题 | 适用场景 |
|------|------|-----------|---------|
| **io_uring** | Linux 内核异步 I/O 框架 | 传统 read/write 系统调用的上下文切换开销 | 文件系统、通用 I/O |
| **SPDK** | 用户态 NVMe 驱动框架 | 绕过内核，直接从用户态访问 NVMe SSD | KV Cache 存储引擎、高性能本地存储 |

两者对比：

```
传统 I/O 路径：    应用 → 内核 VFS → 块设备层 → NVMe 驱动 → SSD
    ↑ 多次上下文切换 + 中断处理

io_uring 路径：    应用 → SQ/CQ 环形队列 → NVMe 驱动 → SSD
    ↑ 减少上下文切换，但仍在内核

SPDK 路径：        应用 → SPDK NVMe 驱动 → SSD
    ↑ 完全绕过内核，用户态轮询，极致延迟
```

### 4. "从第一性原理出发"的工程哲学

核心要求中有一段非常精辟的表述：

> 能灵活运用经典文献及顶尖开源系统中的设计范式和经验（如 RocksDB、FoundationDB、ClickHouse 等），而非简单套用与组合。致力于从第一性原理出发，用简单优雅的方案解决问题。

这传递了三层标准：

| 工程师级别 | 行为 | 评价 |
|-----------|------|------|
| **初级** | 不知道 RocksDB/FoundationDB，从零造轮子 | 缺乏视野 |
| **中级** | 知道这些系统，直接套用 | 简单套用组合 |
| **高级** | 理解这些系统的设计哲学，知道何时借鉴、何时另起炉灶 | **第一性原理思考** |

**"用简单优雅的方案解决问题"**——这是系统设计的最高境界。简单不是简陋，而是经过深思熟虑后的本质简化。

### 5. 学术信号转向：OSDI/FAST/VLDB

加分项列出 OSDI/FAST/VLDB——这与 [[摘要-hpc-cluster-engineer-jd]] 的 osDI/SOSP/EuroSys/NSDI/ATC 部分重叠但各有侧重：

| 会议 | 领域 | 信号 |
|------|------|------|
| **OSDI** (Operating Systems Design and Implementation) | 操作系统、分布式系统 | 顶级系统会议，与集群 JD 共享 |
| **FAST** (File and Storage Technologies) | 文件系统、存储 | **存储领域最顶级的专门会议** |
| **VLDB** (Very Large Data Bases) | 数据库、数据管理 | 分布式存储与数据管理的交叉 |

FAST 的出现标志着**存储**是这个 JD 的核心差异——FAST 是存储领域最权威的学术会议，列出它说明这个团队在意存储领域的学术前沿。

### 6. 与前面四份 JD 的竞赛和学术信号对比

| JD | 竞赛信号 | 学术会议信号 |
|----|---------|------------|
| 超算集群 | SC/ASC/ISC/PAC + CMO/NOI/IOI/ICPC | — |
| 算子/通信/编译器 | SC/ASC/ISC/PAC + NOI/IOI/ICPC | — |
| 训练/推理框架 | SC/ASC/ISC/PAC + IOI/NOI/ICPC + 排名前 10% | — |
| **本 JD（分布式存储）** | **—**（无竞赛加分） | **OSDI/FAST/VLDB** |

这是一个很重要的信号变化：前三个 JD 共享同一套竞赛语言，但存储 JD 完全不同的要求——**学术论文 > 竞赛成绩**。这要么是同一家公司中不同团队的文化差异，要么是来自不同公司的 JD。

### 7. 存储工程师的核心能力模型

结合 JD 全文，存储工程师需要的能力金字塔：

```
                         工程品味
                    （第一性原理/简单优雅）
                          ↕
                 分布式共识 + 一致性
              （Paxos/Raft/分布式事务）
                    ↕              ↕
           存储引擎              网络/硬件
       (RocksDB/io_uring/SPDK)    (RDMA/零拷贝)
                    ↕              ↕
           编程语言              系统基础
         (Rust/C++/异步编程)    (内核 I/O 栈/文件系统)
```
