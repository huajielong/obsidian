# 预训练（数据/算法）研究员职位描述

## 来源概览

- **职位**: 预训练（数据/算法）研究员
- **公司信号**: DeepSeek
- **团队使命**: "探索智能的本质，持续追求 AGI"，"以亿万数据筑就模型智能的坚实底座"——将预训练技术从"工程"提升到"研究"的层面
- **招聘方向**: 预训练算法研究方向、预训练数据研究方向（双方向并行）
- **核心关键词**: 模型架构、优化器、训练动力学、Scaling Law、数据治理、数据配比、合成数据、多模态数据

---

## 关键洞察

### 洞察 1："双核"结构——算法与数据在同一职位下的深度耦合

这是全部十六份 JD 中**唯一将算法研究与数据研究放在同一职位下的 JD**，反映了一个深刻的工程理念：

```
大多数 AI 公司：算法研究员 ≠ 数据研究员（两个独立的团队）
    ↓
本 JD 的做法：算法与数据放在同一职位下
    ↓
这意味着：
    1. 算法创新和数据策略必须紧密耦合——不能各自为政
    2. 架构改进和数据优化需要互相配合
    3. 研究员需要既理解算法又理解数据
```

| 维度 | 传统做法 | 本 JD 的做法 |
|------|---------|-------------|
| **团队结构** | 算法组 + 数据组分隔 | **同一职位下双方向并行** |
| **协作模式** | 数据团队"伺候"算法团队 | **紧密耦合、共同决策** |
| **信息流** | 数据→清洗→算法（单向） | **算法↔数据（双向反馈）** |
| **效率** | 跨团队沟通成本高 | **一个团队内闭环** |

### 洞察 2："蒙尘的富矿"——数据哲学

> "如何寻找蒙尘的富矿，并淘洗炼制成可用的智能燃料，是一门极难的硬功夫。"

这段话包含三个层次：

```
"蒙尘的富矿"：
    高质量数据存在，但被忽略或未得到充分利用
    → 需要敏锐的数据发现眼光

"淘洗炼制"：
    原始数据→清洗→筛选→组织→智能燃料
    → 需要系统化的数据治理能力

"极难的硬功夫"：
    这不仅是工程问题，更是研究问题
    → 需要深入理解"什么数据造就了什么样的智能"
```

### 洞察 3：Scaling Law 作为"工程工具"而非"信仰"

与 [[Frontier研究]] 中"Scaling 可能不够"的质疑不同，本 JD 将 Scaling Law 定位为**科学规划工具**：

```
Frontier 研究（JD 15）：
    "我们与 AGI 的距离也许不只是 scaling"
    → Scaling Law 有局限
    → 寻找下一范式

预训练研究（本 JD）：
    "建立科学的 scaling law，对模型的 scaling 行为建立认知，并进行合理的预测和规划"
    → Scaling Law 是有效的工具
    → 在当前范式内把它用到极致
```

| 维度 | Frontier 研究 | 预训练研究 |
|------|-------------|-----------|
| **对 Scaling Law 的态度** | 质疑：也许不够 | **使用：科学的规划工具** |
| **关注焦点** | 下一范式是什么？ | **当前范式如何做到极致？** |
| **时间尺度** | 年-十年 | **月-年** |
| **风险承受** | 极高（"愿意承担研究风险"） | **中高（系统性探索）** |
| **与工程的距离** | 远 | **近（软硬件协同）** |

### 洞察 4：算法研究方向的六大课题——当前范式的"引擎室"

算法研究方向覆盖了预训练技术的**全部核心技术栈**：

```
1. 模型架构设计：软硬件协同，实际问题驱动
    ↓
2. 优化器设计：高性能、鲁棒，训练动力学与稳定性
    ↓
3. Scaling Law：建立认知，预测规划
    ↓
4. 训练/推理加速算法：降本增效
    ↓
5. 新型架构与方法：开拓性探索（非共识方向）
    ↓
6. 前沿跟进：持续推动基础研究
```

> **"非共识的新型网络架构"**——与 Frontier 研究有交集但定位不同：Frontier 寻找"取代 Transformer 的范式"，本层探索"Transformer 的变体/改进/替代候选"。

### 洞察 5：数据研究方向的七大课题——从"工程"到"研究"

数据研究方向将数据工作从**工程执行**提升到**研究驱动**的高度：

```
1. 数据生产管线：持续搭建与改进，与上下游紧密合作
    ↓
2. 数据治理体系：科学、严谨、可落地的治理框架
    ↓
3. 数据与智能的纽带：通过实验挖掘不足，建立理解
    ↓
4. 多维度模型评估：自动+人工，覆盖通用及垂直场景
    ↓
5. 数据的合成与 scaling：研究如何"制造"高质量数据
    ↓
6. 垂类数据研究：多模态/Agent/用户反馈/高质量学术
    ↓
7. 前沿跟进：持续推动基础研究
```

与现有 [[预训练数据工程]] 的区别：

| 维度 | 预训练数据工程（已有） | 预训练数据研究（本 JD） |
|------|---------------------|---------------------|
| **核心关注** | 如何搭建高效数据管线 | **数据策略与智能的关系** |
| **工作性质** | 工程实现（Pipeline 框架/清洗工具） | **研究驱动（实验/分析/策略）** |
| **关键产出** | 数据平台、清洗框架、去重工具 | **数据配比策略、合成方法、评估体系** |
| **与模型的关系** | 提供训练数据（执行者） | **理解数据→智能映射（研究者）** |
| **实验驱动** | 工程指标（吞吐/延迟/召回） | **模型能力指标（数据贡献度分析）** |

两者是**互补关系而非替代关系**——工程层面解决"怎么高效处理"，研究层面解决"处理成什么样更好"。

### 洞察 6：人才信号——"代表作"与"竞赛成绩"

JD 的加分项透露了清晰的招聘偏好：

```
第一优先级：学术影响力
    "有代表作（一作引用过 100 的文章，综述文章除外）"
    → 看重深度学术贡献，而非灌水或综述

第二优先级：智力水平（竞赛成绩）
    "NOI/CMO/CPhO 银牌及以上，顶尖高校院系排名前 5%"
    → 看重基础学科能力，尤其是数学/信息学竞赛背景

第三优先级：工业经验
    "在顶尖 AI 公司或实验室的核心团队有过长期实习或工作经历"
    → 看重"主线模型"的实战经验

第四优先级：软技能
    "对模型能力边界有敏锐判断""优秀的跨团队沟通能力"
    → 不仅会做，还会判断什么方向更重要
```

### 洞察 7：在整个知识体系中的定位——"当前范式的引擎室"

```
                ┌──────────────────────────────────┐
  范式层         │       Frontier 研究（JD 15）      │ ← "下一范式是什么？"
                └──────────────────────────────────┘
                           ↑ 理论突破驱动
                ┌──────────────────────────────────┐
  预训练研究层  │   预训练研究（本 JD ★，第 16 份）   │ ← "当前范式如何做到极致？"
                │  算法架构 + 数据策略 双核驱动       │
                └──────────────────────────────────┘
                           ↑ 模型能力输出
                ┌──────────────────────────────────┐
  产品/评测/数据 │  AI产品工程 / Agent数据产品工程    │
                │  专业领域/创作/情感数据工程        │
                └──────────────────────────────────┘
                           ↑ 能力 + 框架
                ┌──────────────────────────────────┐
  工程/基础设施层 │  Harness / Prompt / Context       │
                │  训练/计算/集群/存储/硬件/设施/沙箱│
                └──────────────────────────────────┘
```

| 层次 | 角色 | 关注 | 时间尺度 |
|------|------|------|---------|
| **范式层** | Frontier 研究员 | **下一个范式是什么？** | 年-十年 |
| **预训练研究层** | 预训练研究员 | **当前范式如何做到极致？** | 月-年 |
| **产品/数据层** | 产品经理/数据PM | **在当前范式下怎么用到最好？** | 周-月 |
| **能力层** | Agent 能力工程师 | **怎么让模型能力更强？** | 月 |
| **工程层** | 各类工程师 | **怎么高效执行？** | 天-周 |

> **预训练研究是"引擎室"——Frontier 研究提供燃料（新方向），预训练研究把燃料转化为动力（当前范式最优），工程层把动力传递到车轮（产品/基础设施）。**

### 洞察 8：与三层工程模型的关系

本 JD 的工作与三层工程模型有深刻关联：

```
Prompt Engineering：预训练研究员定义"模型理解什么"→ 影响 Prompt 的策略
    ↓
Context Engineering：预训练数据研究员定义"喂给模型什么"→ 与 Context 工程高度重叠
    ↓
Harness Engineering：预训练算法研究员定义"模型怎么学"→ 训练框架/Harness 的底层逻辑
```

具体映射：

| 预训练工作 | 三层工程映射 | 说明 |
|-----------|-------------|------|
| 模型架构设计 | Harness (Training) | 架构决定训练 Harness 的设计 |
| 数据治理/配比 | Context Engineering | 数据选择是 Context 工程的预训练版本 |
| 训练动力学 | Harness (Observability) | 理解训练过程的可观测性 |
| 加速算法 | Harness (Cost Optimization) | 训练/推理加速直接影响成本 |
| 评估体系 | Eval Harness | 模型评估是 Eval Harness 的核心输入 |
