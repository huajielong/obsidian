# 后训练（数据/算法）研究员职位描述

## 来源概览

- **职位**: 后训练（数据/算法）研究员
- **公司信号**: DeepSeek
- **团队使命**: "通过高质量后训练数据与前沿算法，最大化释放 Base 模型潜能，打造兼具强大智能与深度意图理解力的 AI 模型"
- **核心关键词**: RLHF、RLVR、PPO、GRPO、后训练数据集、数据清洗/合成管线、评测体系、Agent 场景
- **招聘方向**: 算法方向（RL 算法迭代）、数据方向（后训练数据集）、评测方向（评测体系）——**三元组结构**

---

## 关键洞察

### 洞察 1：后训练——"基座模型→可用模型"的关键转换层

后训练是预训练之后、模型上线之前的**核心转化环节**：

```
预训练阶段                   后训练阶段                    产品/Agent阶段
Base 模型                   对齐/激活后的模型              用户使用的模型
    │                            │                            │
    │ "知道很多"                 │ "能做很多"                 │ "被用好"
    │ 但不会对话                 │ 能听懂指令                 │
    │ 不会遵循指令               │ 会工具调用                 │
    │ 不会工具调用               │ 会推理步骤                 │
    │ 不会多步推理               │ 行为被对齐                 │
    ↓                            ↓                            ↓
    ┌──────────────┐            ┌──────────────┐            ┌──────────────┐
    │ 预训练研究    │   ───→    │ 后训练研究 ★  │   ───→    │ Agent/AI产品 │
    │ (JD 16)      │            │  (本 JD 17)  │            │ (JD 9-14)    │
    └──────────────┘            └──────────────┘            └──────────────┘
```

> **预训练决定模型"知道多少"，后训练决定模型"能用多少"。一个知识渊博但无法对话的模型，其价值远低于一个知识相当但能协作的模型。**

### 洞察 2：三元组结构——算法 + 数据 + 评测

本 JD 的三条工作职责构成了一个完整的闭环：

```
算法（Algorithm）：                 数据（Data）：                  评测（Eval）：
RLHF/RLVR/PPO/GRPO         高质量后训练数据集             多维度能力评测体系
    ↓                             ↓                              ↓
"怎么训练"                  "用什么训练"                    "训练好了吗"
    │                             │                              │
    └─────────────────────────────┼──────────────────────────────┘
                                  ↓
                        模型能力持续迭代
```

| 方向 | 核心问题 | 技术栈 | 产出 |
|------|---------|-------|------|
| **算法方向** | "在预训练模型与数据既定的条件下，最大限度地释放模型潜力" | RLHF/RLVR/PPO/GRPO | 更好的训练算法、更稳定的训练流程 |
| **数据方向** | "构建高质量、高多样性的后训练数据集" | 数据清洗/合成/自动化管线 | 高质量后训练数据集 |
| **评测方向** | "设计出能真正反映模型能力边界与短板的评测方案" | 自动+人工评测、Agent 场景评测 | 精准的评测体系与能力诊断 |

这种"三元组"结构与 [[预训练研究]] 的"双核"（算法 + 数据）形成对照：

| 维度 | 预训练研究（双核） | 后训练研究（三元组） |
|------|-----------------|------------------|
| **核心结构** | 算法研究 + 数据研究 | **算法 + 数据 + 评测** |
| **为什么评测独立？** | 预训练评测相对标准化（PPL/基准测试） | **后训练评测高度场景化——需要专门设计 Agent 场景评测** |
| **与 Eval_Harness 的关系** | Eval 是外部工具 | **Eval 是内部核心组件** |

### 洞察 3：RL 算法栈——从 RLHF 到 GRPO 的完整光谱

JD 列出了后训练领域的**完整主流 RL 算法栈**：

```
RLHF（Reinforcement Learning from Human Feedback）：
    人类标注偏好 → 训练奖励模型 → PPO 优化
    ↓ 经典但昂贵

RLVR（Reinforcement Learning from Verifiable Rewards）：
    用可验证结果（代码测试/数学答案）作为奖励
    ↓ 无需奖励模型，更客观

PPO（Proximal Policy Optimization）：
    策略梯度方法，稳定的 RL 训练
    ↓ 最广泛使用的 RLHF 实现框架

GRPO（Group Relative Policy Optimization）：
    DeepSeek 提出的改进——组内相对奖励，无需价值网络
    ↓ 更高效、更稳定的 DeepSeek 特色方法
```

#### GRPO 的独特地位

GRPO（Group Relative Policy Optimization）是 DeepSeek 在后训练领域的标志性贡献：

```
传统 PPO：需要价值网络（Critic）估计状态价值
    → 额外的大模型作为 Critic → 计算量大、训练不稳定
    
GRPO（DeepSeek 方案）：
    对同一个 prompt 采样多个 response
    在组内计算相对奖励
    不需要价值网络
    → 更简单、更高效、更稳定
    
    关键应用：DeepSeek-R1 的强化学习训练
```

### 洞察 4：与 Agent 能力工程的关系——"教练"与"训练营"

```
Agent 能力工程（JD 10）：
    构建 RL 训练环境——"训练营"
    └── 定义问题空间、交互接口、奖励信号
    └── 覆盖 Code/安全/Science 三方向

后训练研究（本 JD）：
    设计 RL 训练算法和数据集——"教练"
    └── 选择/迭代训练算法（PPO vs GRPO vs ...）
    └── 构建高质量训练数据
    └── 设计评测方案评估效果

两者关系：环境 × 算法 × 数据 = 模型能力提升
```

| 维度 | Agent 能力工程 | 后训练研究 |
|------|-------------|-----------|
| **角色** | "训练营建造者" | **"教练"** |
| **核心产出** | RL 环境、评测任务、能力数据 | **训练算法、训练数据、评测体系** |
| **关注** | "模型在什么场景下训练？" | **"模型用算法训练？用什么数据？怎么知道效果？"** |
| **与模型的关系** | 提供训练内容（环境 + 任务） | **提供训练方法和数据（算法 + 数据集）** |
| **相互依赖** | 环境需要算法来驱动 | **算法需要环境来实现** |

两者是**深度互补关系**——没有好的环境，算法无法施展；没有好的算法，环境无法产生最大价值。

### 洞察 5："不满足于刷 Benchmark"的评测哲学

JD 的第三点职责提出了一个关键的评测理念：

> "对模型评测有方法论层面的思考，不满足于'刷 benchmark'，而是能设计出真正反映模型能力边界与短板的评测方案。"

这与现有知识体系中的评测理念形成对话：

```
"刷 Benchmark" 的做法：
    在标准测试集上拿到高分 → 报告好的结果
    ↓ 局限
    1. Benchmark 可能被污染（数据泄露到训练集）
    2. Benchmark 覆盖不到能力的边界
    3. 高分不意味着真实场景中好用

"真正反映能力边界"的做法：
    设计目标场景的评测维度
    构建能暴露模型短板的测试
    关注模型在哪失败，而不只是在哪成功
    将评测结果与产品体验关联

与评测四象限（JD 11-14）的关联：
    评测四象限定义"什么标准"（用户满意度/专业/审美/温度）
    后训练评测定义"有没有达到这些标准"
```

### 洞察 6：在完整体系中的定位——十六份→十七份 JD

本 JD 的出现，揭示了从 Base 模型到可用模型之间一直存在的**隐性层**：

```
                ┌──────────────────────────────────┐
  范式层         │       Frontier 研究（JD 15）      │ ← 探索下个范式
                └──────────────────────────────────┘
                           ↑ 方向
                ┌──────────────────────────────────┐
  预训练研究层  │       预训练研究（JD 16）          │ ← 炼基座模型
                └──────────────────────────────────┘
                           ↑ 基座模型
                ┌──────────────────────────────────┐
  后训练研究层  │    后训练研究（本 JD ★，第 17 份）  │ ← 释放潜力 ← NEW
                │  算法 + 数据 + 评测 三元组驱动       │
                └──────────────────────────────────┘
                           ↑ 对齐/激活后的模型
                ┌──────────────────────────────────┐
  产品/数据层   │  AI产品工程 / Agent数据产品工程     │ ← JD 9, 11-14
                │  专业/创作/情感数据工程              │
                └──────────────────────────────────┘
                           ↑ 能力
                ┌──────────────────────────────────┐
  能力构建层    │      Agent能力工程（JD 10）         │ ← 训练环境
                └──────────────────────────────────┘
                           ↑ 框架
                ┌──────────────────────────────────┐
  工程/基础设施层│  训练推理 / 计算 / 集群 / 存储     │ ← JD 1-8 + 已有
                │  硬件 / 设施 / 沙箱                │
                └──────────────────────────────────┘
```

> **预训练研究建造"引擎"，后训练研究调校"引擎"——两者缺一不可。**
