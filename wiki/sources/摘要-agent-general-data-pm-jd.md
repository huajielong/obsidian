# 通用Agent数据产品经理（办公/生活/搜索）职位描述

## 来源概览

- **职位**: 通用Agent数据产品经理（办公/生活/搜索）
- **公司信号**: DeepSeek
- **团队使命**: "Agent 将成为对人本身的又一次「延伸」—— 以搜索打破静态知识的围墙，以工具触达数字世界的疆域，让人与 AI 的关系从对话走向托付"
- **核心关键词**: 人机共生、通用场景、评测体系、数据管线

---

## 关键洞察

### 洞察 1：角色定位——产品×数据×评测的三合一角色

这是十份 JD 中**最独特的产品角色**——它不是纯产品经理（如 JD 9），也不是纯能力工程师（如 JD 10），而是两者的**交汇点**：

```
AI产品工程 (JD 9)                     Agent能力工程 (JD 10)
    ↑ 定义用户体验和产品方向               ↑ 构建 RL 环境和评测任务
    ↓                                    ↓
    ──── 本 JD：数据产品经理 ────
         ↓
    评测体系（如何判断"好"）
    数据管线（如何生产高质量数据）
    场景拆解（办公/生活/搜索）
```

| 维度 | JD 9（产品经理） | 本 JD（数据产品经理） | JD 10（能力工程师） |
|------|----------------|--------------------|------------------|
| **核心问题** | "技术能变成什么产品？" | **"怎么知道产品真的变好了？"** | "模型怎样学会更多能力？" |
| **主要产出** | 产品路线图、功能设计 | **评测标准、数据管线** | RL 环境、评测任务 |
| **场景范围** | 全产品线 | **办公/生活/搜索（通用）** | Code/安全/科学（专业） |
| **技术深度** | 中等（产品视角） | **中高（数据+评测视角）** | 高（工程视角） |
| **与模型的关系** | 外部使用者 | **能力改进的驱动者** | 能力构建者 |

### 洞察 2："能用"vs"好用"——评测的终极挑战

JD 中提出了一个核心命题：

> **建立能真正区分"能用"与"好用"的评测标准**

这揭示了 Agent 评测的深层挑战：

```
"能用"的 Agent：   能完成任务，但体验粗糙、异常多、需频繁人工介入
"好用"的 Agent：   流畅完成任务，边界情况处理得当，用户几乎感受不到它的存在

评测挑战："能用"可以自动化验证，"好用"需要多维度的综合判断
```

这要求评测体系不仅覆盖**结果正确性**（是否完成了任务），还要覆盖**过程质量**（完成得是否优雅）：

| 维度 | "能用"指标 | "好用"指标 |
|------|-----------|-----------|
| **准确性** | 结果正确 | 结果正确 + 无冗余/误导信息 |
| **效率** | 最终完成 | 路径最短 + 资源消耗最低 |
| **鲁棒性** | 理想条件下工作 | 异常输入也能优雅处理 |
| **用户体验** | 功能可用 | 交互自然、反馈及时、可理解 |
| **泛化能力** | 见过的问题能解 | 没见过的问题也能试探出解法 |

### 洞察 3：三段式使命——搜索、工具与人机关系

团队使命宣言体现了一个完整的价值链条：

```
第 1 段：搜索打破静态知识的围墙    ← 信息获取（Search）
第 2 段：工具触达数字世界的疆域    ← 行动执行（Tool Use）
第 3 段：从对话走向托付，解放行动力与创造力 ← 人机关系进化
```

这与此前 JD 共同体现的技术哲学一致——**Agent 不是对话工具，而是延伸人的能力的"数字代理"**。

与 AI 产品经理 JD 对比，可以发现完全一致的产品愿景，但本 JD 更聚焦在**评测和数据**这一实现路径上：

| 愿景元素 | AI 产品经理（JD 9） | 本 JD |
|---------|------------------|-------|
| **产品方向** | Chat / APP / 开放平台 + Harness | **办公/生活/搜索** |
| **实现路径** | 产品规划 + 功能设计 + 项目管理 | **评测体系 + 数据管线 + 场景拆解** |
| **核心能力** | 产品品味 + 用户洞察 + 数据分析 | **评测判断 + 数据工程 + Prompt/Coding** |
| **技术深度** | 理解 Agent 产品 | 深度使用 + **动手构建评测和数据** |

### 洞察 4：评测体系的"多维度"设计

JD 中要求评测覆盖：

> "端到端任务完成度与过程行为质量，打通人工评估与自动化评估"

这映射到评测体系的分层设计：

```
第一层：自动化可验证指标
    ├── 任务完成率（Success Rate）
    ├── 端到端延迟（End-to-end Latency）
    ├── Token 消耗（Cost Efficiency）
    └── 工具调用成功率（Tool Success Rate）

第二层：半自动/LLM-as-Judge
    ├── 回答的准确性与完整性
    ├── 执行路径的合理性
    ├── 错误恢复能力
    └── 与用户意图的对齐程度

第三层：人工评估（黄金标准）
    ├── 整体体验评分
    ├── 边界情况处理
    ├── "惊喜感"（doing the right thing unasked）
    └── 安全与价值观对齐
```

> **核心洞察**：本 JD 的工作就是要在这三层之间建立桥梁——让自动评估尽可能接近人工评估的判断，同时让数据管线源源不断地供给高质量的训练和评测数据。

### 洞察 5："数据生产管线"——从用户反馈到训练数据

JD 第三项职责："定义回答及执行结果的理想态，设计并迭代高质量、高多样性的数据生产管线"

这形成了一个完整的数据闭环：

```
用户使用 Agent
    ↓
采集交互数据（成功/失败/反馈）
    ↓
基于"理想态"标准标注数据
    ↓
构建训练数据 + 评测数据
    ↓
模型训练 + 评测验证
    ↓
部署 → 用户使用 → 下一次循环
```

与 [[Agent能力工程]]（JD 10）的数据闭环相比：

| 维度 | JD 10（能力工程师） | 本 JD（数据产品经理） |
|------|-------------------|--------------------|
| **数据来源** | RL 环境探索 + 真实失败场景 | **用户日常使用 + 评测任务** |
| **数据形态** | RL 训练轨迹（reward-based） | **标注数据 + 评测集 + 理想态样本** |
| **构建方式** | 工程化构建 RL 环境 | **产品化定义"理想态" + 数据管线** |
| **目标** | 模型学会新能力 | **模型在通用场景中的表现更好** |

### 洞察 6："Vibe Coding"的升级版本

JD 要求："精通 prompt engineering 和（vibe）coding；动手能力强，有丰富的使用自动化手段解决真实问题的经验。"

这与 AI 产品经理 JD（JD 9）中的"vibe coding"要求一脉相承，但更进一步：

| JD 9（产品经理） | 本 JD（数据产品经理） |
|----------------|--------------------|
| "能够使用 vibe coding 写代码" | **"精通" + "有丰富的使用自动化手段解决真实问题的经验"** |
| 用 AI 辅助做原型 | **用 AI 做自动化评测和数据管线** |
| 降低技术门槛 | **提升工作效率和规模化能力** |

> **关键信号**：数据产品经理需要比产品经理**更懂技术**——因为他们要构建的是评测系统和数据管线，而不是原型图。

### 洞察 7：跨 JD 全景——十一份 JD 的完整体系

将此 JD 放入十一份 JD 的全景中：

```
产品化层     AI产品工程                    ← JD 9（AI 产品经理）
                ↓ 产品定义 + 用户洞察
评测数据层   Agent数据产品工程              ← ★ JD 11（本 JD）— 评测/数据/场景
                ↓ 数据驱动改进
能力构建层   Agent能力工程                  ← JD 10（Code Agent 数据工程师）
                ↓ 底层能力构建
控制逻辑层   Harness Engineering / 三层工程 / 搜索 / 数据 / 服务
    ↑
══════════ 分界线 ══════════
    ↑
框架/计算/集群/存储/硬件/设施/安全等七层基础设施
```

**完整体系的三条主线**：

| 主线 | 角色 | 核心产出 |
|------|------|---------|
| **产品线** | AI 产品经理（JD 9） | 产品体验、功能设计、项目管理 |
| **评测数据线** | 数据产品经理（JD 11）★ | 评测标准、数据管线、场景定义 |
| **能力线** | Code Agent 数据工程师（JD 10） | RL 环境、能力短板补齐 |

三条线从不同角度推动同一件事——**让模型在真实场景中更好地完成复杂任务**。
