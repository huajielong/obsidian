# IT 基础设施团队职位描述

> **来源**：互联网公开招聘信息（2026 年），由 Claude 获取并整理。公司来源未明确标注。
>
> **核心定位**：基础设施是上限——不写模型但决定模型能跑多快，不写业务代码但决定代码能否顺利部署。

---

## 团队使命

大模型训练能不能跑满带宽，取决于我们的网络配置；GPU 集群能不能发挥出纸面性能，取决于我们的固件和压测基线；新员工第一天能不能顺畅开工，取决于我们的流程有没有提前打通。

基础设施不是背景板——它是上限。

---

## 两个招聘方向

| 方向 | 一句话定位 | 用户 | 核心产出 |
|------|-----------|------|---------|
| **系统硬件&网络方向** | 让每一层硬件和网络在真实负载下接近物理极限 | 训练集群、推理服务 | 硬件选型、固件基线、压测框架、RDMA 网络、DCI |
| **IT桌面&信息化方向** | 零摩擦的办公体验，让每个人第一天就获得可用工具 | 全体员工 | 设备交付、办公网络、视频会议、IT 资产、SaaS 集成 |

## 系统硬件 & 网络方向

### 硬件方向职责

| 职责 | 核心工作 | 相关技术 |
|------|---------|---------|
| **硬件探索与选型** | 研究最新硬件（GPU/CPU/SSD/网卡/服务器），从第一性原理为选型提供决策支持 | 硬件规格分析、性价比建模、竞品对比 |
| **硬件配置管理** | 建立品牌→型号→固件→参数的配置体系，开发自动下发工具，实现异构硬件统一纳管 | Redfish/IPMI、固件基线、配置即代码 |
| **基准压测框架** | 设计覆盖 GPU/NVMe/内存/网络的压测框架，集成或自研工具，上下架/固件升级后自动执行基线对比 | fio、iperf3、gpu-burn、自研引擎、数据建模 |
| **故障应急响应** | 快速响应集群故障，保障 AI 训练与推理连续性 | 硬件诊断、日志分析、应急处置 |

### 网络方向职责

| 职责 | 核心工作 | 相关技术 |
|------|---------|---------|
| **大规模路由域** | 设计、优化和排障大规模路由域，降低故障爆炸半径 | OSPF、BGP、VXLAN、VPN |
| **跨地域 DCI** | 多数据中心互联架构设计与运维，主备切换、双活、流量牵引 | DCI（Data Center Interconnect）、灾备、流量工程 |
| **RDMA 网络运维** | InfiniBand/RoCEv2 全生命周期运维与持续优化 | InfiniBand、RoCEv2、自适应路由、拥塞控制 |
| **网络可观测性与自动化** | 物理层到 RDMA 层的全栈监控，推动配置管理、变更校验、故障自愈 | 网络遥测、自动化运维、NetDevOps |

### 核心要求

- 在硬件架构（CPU/GPU/存储/网络互联）或网络技术（路由协议/RDMA/数通）中至少一个方向有深入理解
- 熟悉 Python/Go，能独立开发压测、配置管理、自动化工具
- 具备数据建模能力
- 强烈的好奇心与求知欲

### 加分项

- 大规模 GPU 集群（NVIDIA/Ascend）运维或压测经验
- 熟悉 Redfish/IPMI；有 Rust 开发经验
- 了解 ACPI、OpenBMC、BIOS/UEFI、PCIe、I2C/I3C 等底层规范
- 跨地域容灾与 DCI 项目经验
- RDMA 生产环境调优经验
- 了解 AI 训练流程

## 关键洞察

### 1. 本 JD 与前七份 JD 的根本区别——"物理层"与"辅助层"

前七份 JD 聚焦于 **AI 特定的系统软件和平台工程**，而本 JD 覆盖了两个全新的维度：

```
前七份 JD（AI 系统层）：      本 JD 新增（物理层 + 辅助层）：
  ├── 训练推理框架                ├── 硬件选型/固件/压测  ← 物理硬件层 ★
  ├── 算子/通信/编译器              ├── RDMA/DCI/BGP 网络运维 ← 物理网络层
  ├── 超算集群架构                   └── IT 桌面/办公/资产   ← 企业辅助层 ★★
  ├── AI 集群可靠性/运维开发
  ├── 分布式存储
  ├── Agent 沙箱
  └── AI 搜索 / 预训练数据
```

| 对比维度 | 前七份 JD | 本 JD 硬件&网络方向 | 本 JD IT桌面方向 |
|---------|----------|-------------------|----------------|
| **抽象层级** | 系统软件、平台、中间件 | **物理硬件、固件、线缆** | 办公设备、人员流程 |
| **用户** | 模型/Agent/研究员 | GPU 集群、训练框架 | **全体员工** |
| **优化目标** | 吞吐、延迟、利用率 | **硬件性能极限、固件基线** | 零摩擦、SLA 响应 |
| **AI 特异性** | 高（专门为 AI 设计） | 中（底层通用但本文针对 AI 集群）| **低（通用 ITIL）** |
| **核心技能** | 分布式系统、CUDA、ML | **硬件架构、路由协议、RDMA** | 桌面运维、ITSM、资产管理 |

### 2. 硬件方向填补了"从比特到砖块"的最底层

[[AI集群可靠性工程]] 中提到可靠性工程师需要"从加速卡→NVLink→网卡→存储→调度→上层框架"的完整地图。硬件工程团队就是**这张地图的起点**：

```
硬件工程（本方向）              ← 选型/固件/压测 → 确保每张卡达到物理极限
    ↓
超算集群工程                   ← 将物理资源组建成可用的集群架构
    ↓
AI集群可靠性工程               ← 确保集群持续稳定运行
    ↓
AI训练推理系统工程             ← 在集群上运行训练/推理任务
```

关键依赖链条：
- 硬件选型错误 → 集群架构受限 → 训练效率天花板
- 固件基线不一致 → 节点间性能差异 → All-Reduce 长尾延迟
- 压测不充分 → 生产环境暴露硬件缺陷 → 训练中断

### 3. "硬件配置管理体系"——异构硬件管理的工程化

加分项中提到的 Redfish/IPMI/OpenBMC 揭示了硬件配置管理的工程化路径：

```
传统：        人手动登录每台服务器 → 检查固件版本 → 手动升级
工程化：      CMDB 登记硬件信息 → Redfish API 批量查询 → 基线比对 → 自动下发
SOTA：        OpenBMC 定制 → 固件+OS 一体化部署 → 持续验证
```

Redfish 是 DMTF 标准的硬件管理 RESTful API，支持：
- 批量查询固件版本、硬件健康状态
- 远程电源管理、启动顺序配置
- 传感器数据（温度、功耗）采集

### 4. "基准压测框架"——硬件验收的自动化演进

JD 中描述的压测框架是多代硬件验收经验的工程化：

```
L1：手动跑 benchmark → 记笔记 → 对比
L2：脚本化 → 自动跑 fio/iperf3/gpu-burn → 输出报告
L3：平台化 → 集成压测引擎 → 自动基线对比 → 阈值告警
L4：持续验证 → 每次变更后自动触发 → 回归基线
```

与 [[AI集群可靠性工程]] 中的"新硬件验收"互为补充：

| AI集群可靠性工程（验收流程）| 硬件工程（压测框架）|
|--------------------------|------------------|
| 定义验收的阶段和标准 | 提供每个阶段的工具和数据 |
| 管理交付的整体流程 | 支撑每个测试的技术实现 |
| 回答"性能是否达标" | 回答"差距来自哪一层" |

### 5. IT 桌面方向——组织 IT 成熟度的晴雨表

IT 桌面方向是七份 JD 中**唯一面向全体员工而非面向模型/算法/集群**的方向。它的存在说明这家公司：

- **规模到了需要专门团队**（数百员工以上）
- **重视员工体验**（新员工第一天设备就绪）
- **ITIL 流程成熟**（工单系统、SLA、资产全生命周期）
- **SaaS 化程度高**（工具集成、API 打通）

这是判断一家 AI 公司"工程文化成熟度"的信号——走向正规化的标志。

### 6. "Infrastructure is the ceiling"——工程哲学

> 基础设施不是背景板——它是上限。

这句话是整份 JD 的工程哲学核心。在 AI 公司中，基础设施团队通常不是最受关注的（光环在模型团队），但这份 JD 明确表达了：

| 角色的自我认知 | 对应的价值 |
|--------------|-----------|
| "我们不写模型" | 但模型训练能跑多快由我们决定 |
| "我们不写业务代码" | 但代码能否顺利部署由我们决定 |
| "我们不是主角" | 但我们是所有主角的上限 |

这是一种**成熟的基础设施文化**——承认自己的支持者角色，但对自己的技术标准有极致追求。
