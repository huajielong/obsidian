# Code Agent 数据工程师职位描述

## 来源概览

- **职位**: Code Agent 数据工程师
- **公司信号**: DeepSeek
- **招聘方向**: Code 专项、网络安全专项、Science 专项
- **团队使命**: "通过高质量的训练环境与精准评估体系，系统性地推高模型 Agent 能力上限，直到模型真正能独立驾驭复杂软件工程的开发"
- **核心公式**: Code = Agent 与物理世界交互的纽带

---

## 关键洞察

### 洞察 1：名字是"数据工程师"，实则是"Agent 能力工程师"

这是十份 JD 中最容易误解角色的一个。标题中的"数据工程师"容易让人联想到 [[预训练数据工程]] 中的数据清洗/Pipeline，但实际工作内容完全不同：

| 维度 | 传统数据工程 | 预训练数据工程 | 本 JD（Agent 能力工程） |
|------|-----------|-------------|----------------------|
| **核心工作** | 数据 ETL、数仓建设 | 语料采集、清洗、多模态处理 | **RL 环境构建、评测设计、能力短板补齐** |
| **数据用途** | 报表/分析 | 预训练 | RL 训练 + 评估 |
| **数据来源** | 业务数据 | 全网文本 | **真实代码工程/安全场景/科研实验** |
| **输出** | 数据表/仪表盘 | 训练语料 | **RL 环境 + 评测任务 + 训练数据** |
| **与模型的关系** | 无直接关系 | 预训练质量 | **直接塑造 Agent 能力** |

> **准确的定位**：这是一个"为 Agent 塑造能力的工程角色"——通过设计 RL 训练环境和评测体系，推动模型在真实工程场景中的能力提升。

### 洞察 2：团队使命直指 AGI 的"临界点"

> "模型在真实复杂场景中解决工程问题的能力则是通用智能最务实的度量标准……直到模型真正能独立驾驭复杂软件工程的开发，那将会是我们可以预见的 AGI 临界点之一。"

这段使命宣言包含几个重要信号：

1. **Code 能力 = AGI 的度量标准**——团队认为"独立驾驭复杂软件工程"是 AGI 的一个可观测临界点
2. **"最务实的度量标准"**——不是学术 benchmark，而是真实工程问题解决能力
3. **系统性方法论**——"通过高质量的训练环境与精准评估体系"，不是靠单点突破

这与此前 JD 共同体现的"第一性原理"工程哲学一致——不追逐 benchmark，而是追求真实能力提升。

### 洞察 3：三个方向覆盖模型 Agent 能力的三个关键维度

| 维度 | Code 专项 | 网络安全专项 | Science 专项 |
|------|----------|------------|-------------|
| **覆盖能力** | 代码编写/工程实现 | 安全分析/漏洞利用 | 科学研究/数据分析 |
| **RL 环境** | 真实代码工程场景 | 真实安全攻防场景 | 科研实验设计流程 |
| **评估方式** | 代码执行正确性 | PoC 能否触发漏洞 | 实验结果的可复现性 |
| **能力层级** | **基础建设（从底层构建）** | **攻击视角（从破坏中学习）** | **从假设到结论的完整链条** |
| **用户信号** | AI 编程工具使用体验 | CVE/CTF/红队实践 | 科研工作流 |
| **行业对标** | SWE-bench / HumanEval | CyberSecEval / 安全 benchmark | Science Benchmark (GPQA 等) |

三个方向合在一起，覆盖了模型在**数字世界**（代码）、**数字安全**（安全）和**物理世界理解**（科学）三个维度的 Agent 能力。

### 洞察 4：RL 环境构建——从"让人学"到"让模型学"

JD 的第一项职责是"将真实的代码工程、网络安全或科研场景转化为可训练的 RL 环境"——这本质上是在做**环境的工程化抽象**：

```
真实场景（100% 复杂度，含噪音）
    ↓ 抽象
RL 环境（保留核心挑战，去掉无关噪音）
    ↓ 设计
奖励信号（什么行为被鼓励/惩罚）
    ↓ 迭代
基于训练结果调整环境设计（"探索不同环境设计对模型 Agent 能力的影响"）
```

这与工程层的关系：
- **问题空间定义**：选择什么场景（不是所有场景都能 RL）
- **奖励设计**：定义什么是对的（自动 vs LLM-as-Judge vs 人工）
- **环境实现**：将场景变为可交互的代码环境（类似 Gym environment for Agent）
- **数据生产**：模型在环境中探索产生的轨迹本身就是训练数据

> **关键洞察**：RL 环境构建是比预训练数据清洗更"高阶"的数据工程——它不是在已有的文本中筛选，而是**创造模型可以与之交互的动态环境**。

### 洞察 5：评测任务设计——Eval Harness 的内容供给层

JD 的第二项职责是"设计高质量评测任务，量化模型在复杂真实场景中的表现"。这与 [[Eval_Harness]] 的关系是：

```
Eval Harness（框架/工具层）    ← [[Harness_Engineering]] 第 7 元件
    ↑ 使用
评测任务（内容/场景定义）       ← 本 JD 的核心产出
    ↑ 定义
真实场景（问题来源）           ← 代码工程 / 安全场景 / 科研流程
```

Eval Harness 是"怎么跑评测"的执行框架，而本 JD 的工作是"测什么、怎么才算对"的内容设计。两者缺一不可。

### 洞察 6："补全能力短板"——从用户数据到训练数据的闭环

JD 的第三项职责："基于真实用户数据与反馈，定位模型当前的能力缺陷，针对性构建训练环境与数据，推动模型在弱项上持续进步。"

这形成了一个完整的能力闭环：

```
真实用户使用 → 失败/低效场景 → 定位能力缺陷
                                     ↓
模型能力提升 ←── RL 训练 ←── 构建训练环境与数据
```

这个闭环体现了该团队的核心工程哲学：

> **用真实世界的失败信号驱动能力提升，而非从教科书出发定义"应该会什么"。**

### 洞察 7：跨 JD 分析——十份 JD 的最后一块拼图

将此 JD 放入完整的堆栈全景中：

```
产品化层     AI产品工程                             ← 第九份
    ↑
══════════ 产品层 / 技术层 分界线 ══════════
    ↑
能力构建层   Agent能力工程                          ← ★ 第十份（本 JD）——横跨多层
    ↑
控制逻辑层   Harness Engineering
            Prompt Engineering / Context Engineering
    ↑
══════════ 应用层 / 基础设施层 分界线 ══════════
    ↑
框架层       训练推理框架工程
计算层       算子/通信/编译器工程
集群层       超算集群工程 / AI集群可靠性工程 / AI平台运维
存储层       AI存储工程
硬件层       AI基础设施硬件工程
设施层       AI数据中心工程
安全层       Agent沙箱工程
```

**十份 JD 最终构成完整的体系**：

| 层级 | JD | 核心角色 |
|------|----|---------|
| **产品化** | AI 产品经理 | 将技术转化为产品 |
| **能力构建** | Code Agent 数据工程师 ★ | 塑造模型的 Agent 能力 |
| **控制逻辑** | Harness 团队 | 设计 Agent 执行框架 |
| **服务** | 服务端工程、AI 搜索 | 支撑在线服务 |
| **数据** | 预训练数据工程 | 提供燃料 |
| **框架** | 训练推理框架工程 | 编排训练/推理 |
| **计算** | 算子/通信/编译器工程 | 逼近硬件极限 |
| **集群** | 超算集群工程 + 可靠性工程 + 平台运维 | 建设与运营集群 |
| **存储** | 分布式存储工程 | 数据底座 |
| **硬件** | IT 基础设施硬件工程 | 物理设备管理 |
| **设施** | IDC 数据中心工程 | 物理建筑设施 |
| **安全** | Agent 沙箱工程 | 安全执行环境 |

> **终极启示**：一个 AI 公司的竞争力是全链条的——从产品体验到物理设施，十个团队各司其职，缺一不可。这份 JD 特别之处在于，它不是管一个"层"，而是横跨多层（数据 × 训练 × 评估 × RL），指向模型能力本身。
