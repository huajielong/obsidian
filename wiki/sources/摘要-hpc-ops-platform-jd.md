# AI 平台运维工程师职位描述

> **来源**：互联网公开招聘信息（2026 年），由 Claude 获取并整理。公司来源未明确标注。
>
> **核心定位**：本 JD 与 [[摘要-hpc-cluster-reliability-jd]] 来自**同一团队/公司**（招聘哲学、素质要求、面试流程、甚至措辞完全一致）。但本 JD 在集群运维方向基础上新增了**运维开发方向**——从"守"到"建"的完整闭环。
>
> **两个方向的关系**：集群运维负责"前线"（故障响应、日常巡检、硬件维修），运维开发负责"后勤"（构建平台工具让前线高效运作）。两者共同构成完整的 AI 集群可靠性工程体系。

---

## 两个招聘方向

| 方向 | 工作内容 | 核心产出 |
|------|---------|---------|
| **集群运维方向** | 十万卡集群的稳定性保障、断点续训、推理服务弹性伸缩、故障排查 | 稳定的训练/推理服务，持续缩短 MTTR |
| **运维开发方向** | 运维平台（CMDB、运维作业、可观测性）的架构设计与开发；对接第三方系统；跨团队协作 | 高效的运维工具链和平台体系 |

---

## 集群运维方向

### 团队使命

在 AGI 的纪元，竞争的本质已升维为超大规模算力系统性能与可靠性的终极角力。

### 招聘哲学

> 与 [[摘要-hpc-cluster-reliability-jd]] 完全一致。包括：
> - "不太在意你过去有没有运维经验"
> - "AGI 本身就是人类从未走过的新路"
> - "请一定投递我们"
> - 同样的面试流程（项目深挖 / 故障推演 / AGI 愿景）
> - 同样的素质要求清单（对 AGI 的热情 / 拆解问题的逻辑能力 / 系统性技术视野 / 清晰的表达 / 动手的冲动 / 对生产环境的敬畏）

### 核心挑战

| 挑战 | 说明 | 新增内容（相对之前可靠性 JD）|
|------|------|--------------------------|
| 十万卡集群稳定性 | 设计自愈系统，让训练在硬件持续故障中不停顿 | — |
| "快"与"稳"的平衡 | 算法同学每天迭代多个版本，调度需要极度灵活，同时生产环境足够稳 | **新增视角** |
| 打开"黑盒" | 让 GPU 利用率波动、NCCL 延迟抖动、Checkpoint 耗时都可观测 | — |
| 探索无人区 | 模型规模扩大时现有调度/网络/存储方案可能失效 | **新增视角** |

> **与上一份可靠性 JD 的区别**：本 JD 更强调"不像运维的运维"——角色随问题流动，不局限于分工边界。

## 运维开发方向（新内容）

### 岗位职责

| 职责 | 说明 | 相关技术 |
|------|------|---------|
| **运维平台架构设计与开发** | 主导 CMDB、运维作业平台、可观测性平台等核心系统的架构设计与实现 | Python / Golang，系统架构设计 |
| **系统集成** | 对接第三方开放平台、监控系统、仓管系统、采购系统 | API 集成、数据同步、协议适配 |
| **跨团队协作** | 与运维团队、采购团队、研发团队深度协作，持续完善运维开发需求 | 需求分析、迭代管理 |

### 核心要求

- 本科及以上学历
- 较好的 Python/Golang 开发能力，扎实的工程能力
- 对运维体系有较全面的认知，深入理解运维流程
- 熟悉 CMDB、工单系统等运维平台
- 较好的沟通能力和理解能力

### 加分项

- 3 年以上一线运维经历，深知运维痛点
- 熟悉 SRE 体系
- 成熟运维平台开发经历

## 关键洞察

### 1. 本 JD = 前一份可靠性 JD 的"运维开发扩展包"

两份 JD 的完整关系：

```
前一份可靠性 JD（单一方向）
    └─ 守护"吞金兽"（故障管理）
    └─ 开箱即巅峰（新硬件验收）
    └─ 打开"黑盒"（可观测性）
    └─ 自动化替代人工（自愈）
                    ↕ 同一团队，补充视角
本 JD（双方向）
    ├─ 集群运维方向（与前一份基本一致，新增"快与稳平衡"、"无人区探索"视角）
    └─ 运维开发方向 ★全新
         └─ CMDB / 运维作业平台 / 可观测性平台 的架构设计与开发
         └─ 第三方系统集成
         └─ 运维流程工程化
```

### 2. "不像运维的运维"——角色流动性的工程文化

JD 开篇即定义了一种**反分工的工程文化**：

> 我们不希望你把自己定义为一个"运维工程师"。在我们团队，"运维"只是一个出发点，而不是一个牢笼。

这反映了：
- **问题驱动**：角色边界由问题定义，不由岗位定义
- **全栈要求**：写代码、调硬件、画架构图——什么都需要
- **AGI 目标对齐**：所有工作最终服务于"加速 AGI 实现"

这种文化在 AI 基础设施团队中越来越普遍——因为 AI 领域的问题往往跨越传统分工边界。

### 3. 运维开发方向填补了"构建 vs 运营"的工具链空白

前一份可靠性 JD 定义了"Run"的哲学（监控、故障、自愈），但缺少了**支撑 Run 的平台工具**。运维开发方向补上了这一块：

```
Run（集群运维 → 前一份 JD）
    ↓ 依赖
Platform（运维开发 → 本 JD 新增）
    ↓ 构建
CMDB / 可观测性平台 / 作业平台 / 工单系统
    ↓ 集成
第三方系统（监控/仓管/采购）
```

**无平台支撑的可靠性工程**：全手工处理故障 → 效率低、容易出错、无法规模化
**有平台支撑的可靠性工程**：自动化处理大多数场景 → 人专注于例外和持续改进

### 4. "快与稳的平衡"——AI 基础设施的核心张力

新增的"快与稳的平衡"挑战揭示了 AI 基础设施的核心矛盾：

| 维度 | 算法团队想要（快）| 平台团队想要（稳）|
|------|----------------|----------------|
| 资源分配 | 立即到卡，随时释放 | 提前规划，按批次分配 |
| 实验频率 | 每天数十次变更 | 每周一次变更窗口 |
| 资源回收 | 随时归还 | 按固定周期回收 |
| 失败容忍 | "跑起来看看结果" | "确保不拖累生产" |

**解决方案不是二选一，而是分层治理**：
- 开发环境：完全灵活，允许快速实验（可接受偶尔失败）
- 预生产环境：有限灵活性，自动化验证
- 生产环境：严格的变更管理，自动化回滚

### 5. 运维开发的"平台工程"属性

运维开发方向本质上是**面向 AI 基础设施的平台工程（Platform Engineering）**：

| 维度 | 传统运维开发 | AI 平台运维开发 |
|------|------------|----------------|
| 用户 | 运维团队（内部）| 算法团队 + 运维团队 + 采购团队 |
| 平台范围 | CMDB + 监控 + 工单 | CMDB + 可观测性 + 训练调度 + 故障自愈 + 资源管理 |
| 核心挑战 | 配置管理和告警 | 训练作业的可观测性和自愈 |
| 关键规模 | 千级服务器 | 十万级 GPU |
| 技术复杂度 | 中等 | 高（需理解 GPU/网络/存储全栈） |

### 6. 与平台工程的关联

运维开发方向与 [[Harness_Engineering]] 的 Tool Registry（#2）和 Telemetry / Observability（#6）有直接交集：

| 运维开发平台 | 对应 Harness 元件 | 说明 |
|------------|-----------------|------|
| CMDB（配置管理数据库）| Tool Registry（#2）| 资源发现、配置管理、权限控制 |
| 可观测性平台 | Telemetry / Observability（#6）| 指标采集、可视化、告警 |
| 运维作业平台 | Agent Loop（#1）| 自动化流水线、作业编排 |
| 工单/变更管理系统 | Safety Layer（#4）| 变更审批、风险管控 |
