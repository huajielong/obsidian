# 高性能算子/通信/编译器工程师职位描述

> **来源**：互联网公开招聘信息（2026 年），由 Claude 获取并整理。公司来源未明确标注。
>
> **核心定位**：AGI 与物理芯片之间的**最后一层软件**——为训练推理框架和研究员提供高性能的计算算子、集合通信和领域编译器。

---

## 团队使命

构建 AGI 与物理芯片之间的最后一层软件，为训练推理框架和研究员提供高性能的计算算子、集合通信和领域编译器，同时也是模型设计和硬件 co-design 的桥梁。

**团队的唯一标准：硬件的物理极限。** 不和任何 baseline 或者其他实现比较性能差异，只关心当前设计和实现距离理论上限还有多远，对每一个 cycle 和每一瓦功耗有着近乎偏执的追求。

为上层框架和研究员提供稳定的接口，使底层的高性能实现以使用友好、可靠且可维护的方式融入整个系统。

加入团队，将同时深入 NVIDIA GPU 和 Ascend NPU 等国产硬件，从指令集、内存模型、编译栈到大规模互联，系统性地掌握多种硬件体系。团队不在意是否有相关背景——会尽可能提供必要的指导，但需要的是对细节的严谨和对性能优化的热情。

---

## 工作职责（四个方向）

| 方向 | 核心工作 | 关键技术 |
|------|---------|---------|
| **高性能算子开发** | 针对 GPU/NPU 架构设计、实现并维护 GEMM、Attention 及其他算子 | CUDA、Ascend C、TileLang、CUTLASS、DeepGEMM |
| **集合通信开发** | 设计、实现并维护通信算子（DeepEP 中的 EP、CP/DP、Engram、PP）；持续对 NCCL/HCCL 等主流通信库调优和排障 | DeepEP、NCCL、HCCL、NVSHMEM |
| **DSL 编译器研发** | 针对深度学习的 DSL 编译器（TileLang 等）；设计面向新一代硬件的编译优化；围绕新模型、新算子和性能优化扩展 DSL/IR/CodeGen 能力 | TileLang、MLIR、Triton、CodeGen |
| **模型-硬件协同设计** | 紧贴模型结构和算法演进，和框架同学、研究员共同为新思路设计硬件友好的模型结构和算法；对真实训练/推理负载持续进行性能分析 | 模型架构分析、性能 Profiling、Co-design |

- 熟悉 C++ 等编程语言，熟练掌握 Git 等工具
- 对细节有较高关注，对性能优化有较高热情，眼界广阔，同时对代码有品味的追求
- 良好的中文沟通能力

## 加分项

- 有过 CUDA 或 Ascend C 算子编写经验，熟悉 CUTLASS、DeepGEMM 等算子库，熟练掌握 PTX ISA、Ascend C API 并深入理解 memory model 和硬件体系结构
- 熟悉主流深度学习分布式通信库（如 NCCL、HCCL、NVSHMEM 和 DeepEP 等）的底层实现原理，具备丰富的性能调优与故障排查经验
- 深入理解 RDMA 技术原理与 RoCE/InfiniBand 网络架构，具备丰富的 RDMA 开发经验，熟悉 IBGDA 等 RDMA 高阶特性
- 具备编译器或 DSL 系统的开发经验，熟悉 MLIR/TileLang/Triton 等任一编译栈或高性能算子 DSL；理解常见编译优化技术，以及 GPU/NPU 等架构的并行执行模型、内存层级和矩阵计算单元等硬件架构特性
- 在大型开源软件（如 LLVM、NCCL 和 PyTorch 等）中有深度贡献，或维护过其他个人开源软件
- 在高性能计算相关赛事（SC/ASC/ISC/PAC 等）或算法竞赛（NOI/IOI/ICPC 等）中取得优异成绩

## 关键洞察

### 1. "硬件的物理极限"——极致性能优化的哲学

> 团队的唯一标准：硬件的物理极限。

这是这份 JD 中最重要的一句话。它定义了团队的工作哲学：

| 传统做法 | 本团队做法 |
|---------|-----------|
| "比上次快 20% 就行" | "距离理论峰值还有多远？" |
| "比 CUTLASS 快就行" | "CUTLASS 离极限还有距离" |
| "足够好"就收手 | 压到每一个 cycle |
| 关注相对性能 | 关注**绝对性能**（与硬件极限的差距） |

这要求团队对硬件架构有**显微级**的理解——知道每个 CUDA Core/Tensor Core 每周期能做什么、内存带宽的理论峰值是多少、各种延迟的精确数值。然后把软件优化到这个极限。

### 2. 三块"硬骨头"——算子、通信、编译器的技术栈全景

这份 JD 覆盖了 AI 基础设施中**最底层的三个核心技术栈**：

```
                  ┌─────────────────────────┐
                  │  训练/推理框架            │  ← PyTorch/Megatron/vLLM
                  │  (调用方)                 │
                  └──────────┬──────────────┘
                             │ 调用
          ┌──────────────────┼──────────────────┐
          ▼                  ▼                  ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│ 高性能算子        │ │ 集合通信          │ │ DSL 编译器       │
│ (Compute Kernels)│ │ (Collective Comm) │ │ (Domain Compiler)│
│                 │ │                 │ │                 │
│ GEMM/Attention  │ │ NCCL/HCCL       │ │ TileLang/Triton │
│ CUTLASS/DeepGEMM │ │ DeepEP/NVSHMEM  │ │ MLIR/LLVM       │
│ CUDA/Ascend C   │ │ RDMA/IBGDA      │ │ CodeGen         │
└────────┬────────┘ └────────┬────────┘ └────────┬────────┘
         │                   │                   │
         └───────────────────┼───────────────────┘
                             │
                    ┌────────▼────────┐
                    │   GPU/NPU 硬件   │
                    │  (物理芯片)      │
                    └─────────────────┘
```

- **算子**：算法的基础原子操作（矩阵乘、注意力机制等），决定单次计算的效率
- **通信**：多卡/多节点之间的数据交换，决定分布式训练的扩展效率
- **编译器**：从高层算子描述到硬件指令的自动映射，决定开发效率和硬件利用率的平衡

### 3. DeepEP 生态——从通信库看团队技术栈

JD 中特别提到了 **DeepEP**（EP、CP/DP、Engram、PP），这是一个值得注意的信号：

| DeepEP 组件 | 全称 | 功能 |
|------------|------|------|
| **EP** | Expert Parallelism | MoE 专家并行通信 |
| **CP/DP** | Context Parallelism / Data Parallelism | 上下文并行与数据并行 |
| **Engram** | — | 内存管理/通信融合 |
| **PP** | Pipeline Parallelism | 流水线并行通信 |

DeepEP 是一个专注于 MoE（Mixture-of-Experts）模型通信的库。JD 提及它说明这个团队的工作负载**高度关注 MoE 架构**——这是当前和下一代大模型（如 DeepSeek V4/V5、GPT-5 等）的主流方向。

### 4. 三套"编译器"技术的代际关系

JD 提到 TileLang、MLIR 和 Triton 代表了 DSL 编译器技术的三个代际/层次：

```
Triton         ← 第一代：OpenAI 提出的算子 DSL，屏蔽 CUDA 复杂性
    ↓
MLIR           ← 第二代：LLVM 生态的多级 IR 框架，可扩展、可组合
    ↓
TileLang       ← 第三代：面向 AI 计算的新一代 DSL，整合 Tile 调度 + 硬件抽象
```

这个团队用 **TileLang** 作为主攻方向，说明他们走在最前沿——不仅是用现成的，还在参与定义下一代 AI 编译器。

### 5. "不在意背景"的真实含义

> 我们不在意你是否有相关背景 —— 我们会尽可能提供必要的指导。

这不是"谁都能来"的意思。结合上下文，它的真实含义是：

| 他们不在意 | 他们在意 |
|-----------|---------|
| 是否已经写过 CUDA/Ascend C | **对细节的严谨** |
| 是否已经有编译器经验 | **对性能优化的热情** |
| 是否来自 AI 领域 | **对代码品味的追求** |
| 是否有相关论文 | **理解问题的深度** |

**愿意招"白纸"但必须是"有热情的白纸"**——这是传递信号：基础能力（C++、系统思维、性能直觉）比已有经验更重要。

### 6. 硬件覆盖——NVIDIA + Ascend 的双轨策略

JD 明确提到了同时深入 NVIDIA GPU 和 **Ascend NPU（华为昇腾）**：

| 维度 | NVIDIA GPU | Ascend NPU |
|------|-----------|-----------|
| 编程模型 | CUDA / PTX | Ascend C / CANN |
| 计算核心 | CUDA Core + Tensor Core | Da Vinci Core (Cube Unit + Vector Unit) |
| 内存模型 | 全局/共享/本地/寄存器 | 全局/L1 Buffer/L0 Buffer/寄存器 |
| 互联 | NVLink / NVSwitch | HCCS (华为高速互联) |
| 通信库 | NCCL | HCCL |
| 编译器 | NVCC / Triton / TileLang | 昇腾图编译器 / TileLang |

双轨策略意味着该团队的产品需要同时适配两条硬件路线，这对算子库和编译器的**硬件抽象层**提出了极高要求。
