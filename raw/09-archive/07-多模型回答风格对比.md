# 实验七：多模型回答风格对比

> [[Ollama LLM 实验系列索引]] | 同一 Prompt → 4 个模型 → 看风格差异

## 实验方法

对同一观点判断类 Prompt（temperature=0.0），同时送给 4 个模型，比较回答风格、长度、判断取舍的差异。

**Prompt**：

> 阅读以下观点，请给出你的判断和分析：
> "远程办公（remote work）会降低团队效率，应该全面取消。"
> 请从正反两方面分析，然后给出你的最终判断。

**候选模型**：

| # | 模型 | 平台 | 类型 |
|:--|:-----|:-----|:-----|
| 1 | llama3.2:3b | Ollama（本地） | 3B 通用 |
| 2 | qwen3.5:0.8b | Ollama（本地） | 0.8B 通用 |
| 3 | DeepSeek V4 Pro | 云端 API | 云端 MoE |
| 4 | 豆包 Seed Character | 云端 API | 云端对话 |

## 实验结果

### 性能指标对比

| 指标 | llama3.2:3b | qwen3.5:0.8b | DeepSeek V4 Pro | 豆包 |
|:----|:-----------:|:------------:|:--------------:|:---:|
| 平台 | Ollama 本地 | Ollama 本地 | 云端 API | 云端 API |
| 输入 Token | 67 | 42 | **37** | 64 |
| 输出 Token | 408 | **2,660** | 1,246 | 730 |
| 中文字数 | 673 | **1,504** | 1,353 | 1,195 |
| 延迟 | 47.3s | **145.4s** | 25.4s | **10.2s** |
| 生成速度 | 8.6 tok/s | 18.3 tok/s | ~49 tok/s | ~72 tok/s |
| 成本/次 | ¥0 | ¥0 | ~¥0.0003 | ~¥0.00006 |

### 各模型完整回答

#### llama3.2:3b（408 tokens, 673 字, 47.3s）

简洁要点式，中英混用，立场平衡：

```
正面分析：
1. 减少交通时间和成本：远程办公可以节省员工通勤时间和成本...
2. 提高工作效率：远程办公可以让员工更专注于工作...
3. 增加工作 Flexibility...

反面分析：
1. 降低团队合作能力：远程办公可能减少团队之间的直接交流...

最终判断：虽然远程办公有其优点，但它也存在一些缺点。
然而，随着技术的进步和团队的适应能力，远程办公可以成为
一个有效的工作方式...
```

#### qwen3.5:0.8b（2,660 tokens, 1,504 字, 145.4s）

长篇学术风，先批评观点再展开分析，话痨倾向明显：

```
总体评价：该观点是一个绝对化的负面主张...虽然远程办公
确实存在效率下降的风险或局限性，但将其视为"应全面取消"
的结论是不准确的。更合理的建议是采取混合模式（Hybrid
Model）...

一、正面分析：为什么不能简单否定？
1. 灵活性与成本效益...
2. 人才与知识共享优势...

二、反面分析：远程办公的挑战不能回避...

最终判断：应当放弃"全面取消"的简单思路...
```

#### DeepSeek V4 Pro（1,246 tokens, 1,353 字, 25.4s）

辩论赛式的正反方结构，立场鲜明层层递进：

```
正方观点：支持"远程办公降低效率，应全面取消"
1. 沟通与协作效率下降...
2. 管理与监督难度加大...
3. 团队凝聚力与文化建设受损...
4. 工作安全与合规风险...

反方观点：反对"全面取消远程办公"
1. 大量证据表明远程并不会必然降低效率...
2. 远程办公能提升员工满意度与保留率...

最终判断：折中方案——混合办公模式...
```

#### 豆包（730 tokens, 1,195 字, 10.2s）

分节论述，条理最清晰，语言流畅自然：

```
一、支持的理由
1. 协作沟通的信息损耗...
2. 工作边界模糊与管理难度...
3. 团队凝聚力与归属感弱化...

二、反对的理由
1. 减少通勤损耗，提升个人效率...
2. 灵活用工降低成本，提升人才吸引力...

三、最终判断
远程办公不应全面取消，但需要建立规范的管理体系...
```

### 风格差异总结

| 维度 | llama3.2:3b | qwen3.5:0.8b | DeepSeek V4 Pro | 豆包 |
|:----|:-----------|:------------|:--------------|:---:|
| 字数 | 673 | **1,504** | 1,353 | 1,195 |
| 结构 | 要点列表 | 段落+分析框架 | **正反辩论式** | 分节论述 |
| 语言 | **中英混用** | 纯中文书面 | 纯中文正式 | 纯中文流畅 |
| 立场 | 中立平衡 | 先批再分析 | 正反分开辩论 | 客观分析 |
| 延迟 | 47.3s | **145.4s** | 25.4s | **10.2s** |
| 速度 | 8.6 t/s | 18.3 t/s | 49 t/s | 72 t/s |

## 分析

### 1. 云端 vs 本地：延迟和回复量

云端模型（DeepSeek 25s / 豆包 10s）比本地模型（llama3.2 47s / qwen3.5 145s）**快 2~14 倍**。豆包以 10.2s 位居最快，生成速度达 72 tok/s。

同时，两个云端模型的回复量（1,195~1,353 字）远超 llama3.2（673 字），达到了 qwen3.5 级别的详细程度，但速度快了 6~14 倍。

### 2. "小模型快"是陷阱（再验证）

qwen3.5:0.8b 生成速度 18.3 tok/s 是 llama3.2 的 2 倍，但因为输出多达 **2,660 tokens**（llama3.2 的 6.5 倍），总耗时反而慢 3 倍——这与实验六的发现完全一致。

### 3. 回答结构的模型差异

四个模型呈现四种不同的回答结构：
- **llama3.2**：简单的正反要点列表，最像人类快速回答
- **qwen3.5**：先定调（"观点过于片面"），再展开分析框架
- **DeepSeek**：模拟辩论赛式的"正方观点 vs 反方观点"结构
- **豆包**：分"支持的理由"和"反对的理由"两节，条理最清晰

### 4. llama3.2 的中英混用

只有 llama3.2 在中文回答中混入英文词（Flexibility / office），其他三个模型纯中文。说明 llama3.2 的训练数据以英文为主，中文是微调阶段加入的。

## 附：实验代码

```python
# 多模型对比 — run_cmp.py
import json, urllib.request, time, sys, os
sys.stdout.reconfigure(encoding="utf-8", errors="replace")
# 从 keys.env 读取 API 密钥
k1 = ""; k2 = ""
_env = os.path.join(os.path.dirname(os.path.abspath(__file__)), "keys.env")
if os.path.exists(_env):
    for _l in open(_env, encoding="utf-8"):
        _l = _l.strip()
        if "=" in _l:
            _kv = _l.split("=", 1)
            if _kv[0] == "DK1": k1 = _kv[1]
            elif _kv[0] == "DK2": k2 = _kv[1]
if not k1: k1 = os.environ.get("K1", "")
if not k2: k2 = os.environ.get("K2", "")

P = '阅读以下观点并给出你的判断："远程办公会降低团队效率，\
应该全面取消。" 请从正反两方面分析，然后给出你的最终判断。'
T = 0.0
RES = []

def ollama(m):
    p = {"model": m, "messages": [{"role": "user", "content": P}],
         "stream": False, "options": {"temperature": T}, "keep_alive": "10m"}
    req = urllib.request.Request("http://localhost:11434/api/chat",
        data=json.dumps(p).encode(), headers={"Content-Type": "application/json"})
    with urllib.request.urlopen(req, timeout=180) as f:
        return json.loads(f.read())

for m in ["llama3.2:3b", "qwen3.5:0.8b"]:
    t0 = time.time(); r = ollama(m); lt = time.time() - t0
    t = r["message"]["content"]
    RES.append((m, "Ollama", lt, r["prompt_eval_count"], r["eval_count"], len(t), t))
    print(f"{m}: {lt:.1f}s  in={r['prompt_eval_count']} out={r['eval_count']} ({len(t)}字)")

def cloud_call(url, model, key):
    h = {"Authorization": "Bearer " + key, "Content-Type": "application/json"}
    d = json.dumps({"model": model, "messages": [{"role": "user", "content": P}], "temperature": T})
    t0 = time.time()
    with urllib.request.urlopen(urllib.request.Request(url, data=d.encode(), headers=h), timeout=120) as f:
        r = json.loads(f.read())
    return r, time.time() - t0

if k1:
    r, lt = cloud_call("https://api.deepseek.com/chat/completions", "deepseek-v4-pro", k1)
    t = r["choices"][0]["message"]["content"]
    RES.append(("DeepSeek V4 Pro", "Cloud", lt, r["usage"]["prompt_tokens"], r["usage"]["completion_tokens"], len(t), t))
    print(f"DeepSeek: {lt:.1f}s  in={r['usage']['prompt_tokens']} out={r['usage']['completion_tokens']} ({len(t)}字)")

if k2:
    r, lt = cloud_call("https://ark.cn-beijing.volces.com/api/v3/chat/completions", "doubao-seed-character-260628", k2)
    t = r["choices"][0]["message"]["content"]
    RES.append(("Doubao", "Cloud", lt, r["usage"]["prompt_tokens"], r["usage"]["completion_tokens"], len(t), t))
    print(f"Doubao: {lt:.1f}s  in={r['usage']['prompt_tokens']} out={r['usage']['completion_tokens']} ({len(t)}字)")

# 汇总输出
print(f"\n{'Model':<18} {'Lat':>6} {'InT':>5} {'OutT':>5} {'Chars':>5}")
print("-" * 45)
for r in RES:
    print(f"{r[0]:<18} {r[2]:>5.1f}s {r[3]:>4d} {r[4]:>4d} {r[5]:>4d}")
for r in RES:
    print(f"\n=== {r[0]} ({r[4]} tokens, {r[5]}字) ===")
    print(r[6][:500])
```

## 核心结论

### 1. 模型固有风格差异巨大，temperature=0 不会消除

四个模型对同一 prompt 给出了 **四种完全不同的回答结构** — 这不是随机采样造成的，而是模型架构和训练数据决定的"性格"差异。即使 temperature=0（完全确定性输出），模型之间的差异依然显著。

### 2. 云端 API 综合优势明显

| 指标 | 本地 Ollama | 云端 API |
|:----|:----------:|:--------:|
| 延迟 | 47~145s | **10~25s** |
| 字数 | 673~1,504 | **1,195~1,353** |
| 成本 | ¥0（但需本地硬件） | ~¥0.00006~0.0003/次 |

豆包综合表现最好（最快+条理最清晰），DeepSeek 的结构创意最独特（辩论式）。

### 3. "小模型快"是错觉

qwen3.5:0.8b（0.8B）的生成速度是 llama3.2:3b（3B）的 2 倍，但输出量是 6.5 倍，实际耗时反而慢 3 倍。**选模型不能只看参数大小和生成速度，要看实际输出长度。**

### 4. 实践建议

- **需要快速、稳定、高质量回复** → 用云端 API（豆包首选）
- **需要创意性回答结构** → 用 DeepSeek（辩论/对比式）
- **需要离线/隐私场景** → 用 Ollama 本地模型，但要实测输出长度
- **需要简洁回复** → llama3.2（惜字如金，但中英混用需注意）
- **需要详细分析** → qwen3.5（但要做好等 2 分钟的心理准备）
